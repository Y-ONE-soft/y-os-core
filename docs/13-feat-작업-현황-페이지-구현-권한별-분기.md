## 1. 작업 내용 요약

`/projects` 자리표시 페이지를 실제 **작업 현황** 화면으로 구현했다. 핵심 특징은 **로그인한 계정의 권한(MASTER/STAFF)에 따라 화면이 달라지는 것** — 병렬로 머지된 세션 기반 인증(#7)의 실세션 `useSession().user.role`을 그대로 사용한다 (master01/step01 실로그인으로 분기 검증).

| 요소 | Master (`84:11`) | Staff (`87:1029`) |
|---|---|---|
| 필터 | **그룹 칩** + 프로젝트 칩 | 프로젝트 칩만 (그룹 행 없음) |
| 프로젝트 범위 | 전체 (4개) | **배정 프로젝트만** (Soft 3개, 와이즈 제외) |
| 로드맵 | 그룹 헤더 행(접기 토글) → 프로젝트 → 단계 | 그룹 행 없이 플랫 |

공통: 헤더(작업 현황 + 설명 + `＋ 작업` 버튼), 선택 요약(N개 프로젝트 · M개 작업 — 실계산), 뷰 스위처(로드맵 활성/담당자/캘린더), 멀티 프로젝트 로드맵(프로젝트 `전체 %` 스팬 막대 + 단계별 색상 막대, 보드 스토어 실데이터).

## 2. 주요 구현 결정

| 항목 | 결정 | 이유 |
|---|---|---|
| 권한 소스 | 실세션 `useSession()` (인증 #7) | 당초 개발용 권한 전환 훅을 만들었으나 작업 중 인증이 main에 머지되어 **폐기하고 실세션 채택** — "로그인한 계정 권한" 요구에 정확히 부합 |
| 스탭 배정 프로젝트 | `STAFF_ASSIGNED_PROJECT_IDS` 상수 (디자인 기준 Soft 3개) | 배정(멤버) 도메인 미구현 — 자리표시, 멤버 태스크에서 대체 |
| 로드맵 데이터 | 보드 스토어(#6) 재사용 + `useBoardState()`(전체 집계용) 추가 | 상세 보드와 단일 데이터 소스 — 상세에서 단계 추가하면 작업 현황에 즉시 반영 |
| 시드 확장 | 디자인의 단계 구성대로 p-cms(7단계)·p-contents(3)·p-wise(1) 추가, 스토리지 키 v1→**v2** | 기존 로컬 저장분은 초기화됨 (개발 단계 허용). 시작일만 있는 진행형 단계 지원 추가 |
| 공용 유틸 추출 | `roadmap-utils.ts` (dayOffset/formatShort/hexToRgba/barRange) | 상세 로드맵·보드·작업 현황 3곳 중복 제거, 진행형(종료일 없음) 막대 계산 일원화 |
| 프로젝트 `전체` 막대 | 단계 시작일 min ~ 종료일 max 스팬, 완료율 = 전체 작업 done/total | 정적 값 제거, 실데이터 파생 |

- **main 빌드 깨짐**: #6(보드)과 #7(인증)이 각자 통과했지만, #7이 `CURRENT_USER` 상수를 제거하면서 #6의 `stage-add-dialog.tsx`가 깨진 import를 갖게 됨 (머지 순서상 어느 PR도 감지 못함). 이 태스크에서 세션 기반으로 수정했는데, **다른 세션도 같은 문제를 발견해 동일 수정(#8)을 먼저 머지** — 리베이스에서 한 줄 충돌(fallback 표기)만 main 쪽으로 채택해 수렴. docs 11→12 rename도 동일하게 #8과 중복 → main 쪽 채택
- 리베이스 후 스테일 Prisma 클라이언트(Session 모델 부재)로 빌드 실패 → `prisma generate` 재실행으로 해결. **교훈: 머지 전 최신 main 리베이스 + 빌드 재확인이 병렬 체계의 필수 관문** (규칙화 검토)
