# docs: Figma에 없는 화면 디자인 규약 정리

요청 사이클 `디자인-규약-문서화`의 단일 태스크. Figma에 디자인이 없는 화면을 만들 때 따를 기본값을 CLAUDE.md에 남기고, 그 규약에서 벗어나 있던 내 정보 페이지를 맞춘다.

## 배경

docs/87(내 정보 UI 정합)에서 드러난 문제다. **Figma에 내 정보 화면 디자인이 없어** 그 화면만 근거 없이 만들어졌고, 결과적으로 앱의 시각 언어에서 벗어났다.

같은 처지의 화면이 계속 늘고 있다 — 프리셋 관리 페이지가 동시에 만들어지는 중이고, 죽어 있는 메뉴 항목(`알림`·`설정`)도 언젠가 페이지가 된다. 규약이 없으면 **화면마다 다르게 굳는다.**

## 착수 전 정정한 것

처음에는 "규격이 이미 사실상 정해져 있으니 정리만 하면 된다"고 보고했다. **실측해 보니 사실이 아니었다.**

| 페이지 | 컨테이너 | 제목 |
|---|---|---|
| 작업 현황 | `gap-3.5 p-6` | `text-xl` (20px) |
| 프로젝트 상세 | `gap-3.5 p-6` | `text-xl` (20px) |
| 내 할일 | `gap-4 px-6 pb-6 pt-5` | `text-[22px]` |
| 내 정보 | `gap-4 px-6 pb-6 pt-5` | `text-[22px]` |

게다가 **내 할일도 Figma 기반 화면이다**(타임라인 패널이 Figma 199:1139 참조). 즉 "Figma 있는 화면 vs 없는 화면"의 차이가 아니라 **Figma 화면들끼리도 규격이 다르다.**

그대로 받아쓰면 존재하지 않는 규약을 지어내는 셈이라, 사용자에게 보고하고 방향을 다시 정했다.

**결정(사용자 승인)** — 현황을 그대로 기록하고 새 화면용 권장값만 명시한다. **기존 화면은 건드리지 않는다**(Figma 기반 화면을 임의로 바꾸는 것이라 디자인 확인이 필요하고, 진행 중인 5개 세션과 충돌한다). 기록 위치는 CLAUDE.md — 모든 세션이 매번 읽으므로 docs/에만 두면 다른 세션이 안 보고 지나칠 가능성이 크다.

## 실측 근거

규약 값은 추측이 아니라 코드 전수 조사로 뽑았다.

**패널** — 6곳 모두 `border bg-background shadow-[0px_1px_3px_0px_rgba(0,0,0,0.05)]`로 동일. 라운딩만 갈린다.

| 라운딩 | 사용처 |
|---|---|
| `12px` | `my-work-timeline-panel`·`workload-roadmap`·`task-status-page` — 모두 `w-full overflow-hidden` 전폭 대형 섹션 |
| `8px` | aside·카드·오버레이 8곳 (`project-backlog`·`my-work-backlog`·`task-detail-overlay` 등) |

용도로 갈리는 것이 확인돼 둘 다 규약에 넣었다.

**필드 라벨** — `text-xs font-medium text-muted-foreground` 28곳 vs `text-xs text-muted-foreground` 14곳. 전자를 기본값으로.

**섹션 제목** — `text-sm` 17곳으로 우세.

**입력** — `h-9 rounded-[8px]` 9곳 우세(로그인만 `rounded-md`).

## 변경 내용

### 1. `CLAUDE.md` — "디자인 규약 (Figma에 없는 화면)" 섹션 추가

`코드 규약`과 `주의사항` 사이에 넣었다. 첫 줄에서 **Figma에 있으면 그것이 유일한 기준**임을 못박고, 없는 경우의 기본값을 표로 정리했다.

제목 규격이 갈라진 사실도 그대로 적었다.

> **페이지 제목은 두 갈래로 갈라져 있다.** … 둘 다 Figma에서 왔으므로 **기존 화면을 임의로 통일하지 않는다.** 새 화면은 위 표(후자)를 따르고, 통일이 필요하면 디자인 확인을 거쳐 별도 사이클로 다룬다.

규약을 "정답"처럼 쓰면 나중 세션이 기존 화면을 고치러 들 수 있어, **하지 말아야 할 것**을 함께 적었다.

원칙 세 줄도 덧붙였다 — 색상 토큰 사용(패널 그림자만 예외), 읽기 전용 값은 `disabled` 입력이 아닌 텍스트, 폼 액션은 하단 밴드 우측 정렬.

### 2. `src/components/features/me/my-info-page.tsx` — 규약에 맞춤

문서를 쓰면서 **바로 그 페이지가 우세 규약을 2곳에서 벗어난 것**이 드러났다. 규약을 발행하면서 최신 화면이 그것을 어기고 있으면 앞뒤가 맞지 않으므로 함께 고쳤다.

| 위치 | 이전 | 이후 |
|---|---|---|
| 필드 라벨 | `text-xs text-muted-foreground` | `text-xs font-medium text-muted-foreground` |
| `계정` 섹션 제목 | `text-xs font-semibold` (`pb-1`) | `text-sm font-semibold` (`pb-1.5`) |

`계정`은 같은 화면의 `프로필`(`text-sm`)과도 크기가 달라 내부적으로도 어긋나 있었다.

**"기존 화면은 건드리지 않는다"는 결정의 예외다.** 이 페이지는 직전 사이클에서 내가 만든 것이고 Figma 원본이 없어 디자인 확인이 필요 없다. 다른 화면은 그대로 두었다.

## 알려진 이슈 / 주의점

### 규약은 실측값이지 디자인 승인을 받은 값이 아니다

기존 화면에서 다수결로 뽑았다. 디자이너가 다른 값을 의도했을 수 있다. **Figma에 해당 화면 디자인이 생기면 그쪽이 우선**이며, 규약 표도 그때 갱신해야 한다.

### 제목 규격 이원화는 그대로 남는다

`text-xl` 계열(작업 현황·프로젝트 상세)과 `text-[22px]` 계열(내 할일·내 정보)이 공존한다. 사용자가 화면을 옮겨 다닐 때 제목 크기가 미묘하게 달라진다. 통일하려면 디자인 확인 + 4개 화면 동시 수정이 필요해 이번 범위에서 제외했다.

### 규약이 커버하지 않는 것

다이얼로그·오버레이·테이블·빈 상태(empty state)·토스트는 아직 정리하지 않았다. 오버레이는 이미 여러 개 있어(`stage-detail`·`task-detail`·`stage-add`) 실측 대상이 충분하므로, 필요해지면 같은 방식으로 추가하면 된다.

### CLAUDE.md가 길어진다

이미 긴 문서에 섹션이 하나 더 늘었다. 규약이 더 자라면 별도 파일로 분리하고 CLAUDE.md에서는 링크만 두는 편이 나을 수 있다. 지금은 한 화면 분량이라 인라인으로 두었다.
