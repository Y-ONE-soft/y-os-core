# refactor: 단계·할일 막대 색 규칙을 팔레트로 단일화

## 작업 요약

"프로젝트 색을 고르면 화면마다 단계 바·할일 색이 조금씩 다르다"는 문제의 **원인을 제거**한다. 색을 파생하는 체계(`stageTone`/`taskTone`)는 이미 있었지만, **막대를 그리는 규칙이 화면마다 흩어져 있어** 같은 파생 색이 다르게 보였다.

이번 커밋은 규칙을 `project-palette.ts` 한곳에 모으고, **다른 세션이 잡고 있지 않은 파일**부터 그 규칙으로 전환한다.

## 원인 — 파생은 하나, 표현은 둘

전수 조사 결과 색이 어긋나는 지점은 두 갈래였다.

### 1. 막대 표면 규칙이 화면마다 다름

| | 로드맵·타임라인 (`roadmap-bar`) | 내 할일 캘린더 (`my-work-calendar`) |
| --- | --- | --- |
| 채우기 | `hexToRgba(color, 0.12)` | `hexToRgba(color, 0.18)` (강조 0.3) |
| 테두리·링 | `hexToRgba(color, 0.8)` | `hexToRgba(color, 0.8)` |
| 글자 | `color` 원색 | `shade(color, 0.6)` — 캘린더에만 있는 로컬 함수 |

파생 색이 동일해도 채우기 농도가 0.12 vs 0.18, 글자가 원색 vs 어둡게라 **다른 색으로 읽힌다.**

### 2. 단계 없는 할일(백로그)의 색 출처가 다름

- 캘린더: `taskTone(project.color)` — 시그니처는 `stageColor`인데 **프로젝트 색을 넘기고 있었다**
- 타임라인: `project.color` **원색 그대로** — 팔레트를 아예 거치지 않는다

같은 할일이 두 화면에서 다른 색이었다.

## 검증하다 내 주장이 틀린 것을 발견

캘린더의 `shade()`가 RGB 채널에 계수를 곱하는 방식이라, 처음에는 **"RGB 곱셈은 색상(hue)이 틀어진다"** 고 판단하고 그것을 HSL 전환의 근거로 주석에 적었다.

검증 스크립트를 돌리자 **틀렸다.** 채널을 같은 비율로 줄이면 채널 간 비율이 유지되므로 hue는 그대로다 — 실측 Δh는 양쪽 다 1° 미만이었다.

```
--- 색상(hue)은 양쪽 다 보존 (내 최초 주장은 틀렸음) ---
PASS  #8a5bf6  둘 다 Δh<1°
PASS  #27ecab  둘 다 Δh<1°
```

근거를 검증 없이 단정했다가 테스트에 걸린 것이다. 실제 차이를 다시 측정해 주석을 사실에 맞게 고쳤다.

| | `shade(color, 0.6)` | `barTextTone(color)` |
| --- | --- | --- |
| 색상(hue) | 보존 | 보존 |
| **명도 편차** | **7.5pt** (L 32.4~39.8) | **0pt** (전부 L 38.0) |
| **채도** | **−5~−44pt** (색상마다 다름) | 보존 |

즉 HSL을 쓸 이유는 hue가 아니라 **(a) 프로젝트 색이 무엇이든 글자 대비가 일정하고 (b) 색이 탈색되지 않는다** 는 것이다. 채도 감소폭이 초록 −12pt, 보라 −44pt로 들쭉날쭉한 것도 화면 간 인상 차이에 한몫한다.

## 통일 기준 — 양쪽에서 나은 쪽을 취함

두 방식 중 하나를 그대로 고르지 않고 항목별로 정했다.

| 항목 | 채택 | 이유 |
| --- | --- | --- |
| 채우기 `0.12` | 로드맵 쪽 | 더 가볍고, 3개 화면이 이미 쓰고 있어 바뀌는 범위가 작다 |
| 강조 채우기 `0.3` | 캘린더 쪽 | 로드맵에는 강조 상태가 없어 충돌하지 않는다 |
| 테두리·링 `0.8` | 양쪽 동일 | — |
| 글자 **어둡게** | 캘린더 쪽 의도 | 옅은 틴트 위에서 원색은 대비가 모자란다 |
| 어둡게 하는 **방법** | 새로 (HSL) | 위 실측대로 명도 편차 0 · 채도 보존 |

## 변경 파일

### `src/components/features/projects/project-palette.ts`

- **`backlogTone(projectColor)`** — 단계 없는 할일 톤. 계산은 `taskTone`과 같지만 **인자가 단계 색이 아니라 프로젝트 색임을 이름으로 못박는다.** 이 함수가 없어서 캘린더는 잘못된 도메인으로, 타임라인은 원색으로 갈렸다.
- **`barSurface(color, active?)`** — 배경 + 테두리를 **함께** 돌려준다. 하나씩 노출하면 한쪽만 다른 값을 쓰는 일이 다시 생긴다.
- **`barEdge(color)`** — 테두리·링 단독이 필요한 곳(캘린더의 `--tw-ring-color`)용.
- **`barTextTone(color)`** — 막대 위 글자·아이콘 톤. HSL에서 명도 상한만 38로 고정.
- 알파 값 3개(`0.12`/`0.3`/`0.8`)는 모듈 내부 상수로만 두고 **밖으로 내보내지 않는다.** 숫자를 꺼내 쓰면 다시 흩어진다.

`hexToRgba`는 `roadmap-utils.ts`에 그대로 두고 팔레트가 import한다. 색 유틸이므로 팔레트로 옮기는 편이 맞지만, 그러면 `my-work-calendar.tsx` 등 **다른 세션이 수정 중인 파일의 import를 건드려야 해서** 미뤘다. 후속 정리 대상이다.

### `src/components/features/projects/roadmap-bar.tsx`

- 인라인 `hexToRgba(color, 0.12)` / `0.8` → `...barSurface(color)`
- 글자 `style={{ color }}` → `style={{ color: barTextTone(color) }}`

**막대 글자가 원색에서 한 단계 어두워진다** — 프로젝트 상세 로드맵·작업 현황 로드맵·내 할일 타임라인 3개 화면에 보이는 변화다. 채우기 값은 그대로라 배경은 달라지지 않는다.

배지(`backgroundColor: color` + 흰 글자)는 원색을 유지했다. 흰 글자 대비가 필요해 어둡게 하면 안 된다.

### `src/components/features/my-work/my-work-calendar-source.ts`

- `taskTone(project.color)` → `backlogTone(project.color)`

**계산이 같아 시각 변화는 없다.** 의미를 바로잡는 변경이며, 타임라인을 전환할 때 같은 함수를 부르게 하는 것이 목적이다.

**이 커밋만으로는 화면 간 색이 아직 완전히 같아지지 않는다.** 남은 두 곳이 다른 세션의 작업 중인 파일이라 건드리지 않았다.

| 파일 | 남은 문제 | 상태 |
| --- | --- | --- |
| `my-work-calendar.tsx` | 채우기 `0.18`, 로컬 `shade()` | `작업현황-캘린더-내작업-동일화` 세션이 미커밋 수정 중 |

## 조사에서 함께 드러난 것 (이번 범위 밖)

전수 조사 중 확인했으나 이번에 손대지 않은 것들이다.

- **`STAGE_TONE_STEPS`가 6단계뿐이라 `index % 6`으로 7번째 단계가 1번째와 같은 색**이 된다. 같은 화면 안에서의 충돌이라 성격이 다르다.
- **`Stage.color` DB 값은 렌더에 쓰이지 않는데 여전히 저장된다.** 생성 시 한 번 쓰고 갱신되지 않아, 단계 순서를 바꾸거나 프로젝트 색을 바꾸면 **저장값이 낡는다.** 화면은 `board-store`가 매번 다시 파생하므로 문제가 없지만, 낡은 값이 새는 경로가 셋 있다 — ① 프로젝트 색을 못 찾을 때의 폴백 ② **프리셋 스냅샷**(`presets/service.ts`가 DB 색을 그대로 담는다) ③ 첫 페인트.
- 특히 ②는 내가 만든 프리셋 관리 페이지에서 낡은 색이 보일 수 있다는 뜻이다. 프리셋 적용 기능을 만드는 세션이 있으므로 그쪽과 함께 정리하는 편이 낫다.
