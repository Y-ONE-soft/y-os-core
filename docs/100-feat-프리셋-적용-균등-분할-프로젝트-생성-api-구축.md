# feat: 프리셋 적용·균등 분할 프로젝트 생성 API 구축

## 작업 요약

`내 할일`에서 프로젝트를 생성하는 기능의 **서버 기반**을 만들었다. 두 가지 생성 경로를 각각 원자적으로 처리하는 엔드포인트와, 서버·클라이언트가 공유하는 일정 계산 순수 모듈을 추가했다.

이 태스크는 **화면 없이 API까지만** 다룬다. 버튼·다이얼로그·미니 캘린더는 후속 태스크다.

| 경로 | 입력 | 결과 |
| --- | --- | --- |
| 프리셋 적용 | 프리셋 id + 기준일 | 프로젝트 + 단계 + 할일 |
| 직접 만들기 | 시작일·종료일 + 단계 수 | 프로젝트 + 균등 분할된 단계 |

## 설계 판단

### 1. 왜 서버에서 한 번에 만드는가

프리셋 적용은 `프로젝트 1 + 단계 N + 할일 M`을 만든다. 클라이언트에서 기존 단일 API를 나눠 호출하면 **중간 실패 시 단계가 절반만 붙은 프로젝트가 남는다.** 되돌릴 방법도 없다. 그래서 `db.$transaction`으로 묶는 전용 엔드포인트를 만들었다.

### 2. 프리셋은 왜 "시작일 하나"만 받는가

프리셋 도메인(docs/65)이 날짜를 절대값이 아니라 **상대 오프셋**으로 담고 있다.

```prisma
offsetDays   Int?   // 기준일로부터 며칠 뒤 시작
durationDays Int?   // 기간(일)
```

`src/types/preset.ts` 주석에도 `"적용 시 지정한 시작일부터 재현"`이라고 적혀 있다. 즉 기준일만 정하면 전체 일정이 복원되므로, 적용 UI에서 종료일을 물을 필요가 없다.

### 3. 순수 모듈을 `src/lib`에 둔 이유

단계 일정 계산은 **서버(실제 생성)와 클라이언트(생성 전 미리보기)가 같은 결과를 내야 한다.** 그런데 기존 날짜 유틸(`roadmap-utils.ts`)은 `src/components` 아래에 있고, `src/server`는 `@/components`를 import하지 않는 경계를 지키고 있다(확인함 — 위반 사례 0건). 반면 `src/server`가 `@/lib`을 쓰는 선례는 있다(`session.ts` → `@/lib/constants`).

그래서 `src/lib/stage-plan.ts`를 **의존성 없는 순수 모듈**로 만들었다.

### 4. 단계 순서 보존 — id를 정렬 가능하게

워크스페이스 조회 정렬은 `[createdAt asc, id asc]`다. 한 트랜잭션에서 만든 행은 `createdAt(now())`이 같은 값으로 잡힐 수 있고, 그러면 **실질 정렬 키가 id**가 된다. 순수 uuid를 쓰면 프리셋의 단계 순서가 무작위로 섞인다.

```ts
function orderedId(prefix: string, index: number): string {
  return `${prefix}-${String(index).padStart(3, "0")}-${crypto.randomUUID()}`;
}
```

접두사 뒤에 0으로 채운 순번을 넣어 **id 사전순 = 의도한 순서**가 되게 했다. `createdAt` 해상도와 무관하게 순서가 보장된다. 검증에서 프리셋 적용 후에도 `1단계~4단계` 순서가 유지되는 것을 확인했다.

### 5. 단계 색

보드 화면은 **DB에 저장된 단계 색을 쓰지 않고 프로젝트 색에서 파생**한다(`board-store.tsx`의 `withDerivedColors` — "저장값을 쓰면 프로젝트 색을 바꿨을 때 옛 색이 남아 다시 어긋난다"). 따라서 저장하는 색은 표시에 영향이 없다.

- 프리셋 적용: 프리셋이 스냅샷한 색을 그대로 저장 (원본 보존)
- 직접 만들기: 프로젝트 색을 그대로 저장

`stageTone()`은 `src/components` 아래에 있어 서버에서 쓸 수 없다. 표시에 영향이 없으므로 굳이 옮기지 않았다.

## 변경 파일 내역

### 신규 `src/lib/stage-plan.ts`

의존성 없는 순수 계산 모듈.

| 함수 | 역할 |
| --- | --- |
| `addDaysISO` / `inclusiveDays` | `YYYY-MM-DD` 하루 단위 연산. **UTC 기준**으로 계산해 로컬 타임존 DST 보정이 날짜를 미는 것을 막는다 |
| `evenSplitError` | 균등 분할 거부 사유. 라우트와 화면이 **같은 기준·같은 문구**를 쓰도록 공유 |
| `splitRangeEvenly` | 기간을 N개로 분할. 나머지 일수는 **앞 단계부터** 하루씩 배분 |
| `presetStageSpan` | 프리셋 오프셋 → 실제 날짜 |

나머지 일수를 앞에서부터 주는 이유: 뒤에 몰아주면 마지막 단계만 길어져 로드맵이 한쪽으로 치우쳐 보인다. (30일/4단계 → `8, 8, 7, 7`)

`offsetDays`가 없던 단계는 날짜 없이 그대로 둔다 — 프리셋이 담은 "날짜 미정" 상태를 보존한다. `durationDays`가 없으면 종료일을 만들지 않는다(진행형 막대).

### 신규 `src/server/workspace/compose.ts`

`createProjectFromPreset`, `createProjectWithEvenStages`.

`service.ts`가 아니라 **별도 파일**로 둔 이유는 두 가지다. (1) `service.ts`의 단일 행 헬퍼(`createStage`/`createTask`)는 각자 `db`를 직접 호출해 트랜잭션에 참여시킬 수 없다. (2) `service.ts`는 병렬 세션이 자주 건드리는 파일이라 신규 코드를 분리하면 충돌 면적이 줄어든다.

프리셋 소유자 검증은 별도로 하지 않는다 — `getPreset(ownerId, presetId)`가 `ownerId`를 조회 조건에 함께 걸어 **남의 프리셋 id는 not found**가 된다.

직접 만들기는 할일을 만들지 않는다. 뼈대만 잡고 내용은 사용자가 채우는 흐름이다.

### `src/app/api/admin/guard.ts`

- `isISODate` — `YYYY-MM-DD` 형식 검증
- `resolveProjectGroupId` — 스탭은 세션 그룹 강제, 마스터는 요청 `groupId` 필수. 기존 `POST /api/admin/projects`의 규칙을 그대로 옮긴 것으로, 새 라우트 2개가 같은 로직을 중복하지 않게 했다.

### 신규 라우트

- `POST /api/admin/projects/from-preset` — `{id, groupId?, name, color, presetId, baseDate}`
- `POST /api/admin/projects/even-stages` — `{id, groupId?, name, color, startDate, endDate, stageCount}`

`even-stages`는 `evenSplitError`로 먼저 검증해 **화면 안내와 같은 문구**를 400으로 돌려준다.

### `src/lib/api/workspace.ts`

`createProjectFromPresetApi`, `createProjectWithEvenStagesApi` 추가.

## 검증

### 순수 함수 — 경계값 24건

`node --experimental-strip-types`로 모듈을 직접 로드해 확인했다. **전부 통과.**

| 묶음 | 확인 내용 |
| --- | --- |
| 날짜 연산 | 같은 날 = 1일, 월말 넘김, **윤년**(2024-02-28 +1 = 02-29), 연말 넘김 |
| 균등 분할 | 30일/3단계 = 각 10일, 30일/4단계 = `8,8,7,7` |
| 경계 | 1단계 = 전체 기간, 일수 == 단계수 → 각 1일, 하루 기간 |
| 무결성 | 1년/7단계·43일/5단계에서 **총일수 일치 + 빈틈·겹침 없이 연속**, 마지막 단계 끝 = 종료일 |
| 거부 | 단계수 0, 소수 단계수, 종료 < 시작, 단계수 > 일수 |
| 프리셋 복원 | offset+duration, duration 없음(진행형), offset 없음(미정) |

### API — 실제 DB 종단 14건

dev 서버(`localhost:3023`, 본 워크트리 전용 포트)에 로그인해 실제로 호출했다. **전부 통과.**

- 균등 분할 생성 → 단계 4개, 날짜 `8/1~8/8 | 8/9~8/16 | 8/17~8/23 | 8/24~8/30`, 이름 `1~4단계` 순서 보존
- 단계수 > 일수 → 400 + `"기간이 3일이라 단계를 9개로 나눌 수 없습니다. 단계 수를 3개 이하로 줄이세요."`
- 잘못된 날짜 형식(`2026/08/01`) → 400
- **왕복 테스트**: 만든 프로젝트로 프리셋 생성 → 기준일 `2026-10-05`로 적용 → `10/5~10/12 | 10/13~10/20 | 10/21~10/27 | 10/28~11/3`로 **간격이 그대로 재현**되고 단계 순서도 유지
- 없는/남의 프리셋 id → 400 `"프리셋을 찾을 수 없습니다."`
- 비로그인 → 401

### 공유 DB 정리

개발 DB는 전 세션이 공유하므로 테스트가 만든 프로젝트 2건·프리셋 1건을 `finally`에서 삭제하고, **워크스페이스를 다시 조회해 `[테스트]` 접두 데이터가 0건임을 확인**했다.

- `npm run lint` 통과
- `npm run build` 통과

## 알려진 이슈 / 후속

- **후속 태스크가 남아 있다.** 이 커밋은 API까지다. (2) 드래그 기간 선택 미니 캘린더, (3) 생성 버튼 + 프리셋 적용 모드, (4) 직접 만들기 모드가 이어진다.
- **날짜 유틸이 두 벌이다.** `src/lib/stage-plan.ts`의 `addDaysISO`와 `roadmap-utils.ts`의 `shiftISO`가 같은 일을 한다. 레이어 경계 때문에 서버가 후자를 쓸 수 없어 생긴 중복이다. `roadmap-utils`의 순수 날짜 부분을 `@/lib`으로 내리고 양쪽이 공유하게 하는 정리가 가능하나, 그 파일은 병렬 세션이 여럿 쓰고 있어 이번에는 건드리지 않았다.
- **기존 `POST /api/admin/projects`는 아직 `resolveProjectGroupId`를 쓰지 않는다.** 같은 로직이 인라인으로 남아 있다. 그 파일의 충돌 면적을 늘리지 않으려 이번에는 새 라우트에만 적용했다. 별도 정리 대상.
- 직접 만들기의 단계 이름은 `1단계`, `2단계`… 고정이다. 이름 편집은 생성 후 보드에서 한다.
- docs 번호는 작성 시점 main 최대 번호(85) + 1로 86을 썼다. 병렬 작업이 많아 머지 순서에 따라 조정이 필요할 수 있다.
