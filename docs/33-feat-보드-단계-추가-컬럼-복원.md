## 1. 작업 내용 요약

프로젝트 상세 보드(칸반)에 **단계 추가 진입 UI가 없던 것**을, Figma `Project Detail Layout — Stage Add`(**113:333**)의 `단계 추가 (대기)` 컬럼(**113:452**)으로 복원했다. 마지막 단계 컬럼 뒤에 점선 컬럼이 놓이고, 클릭하면 기존 단계 추가 다이얼로그가 열린다.

### 이력 — 추가 → 제거 → 복원

| 시점 | 문서 | 내용 |
|---|---|---|
| 최초 | docs/11 | 디자인 113:333 기준으로 점선 단계 추가 컬럼 구현 |
| 제거 | docs/18 (PR #12) | 기준 프레임 `Project Detail Layout`(96:166)에 해당 컬럼이 없어 **제거** — "진입점은 로드맵 헤더 ＋단계, 로드맵 하단 ＋단계 추가 행 2곳" 으로 정리. 당시 문서에 *"검토 시 이견 있으면 되돌림"* 명시 |
| 복원 | 이 문서 | 사용자 요청 — 보드에도 진입 UI 필요. **96:166은 기본 상태, 113:333은 단계 추가 상태**로 두 프레임이 별개 상태였던 것이 정리의 근거 |

### 검증 방법론 교훈 (중요)

보드 액션은 **낙관적 UI 갱신 + fire-and-forget POST** 구조다(`board-store.tsx` → `cache.persist`). 따라서 *"화면에 나타났다"는 저장을 보장하지 않는다* — 초기 검증에서 화면 확인 직후 브라우저를 닫자 dev 라우트 최초 컴파일 중이던 POST가 중단되어 DB에 아무것도 남지 않았다(영속화 버그로 오인할 뻔했다). 검증 스크립트에 **① POST 응답 200 대기 ② 새로고침 후 잔존 확인** 두 단계를 추가해 해결했다. 이후 보드 관련 검증은 이 두 가지를 반드시 포함할 것.

## 5. 알려진 사항 / 후속 과제

- **단계가 3개 이상이면 추가 컬럼이 기본 화면에서 12px만 노출된다** — 보드 폭 822px에 컬럼 260px+간격 10px이라 3단계부터 총 1070px로 가로 스크롤이 생긴다(디자인 프레임은 단계 2개 기준이라 이 상황이 없다). 끝까지 스크롤하면 260px 전체가 보이는 것은 확인했다(일반적인 칸반 동작). **항상 보이게 고정(sticky)할지 여부는 사용자 판단 대기** — 디자인에는 고정 규정이 없어 임의 변경하지 않았다
- 컬럼 헤더의 `⋯`(단계 메뉴)는 여전히 시각만 — 후속 태스크
- 진입점이 3곳(로드맵 헤더 ＋단계, 로드맵 하단 ＋단계 추가 행, 보드 점선 컬럼)이 되었다. 모두 같은 다이얼로그를 열며 디자인상 의도된 구성이다
