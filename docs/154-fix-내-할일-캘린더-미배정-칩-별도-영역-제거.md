# fix: 내 할일 캘린더 미배정 칩 별도 영역 제거

## 배경 / 문제

내 할일 캘린더에서 프로젝트 없는(미배정) 할일 칩이 `UNASSIGNED_BOX`라는 가짜
프로젝트 키로 묶여, 배치 모듈(`my-work-calendar-layout`)이 **실제 프로젝트와 똑같이
박스 블록**을 잡아 줬다. 그 박스는

- 맨 위에 **빈 단계줄(STAGE_LANES)** 한 줄을 예약하고,
- 최소 높이 2줄(MIN_BOX_LANES)을 차지하며,
- 프로젝트 박스들 **아래에 하나의 묶음(별도 영역)** 으로 앉았다.

PR #140(docs/150)은 이 묶음의 **배경(meta 등록)만** 없앴을 뿐, 레이아웃이 잡던
별도 영역 자체는 그대로 남아 있었다. 그래서 미배정 칩들이 여전히 날짜 칸 하단에
한 덩어리로 쌓여 보였다. 사용자 요청: "미배정은 별도의 프로젝트 영역을 만들지 말라."

## 변경 내용

미배정 칩을 **박스로 묶지 않고**, 그 날짜 열이 비는 **첫 레인에 하나씩 자유 배치**하도록
배치 규칙을 바꿨다. 프로젝트 박스가 레인을 먼저 잡으므로 미배정 칩은 자연히 그 아래
빈 레인으로 채워지고, 프로젝트가 없는 날은 맨 위 레인(lane 0)부터 쓴다. 빈 단계줄·
최소 높이 같은 예약이 사라져 별도 영역이 생기지 않는다.

### `src/components/features/my-work/my-work-calendar-layout.ts`

- `UNASSIGNED_BOX = "__unassigned__"` 상수를 이 배치 모듈로 **이관**(원 소유가 여기로).
  미배정 칩을 식별하는 키이며, 배치 규칙의 소유가 배치 모듈에 있는 게 자연스럽다.
- `layoutWeek`의 박스 배치 대상(`active`)에서 `UNASSIGNED_BOX`를 **제외**한다.
- 박스 배치 루프 뒤에, `project === UNASSIGNED_BOX`인 오버레이(칩)를 순회하며
  그 날짜 열이 겹치지 않는 **첫 레인**을 찾아 하나씩 앉히는 로직을 추가했다.
  (span 1짜리 칩이므로 같은 레인에 다른 날짜의 칩이 함께 들어갈 수 있다.)

### `src/components/features/my-work/my-work-calendar-source.ts`

- `UNASSIGNED_BOX` 로컬 상수 정의를 없애고 배치 모듈에서 import,
  기존 import 경로(`my-work-calendar` 등이 source에서 가져감) 호환을 위해 **재노출**한다.
- 미배정 칩 생성부의 주석을 자유 배치(별도 영역 없음) 기준으로 갱신.

## 검증

로컬 dev(워크트리, 포트 3061)에서 Playwright로 확인.

- 과거 예정일은 `rollOverOverdueTasks`가 오늘로 당기므로, 워크스페이스에서 어떤 단계·
  할일도 커버하지 않는 **미래의 빈 날**(자동 탐색, 이번 실행 2026-09-07)을 골라 미배정
  할일을 만들고 그 달로 이동해 칩을 확인했다.
- **미배정 칩이 캘린더에 뜸** — OK
- **빈 날의 미배정 칩이 맨 위 레인(lane 0)에 앉음** — 칩 컨테이너 인라인 `top="24px"`
  (= laneTop(0)). 구 동작이면 빈 단계줄 위에 얹혀 `top="52px"`(lane 1)이었을 값.
- **미배정 박스 이름표 없음** — "미배정" 라벨 0개.
- 스크린샷(`verify-area.png`): 9월 7일 칸 맨 위에 칩 하나가 단독으로 붙어 있고, 별도
  묶음·빈 예약 영역이 없다.
- `npx tsc --noEmit` 통과.

## 알려진 이슈 / 참고

- 병렬 세션 두 개(`캘린더-마감-깃발`, `캘린더-가로-간격`)가 `my-work-calendar.tsx`를
  미커밋 수정 중이다. 이 태스크는 그 파일을 건드리지 않아(레이아웃·소스만) 충돌면이 없다.
- `my-work-calendar.tsx`에 남아 있던 `UNASSIGNED_BOX` 분기(`ProjectBoxItem`의 링크 대신
  span, `handleAddClick`의 `box.project === UNASSIGNED_BOX`)는 이제 미배정이 박스로
  절대 안 잡히므로 도달하지 않는 dead code가 됐다. 위 병렬 작업과의 충돌면을 줄이려고
  이번엔 그대로 뒀다(무해). 다음 사이클에서 정리 가능.
