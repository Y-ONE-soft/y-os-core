## 1. 작업 내용 요약

프로젝트 상세 **페이지 헤더 우측**에 `프리셋 사용하기` · `프리셋 저장하기` 두 버튼을 놓았다.

위치는 사용자 지정대로 **내 할일의 `프로젝트 생성`과 같은 자리**다 — 페이지 헤더가 `flex items-start justify-between`이고 제목 블록이 왼쪽, 액션이 오른쪽에 붙는 구조 (`my-work-page.tsx`의 `<ProjectCreateButton />` 자리).

## 3. 버튼 활성 규칙

| 버튼 | 활성 조건 | 비활성 시 툴팁 |
|---|---|---|
| 프리셋 사용하기 | 단계가 **하나도 없을 때** | "이미 단계가 있어 프리셋을 적용할 수 없습니다" |
| 프리셋 저장하기 | 단계가 **하나라도 있을 때** | "저장할 단계 구성이 없습니다" |

두 버튼의 조건이 정확히 반대다 — 빈 프로젝트는 채우는 쪽만, 구성이 있는 프로젝트는 저장하는 쪽만 의미가 있다. 서버도 적용 요청을 같은 규칙으로 최종 판정한다(docs/114).

## 4. 기존 '프리셋 저장' 버튼 이동

로드맵 카드 헤더(`＋단계`·레인지 스위처 옆)에 있던 `프리셋 저장` 버튼을 **제거하고 페이지 헤더로 일원화**했다 (사용자 결정).

같은 기능 진입점이 두 곳이면 어느 쪽이 무엇을 저장하는지 헷갈린다. 저장 대상은 어차피 프로젝트 전체 단계 구성이므로 페이지 단위 액션이 맞다. 쓰이지 않게 된 `onSavePreset` prop과 `SlidersHorizontal` import도 함께 정리했다.

## 5. 적용 다이얼로그

프로젝트 생성 다이얼로그의 프리셋 모드에서 **생성에만 필요한 입력(이름·색·그룹)을 덜어낸 형태**다. 프로젝트는 이미 있으므로 프리셋 선택과 기준일만 받는다.

- 프리셋 목록: `fetchPresets()` — 없으면 "저장된 프리셋이 없습니다" 안내
- 기준일: 기본값 오늘. 프리셋의 일정은 이 날짜 기준 상대 오프셋으로 재현된다
- 적용 성공 시 **먼저 닫고** 워크스페이스 캐시를 갱신한다 — 갱신을 기다렸다가 닫으면 실패 시 창이 남아 사용자가 다시 눌러 중복 적용한다 (프로젝트 생성 다이얼로그와 같은 처리)

## 6. 검증

`npm run lint` ✓ · `npx tsc --noEmit` ✓ · `npm run build` ✓

**브라우저 실검증** — dev 서버(워크트리 전용 포트 3030, 점유 프로세스 CommandLine으로 자기 워크트리 확인), 실제 로그인(`master01`). 검증용 프로젝트 2개(빈 것 / 단계 있는 것)를 만들어 대조했다.

| # | 검증 | 결과 |
|---|---|---|
| 1 | 단계 있는 프로젝트 헤더 | `프리셋 사용하기` **disabled** · `프리셋 저장하기` enabled |
| 2 | 빈 프로젝트 헤더 | `프리셋 사용하기` enabled · `프리셋 저장하기` **disabled** |
| 3 | 로드맵 카드에 `프리셋 저장` 잔존 여부 | **없음** (헤더로 이동 완료) |
| 4 | `프리셋 저장하기` 클릭 | 저장 다이얼로그 열림 |
| 5 | `프리셋 사용하기` 클릭 | 적용 다이얼로그 열림 |
| 6 | 빈 프로젝트에 적용 | **200** |
| 7 | 단계가 생긴 뒤 재적용 | **400** `이미 단계가 있는 프로젝트에는 프리셋을 적용할 수 없습니다.` |
| 8 | 적용 후 헤더 재확인 | 버튼 상태가 **뒤집힘**(사용 disabled / 저장 enabled) — 스토어 갱신 반영 확인 |

기록된 콘솔 400 에러는 7번 **의도적 실패 케이스**가 낸 것이다. 검증용 프로젝트·프리셋은 스크립트가 만들고 끝나면 삭제했다(삭제 status 200×3).

## 7. 알려진 사항 / 후속 과제

- **적용 성공 토스트가 없다.** `sonner`가 마운트돼 있지 않아(루트 레이아웃 변경은 병렬 세션과 충돌) 다이얼로그가 닫히고 로드맵에 단계가 나타나는 것으로 결과를 확인한다.
- **버튼 라벨이 길다.** 헤더가 좁은 화면에서는 제목과 겹칠 수 있다. 반응형 축약(아이콘만 표시 등)은 넣지 않았다.
- 프로젝트 생성 다이얼로그의 프리셋 선택 UI와 적용 다이얼로그의 목록이 **거의 같은 마크업**이다. 세 번째 사용처가 생기면 공용 `PresetPicker`로 뽑는 게 맞다.
