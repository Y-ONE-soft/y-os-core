## 3. 설계 결정과 이유

**요청 스토어를 모듈 수준 구독형으로 둔 이유** — 보내는 화면(오버레이)과 받는 화면(내 할일 요청 알림)이 동시에 떠 있을 수 있다. 한쪽에서 보내거나 응답하면 다른 쪽도 같이 갱신돼야 해서 `useSyncExternalStore` 기반 스토어 하나를 공유한다. `useUsers`처럼 단순 캐시로 두면 갱신 전파가 안 된다.

**`initialSelected` → `alreadyRequested`로 바꾼 이유** — 이전에는 현재 공동작업자를 체크된 상태로 넘겨 "선택 목록 = 최종 상태"였다. 이제는 요청 모델이라 **새로 고른 사람만 발송 대상**이어야 한다. 기존 대상(이미 공동작업자이거나 응답 대기 중)은 체크·비활성 + "요청됨"으로 표기해 중복 발송을 막았다.

**단계 추가에서 생성 직후 발송하는 이유** — 요청은 대상(`stageId`)이 있어야 만들 수 있는데 생성 전에는 단계가 없다. 그래서 로컬로 모아두고 `addStage`가 돌려준 id로 한 번에 보낸다. 동시에 `addStage`의 `requestedCollaborators` 사전 지정은 **제거**했다 — 그대로 두면 상대 동의 없이 공동작업자가 되어 요청 모델과 모순된다.

**메시지 입력이 실제로 전달되도록 연결** — 할일 상세의 요청 모달 `Textarea`는 `value`/`onChange`가 없는 비제어 요소라 입력해도 어디에도 담기지 않았다. 상태를 붙여 요청의 `message`로 보낸다.

## 5. 작업 중 겪은 함정 (재현 시 참고)

**dev 서버를 마이그레이션보다 먼저 띄우면 `db.request`가 `undefined`가 된다.** 발송·조회가 모두 500이 나고 로그에 `Cannot read properties of undefined (reading 'createMany')`가 찍힌다.

`predev` 훅의 `prisma generate`는 **서버 시작 시점에 한 번만** 돌기 때문에, 서버가 뜬 뒤 추가한 모델은 생성 클라이언트에 없다. Turbopack HMR도 생성 파일 변화는 따라오지 않는다. **`prisma generate` 후 dev 서버 재시작**이 필요하다.

## 6. 알려진 사항 / 후속 과제

- **발송 성공 피드백이 약하다.** `sonner`가 설치돼 있지만 `<Toaster />`가 어디에도 마운트돼 있지 않다. 마운트하려면 루트 레이아웃을 건드려야 해서(다른 세션과 충돌 위험) 이번엔 넣지 않았다. 대신 단계 상세에 "응답 대기 중 · 이름" 줄을 두어 보낸 상태를 확인할 수 있게 했다.
- **할일 상세에는 대기 표기가 없다.** 단계 상세에만 넣었다 — 할일 상세의 요청 버튼은 사이드바에 있고 그 아래 공간이 이미 빽빽하다.
- 요청 스토어는 **폴링하지 않는다.** 상대가 보낸 요청은 화면을 다시 열거나 새로고침해야 보인다. 실시간 알림은 별도 사이클 과제다.
