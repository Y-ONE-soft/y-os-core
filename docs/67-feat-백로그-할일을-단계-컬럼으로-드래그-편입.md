# 67. 백로그 할일을 단계 컬럼으로 드래그 편입

- **예정 커밋 메시지**: `feat: 백로그 할일을 단계 컬럼으로 드래그 편입`
- **작업일**: 2026-07-22
- **작업 브랜치**: `백로그-단계-드롭` (워크트리 `.claude/worktrees/backlog-drop-stage`, base: main `2cac16b`)

---

## 1. 작업 요약

프로젝트 상세에서 **백로그의 할일을 보드의 단계 컬럼으로 끌어다 놓으면 그 단계로 편입**된다. 화면에 있던 안내 문구("백로그 → 단계로 드래그하면 편입됩니다")가 이제 실제로 동작한다.

편입 시 **예정일은 기존 규칙 그대로** 잡힌다 — `assignTask`가 `max(단계 시작일, 오늘)`로 계산한다(docs/51). 드롭다운으로 단계를 고르는 기존 경로와 결과가 같다.

## 2. 구현

HTML5 드래그 앤 드롭을 썼다. 캘린더·로드맵의 포인터 기반 드래그(막대를 날짜 축 위에서 미세 조정)와 달리, 여기는 **목록 → 컨테이너**로 옮기는 조작이라 브라우저 기본 DnD가 자연스럽다.

| 파일 | 역할 |
|---|---|
| `task-drag.ts` (신규) | 드래그 데이터 규약 — 전용 MIME(`application/x-yos-task`)으로 외부 텍스트·파일 드롭을 배제. `setTaskDragData` / `isTaskDrag` / `getTaskDragData` |
| `project-backlog.tsx` | 백로그 행을 `draggable`로, 드래그 시작 시 작업 id 실음. grab 커서와 안내 title |
| `project-board.tsx` | 단계 컬럼에 `onDragOver`/`onDragLeave`/`onDrop`. 드롭 시 `assignTask(projectId, taskId, projectId, stage.id)` |

**세부 처리 두 가지**

- `dragover`에서는 보안상 데이터 값을 읽을 수 없어 **타입 목록만** 확인해 하이라이트를 판정한다(`isTaskDrag`). 실제 id는 `drop`에서 읽는다
- `dragleave`는 자식 요소로 이동할 때도 발생한다. `currentTarget.contains(relatedTarget)`로 **컬럼 밖으로 나간 경우에만** 하이라이트를 끈다

드롭 대상 컬럼은 `ring-2 ring-primary`로 표시해 어디에 놓이는지 보이게 했다.

## 5. 사후 검증 결과 (추록)

- 최신 main(작업 완료날짜 PR #59, 상세 헤더 브레드크럼 통일 PR #60 포함) 머지 후 **드롭 회귀 재검증 2회 연속 통과** — 백로그 → 단계 편입, 예정일 = 단계 시작일
- 문서 번호는 병렬 세션이 66을 먼저 머지해 **67로 조정**(선행 커밋)
- 회귀 검증이 한 번 실패했는데 원인은 앱이 아니라 **테스트의 하이드레이션 대기 부족**이었다 — `draggable` 속성은 마크업에 먼저 나타나므로 요소가 있다고 해서 React 핸들러가 붙었다는 보장이 없다. 계측(dragstart 후 dataTransfer 타입·drop의 defaultPrevented·저장 결과)으로 앱이 정상임을 확인하고, 대기를 늘려 2회 연속 통과로 재현성을 확보했다
- 프로덕션 배포 success 확인은 머지 후 사용자 보고로 기록

## 6. 알려진 사항 / 후속 과제

- **반대 방향(보드 카드 → 백로그)은 아직이다.** 같은 안내 문구가 약속하고 있고 `task-drag.ts` 규약을 그대로 쓰면 되므로 후속으로 짧게 붙일 수 있다
- 단계 컬럼 **사이의 카드 이동**(A단계 → B단계)도 같은 방식으로 확장 가능하다. 지금은 드롭 대상만 열려 있고 보드 카드는 아직 `draggable`이 아니다
- 컬럼 안에서의 **순서 바꾸기**는 범위 밖(작업에 정렬 필드가 없다)
- 내 작업 페이지의 통합 백로그에는 적용하지 않았다 — 거기는 캘린더로 끄는 조작이 따로 있다
