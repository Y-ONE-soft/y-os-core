# feat: 내 정보 페이지 추가 및 유저 메뉴 연결

요청 사이클 `내-정보-페이지`의 2번(마지막) 태스크. 1번 태스크에서 만든 API를 화면에 연결한다. 이 커밋으로 요청이 완결된다.

## 변경 내용

### 1. `src/components/features/auth/session-context.tsx` — `setUser` 노출

```ts
/**
 * 내 정보 수정 후 헤더·아바타가 즉시 옛 값을 벗도록 컨텍스트를 갈아끼운다.
 * 서버가 돌려준 SessionUser를 그대로 넣으므로 재조회가 필요 없다.
 */
setUser: (user: SessionUser) => void;
```

상태 setter가 이미 있었으므로 컨텍스트 값에 얹기만 했다. `fetchMe()`를 다시 부르는 방식은 왕복이 한 번 더 늘고 그 사이 헤더가 옛 이름을 보여준다.

### 2. `src/app/(main)/me/page.tsx` (신규)

`(main)` 라우트 그룹이라 헤더·사이드바 셸을 그대로 쓴다. 라우팅 진입점만 두고 화면은 features에 위임하는 기존 컨벤션(`my-tasks/page.tsx`)을 따랐다.

### 3. `src/components/features/me/my-info-page.tsx` (신규)

**컴포넌트를 둘로 나눴다.** 바깥은 세션 로딩을, 안쪽 `MyInfoForm`이 폼 상태를 맡는다.

```tsx
// 세션이 도착한 뒤에 폼을 마운트해 초기값을 그 자리에서 넣는다. 이펙트로
// 나중에 채우면 첫 렌더가 빈 폼이고, 상태 동기화 규칙도 복잡해진다.
// key로 계정이 바뀌면(재로그인) 폼이 통째로 새로 만들어진다.
return <MyInfoForm key={user.id} user={user} setUser={setUser} />;
```

처음에는 `useEffect`로 세션 도착 시 폼을 채웠는데 **린트가 `react-hooks/set-state-in-effect`로 잡았다.** 규칙을 끄는 대신 구조를 바꿨다 — 세션이 없는 동안은 스켈레톤을 렌더하고, 도착한 뒤 `useState(() => …)`로 초기화한다. 이펙트도, 동기화 규칙도 사라진다.

**바뀐 필드만 보낸다.**

```ts
// 값이 실제로 바뀐 필드만 보낸다 — 안 바뀐 이메일까지 보내면 중복 검사가
// 불필요하게 돌고, 서버가 받은 필드를 그대로 덮어쓰는 것도 피한다.
```

`dirty`가 false면 저장 버튼이 비활성이라 의미 없는 요청이 나가지 않는다. 저장 성공 후 `setUser(updated)`로 컨텍스트가 갱신되면 `changed()`가 다시 계산돼 버튼이 자동으로 비활성이 된다 — 별도 리셋 로직이 필요 없다.

**읽기 전용 필드는 입력처럼 보이지 않게 했다.**

```tsx
{/* 입력처럼 보이면 고칠 수 있다고 오해한다 — 배경 없는 텍스트로 둔다 */}
```

아이디·역할·소속 그룹은 테두리·배경 없는 텍스트로 표시한다. `disabled` 입력으로 두면 "왜 못 고치지"를 유발한다.

**소속 그룹 이름은 스토어에서 찾는다.** 세션에는 `groupId`("g-soft")만 있고 이름("Soft")이 없다.

```ts
// 그룹 이름은 세션에 없어 프로젝트 스토어에서 찾는다. 스토어가 늦게 로드되는
// 동안 groupId("g-soft")를 그대로 보여주면 내부 id가 사용자에게 노출되므로,
// 로드 전에는 빈 값으로 둔다.
```

이건 검증 중 스크린샷에서 실제로 발견했다 — 아래 검증 절 참고.

상태 메시지는 인라인 텍스트다. `sonner`가 의존성에 있지만 `Toaster`가 어디에도 마운트돼 있지 않아, 토스트를 쓰려면 레이아웃까지 건드려야 한다. 요청 범위 밖이라 폼 옆 `role="status"` 문구로 처리했다.

### 4. `src/components/layout/user-menu.tsx`

"내 정보" 항목에 `onSelect={() => router.push("/me")}` 추가. `useRouter` import.

## 검증

`npm run build` 성공, `npm run lint` 경고 0.

**헤드리스 Chrome + CDP로 전체 흐름을 실제로 몰았다**(docs/74에서 만든 방식, 새 의존성 없음).

### 전체 흐름

```
드롭다운 항목: ["알림2","내 정보","프리셋 관리","설정","로그아웃"]
이동한 경로: /me
저장 결과: 저장했습니다.
헤더 표시 이름: 검검증용 이름          ← 아바타 이니셜 "검" + 이름
저장 버튼 비활성(변경 없음): true
서버 값: 검증용 이름 / 책임
```

아바타 클릭 → 드롭다운 → "내 정보" 클릭 → `/me` 이동 → 이름·직책 수정 → 저장 → **헤더가 즉시 새 이름으로 바뀌고** 저장 버튼이 비활성으로 돌아간다. 서버 값도 일치한다.

스크린샷으로 카드 헤더가 `검증용 이름 / 스탭 · Soft`, 소속 그룹이 `Soft`로 표시되는 것을 확인했다.

### 검증 중 발견하고 고친 것 — 내부 id 노출

첫 스크린샷에서 **소속 그룹이 `g-soft`로 보였다.** 프로젝트 스토어가 아직 로드되지 않아 폴백인 `user.groupId`가 그대로 나온 것이다.

```
1500ms 시점: (렌더 전)
3000ms 시점: Soft
6000ms 시점: Soft
```

로드 후에는 정상이지만 **그 전까지 내부 id가 사용자에게 노출된다.** 폴백을 빈 값으로 바꿔 해소했고, 재확인 결과 id가 더 이상 보이지 않는다.

### 정리

검증 중 바꾼 계정 정보는 시드 값(`스탭 / 사원 / step01@y-os.local / 010-0000-0002`)으로 복구했다.

## 알려진 이슈 / 주의점

### 나머지 메뉴 항목은 여전히 죽어 있다

`알림`·`프리셋 관리`·`설정`은 `onSelect`가 없어 눌러도 반응이 없다. 요청 범위 밖이라 두었으나, **"내 정보"만 동작하게 되면서 나머지가 죽어 있는 게 더 도드라진다.** 각각 페이지가 필요해 별도 사이클이 맞다.

### 비밀번호를 바꿀 수 없다

1번 태스크에서 제외했다. 시드 비밀번호가 `1111`로 고정이고 변경 수단이 없으므로 **운영 전 필수 작업**이다.

### 이메일 형식을 서버가 검증하지 않는다

`input[type=email]`이 브라우저에서 1차로 걸러주지만 API 직접 호출은 막지 못한다. 이메일이 로그인 식별자로도 쓰이므로 서버 검증을 붙일 가치가 있다.

### 저장하지 않고 페이지를 벗어나면 경고가 없다

수정 중 다른 메뉴로 이동하면 변경분이 조용히 사라진다. `beforeunload`나 라우트 이탈 확인은 붙이지 않았다.

### 그룹 이름 조회가 프로젝트 스토어에 의존한다

내 정보 화면이 워크스페이스 데이터 로드를 기다리는 구조다. 세션 페이로드에 그룹 이름을 포함시키면 의존이 사라지지만, `toSessionUser`가 관계를 함께 조회해야 해서 로그인·세션 조회 쿼리에 모두 영향을 준다. 이번엔 화면 쪽 폴백으로 처리했다.
