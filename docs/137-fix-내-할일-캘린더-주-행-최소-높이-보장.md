# fix: 내 할일 캘린더 주 행 최소 높이 보장

## 증상

step01로 로그인해 `내 할일` 캘린더를 열면 "엄청 작고 하나도 안 보인다"는 제보. 재현 결과 **화면(뷰포트)이 낮을 때** 캘린더의 각 주 행이 극단적으로 눌렸다.

- 1280×**700** 뷰포트: 각 주 행이 **59px**로 압축 (5주가 균등 분할, 스크롤 없음)
- 59px면 날짜 숫자와 여백만으로도 빡빡해 할일 칩이 들어갈 자리가 없다 → "안 보임"

큰 뷰포트(1440×900)에서는 정상으로 보여, 화면이 작은 환경에서만 드러나는 문제였다.

## 원인

주 행은 `flex w-full flex-1`이고, 최소 높이를 **내용이 있을 때만** 줬다.

```ts
const minHeight = laneCount > 0 ? DATE_ROW_PX + laneCount * LANE_PX + 8 : undefined;
```

캘린더 컨테이너는 `flex-1`이라 화면 높이에 맞춰 정해지는데, 그 안의 주 행들이 `flex-1`이라 **컨테이너 높이를 균등 분할**한다. 이 달에 할일이 없어 `laneCount = 0`이면 `minHeight`가 없으므로, 낮은 화면에서는 5~6주가 좁은 공간을 나눠 갖아 각 59px까지 찌그러진다.

## 수정

`src/components/features/my-work/my-work-calendar.tsx` — 내용 유무와 무관하게 주 행에 **최소 높이**를 보장한다.

```ts
const WEEK_MIN_PX = 104; // 날짜 숫자(24) + 할일 두 줄 남짓 여유

const minHeight = Math.max(
  WEEK_MIN_PX,
  DATE_ROW_PX + laneCount * LANE_PX + 8,
);
```

- 빈 달도 각 주가 최소 104px를 확보해 날짜·할일이 찌그러지지 않는다.
- 내용이 많은 주는 종전대로 그만큼 커진다(`Math.max`의 두 번째 항).
- 6주 합계가 캘린더 컨테이너보다 크면 컨테이너(`overflow-y-auto`)가 스크롤한다 — 각 칸 크기를 지키는 대신 넘치는 주는 스크롤로 본다.

## master와의 통일

`내 할일` 캘린더는 `MyWorkCalendar` 한 컴포넌트이고 step01(스탭)·master(마스터)가 **같은 코드를 쓴다.** 재현에서도 두 계정의 레이아웃 높이가 동일했다(차이는 상단 필터 개수뿐). 이 수정은 컴포넌트 한 곳이라 두 역할에 자동으로 같게 적용된다. 같은 컴포넌트를 쓰는 **작업 현황 캘린더 뷰**(`task-status-calendar`)도 함께 개선된다.

## 검증

수정 브랜치 dev(`localhost:3031`)에서 확인. **전부 통과.**

| 계정 · 뷰포트 | 각 주 높이 |
| --- | --- |
| step01 · 1280×700 (작은 화면) | 104px ×5 ✅ (이전 59px) |
| master01 · 1280×700 | 104px ×5 ✅ (step01과 동일) |
| step01 · 1440×900 (큰 화면) | 104px ✅ |

화면 캡처로 700px 뷰포트에서 각 날짜 칸이 시원하게 커지고(3주 노출 + 스크롤) 날짜가 또렷이 보이는 것을 확인했다.

- `npm run lint` 통과 (경고 0)
- `npx tsc --noEmit` 통과
- `npm run build` 통과

## 알려진 이슈 / 후속

- **큰 화면에서도 6주 합계(624px)가 캘린더 영역보다 크면 스크롤이 생긴다.** 각 칸을 넉넉히(104px) 지키기 위한 트레이드오프다. "칸을 작게 하지 말라"는 요청을 우선했다. 만약 큰 화면에서 6주를 한눈에(스크롤 없이) 보는 걸 더 원하면 `WEEK_MIN_PX`를 낮추면 된다(예: 88 — 그 경우 작은 화면 칸도 그만큼만 커진다).
- docs 번호는 작성 시점 main 최대(136) + 1로 137을 썼다. 병렬 머지로 조정될 수 있다.
