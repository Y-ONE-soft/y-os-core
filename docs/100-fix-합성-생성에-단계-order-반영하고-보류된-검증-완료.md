# fix: 합성 생성에 단계 order 반영하고 보류된 검증 완료

## 작업 요약

`stage-order`(단계 순서 필드) 작업이 main에 머지되면서, 이 브랜치의 합성 생성(`compose.ts`)도 새 `Stage.order` 규약을 따르도록 맞췄다. 그리고 **그동안 막혀 있던 단계 생성 검증을 실제로 끝냈다.**

docs/88에 "검증 불가"로 남겨 두었던 항목의 후속이다.

## 배경 — 무엇이 막혀 있었나

작업 도중 개발 DB에서 모든 단계 생성이 실패했다.

```
P2011  null value in column "order" of relation "Stage" violates not-null constraint
```

`stage-order` 세션이 아직 머지하지 않은 마이그레이션을 **공유 개발 DB에 선반영**한 상태였고, 그 SQL이 `ALTER COLUMN "order" DROP DEFAULT`를 하기 때문에 `order`를 넣지 않는 모든 삽입이 실패했다. 당시 진단 결과 이 브랜치뿐 아니라 **main의 기존 `createStage`도 동일하게 실패**했다(프로덕션은 마이그레이션이 없어 무관).

`schema.prisma`에 필드가 없으니 이쪽에서 대응할 방법이 없어 대기했고, 해당 작업이 머지된 지금 반영한다.

## 변경 파일 내역

### `src/server/workspace/compose.ts`

**1. 단계 생성에 `order` 추가**

새 프로젝트를 만드는 경로이므로 기존 행과 충돌할 일이 없다. 그대로 `1..N`을 매긴다.

```ts
order: stageIndex + 1,   // @@unique([projectId, order])
```

기존 `createStage`는 이미 있는 프로젝트에 단계를 덧붙이므로 `마지막 order + 1`을 조회해 쓰지만, 여기서는 그 조회가 불필요하다.

**2. 단계 id에서 정렬 트릭 제거**

태스크 1에서는 `st-000-<uuid>` 형태로 id에 순번을 넣었다. 한 트랜잭션 안의 행은 `createdAt`이 같아 정렬 키가 id로 떨어지는 문제 때문이었다. 그런데 `stage-order`가 정렬 규약을 바꿨다.

```ts
const STAGE_ORDER = [{ order: "asc" }, { id: "asc" }];   // 단계
const ORDER       = [{ createdAt: "asc" }, { id: "asc" }]; // 할일·댓글 등
```

**단계는 이제 명시적 `order`로 정렬**하므로 id 트릭이 불필요해졌다. 평범한 `st-<uuid>`로 되돌렸다.

**3. 할일 id의 정렬 트릭은 그대로 둔다**

할일 정렬은 여전히 `[createdAt asc, id asc]`이다. 프리셋의 할일 순서를 지키려면 id 순번이 계속 필요하다. `orderedId`의 주석을 "할일 전용"으로 바꿔 이 구분을 명시했다.

정렬 규약이 단계와 할일로 갈렸다는 점이 이 변경의 핵심이다 — 한쪽만 보고 양쪽을 같이 고치면 할일 순서가 깨진다.

### 검증 도구에서 고친 것 두 가지

**1. 고정 sleep → 상태 대기.** 단계·할일까지 만들면서 생성이 무거워지자 `waitForTimeout(2500)`이 모자라 "다이얼로그 닫힘"과 "사이드바 노출"이 실패했다. **제품 결함이 아니라 테스트 타이밍**이었다. `waitFor({ state: "hidden" })`과 링크 `waitFor()`로 바꾸니 통과했다. 고정 sleep은 대상이 무거워지면 조용히 거짓 실패를 만든다.

**2. 낡은 dev 서버.** 리베이스 후 재검증에서 9건이 실패했는데, 원인은 이전 dev 서버가 포트를 잡고 있어 새 서버가 `EADDRINUSE`로 뜨지 못했고 **리베이스 이전 코드가 응답**한 것이었다. `npm run dev`를 백그라운드로 띄울 때 이전 프로세스가 남는 문제로, 포트 점유를 확인하고 해당 워크트리 소속 프로세스만 정리한 뒤 재시작해 해결했다.

- `npm run lint` 통과 (종료코드 0)
- `npm run build` 통과 (종료코드 0)

## 알려진 이슈 / 후속

- 남은 태스크: 직접 만들기 모드 + 모드 전환 UI.
- **`Stage.order`가 생겼으므로 `even-stages`의 균등 분할 결과 순서도 `order`가 보장한다.** 이후 사용자가 보드에서 단계를 드래그해 순서를 바꾸면 날짜와 `order`가 어긋날 수 있는데, 이는 `stage-order` 작업이 정의할 영역이라 이번 범위에서 다루지 않았다.
- docs 번호는 병렬 머지로 여러 번 밀렸다. 이 브랜치의 네 문서는 push 직전에 한 번에 재조정한다.
