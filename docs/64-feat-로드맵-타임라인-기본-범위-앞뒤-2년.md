## 1. 작업 내용 요약

작업 현황 로드맵과 프로젝트 상세 단계 로드맵의 타임라인이 **데이터가 없어도 오늘 기준 앞뒤 2년까지 스크롤/드래그해 이동할 수 있도록** 범위를 넓혔다. 데이터가 그 범위 밖에 있으면 **그 지점까지 자동으로 확장**된다.

### 변경 전 — 배율마다 범위가 제각각이고 좁았다

`RANGE_CONFIG`가 배율별로 과거·미래 개월 수를 따로 갖고 있었다.

| 배율 | 과거 | 미래 | 총 범위 |
|---|---|---|---|
| 일 | 1개월 | 2개월 | 3개월 |
| 주 | 2개월 | 4개월 | 6개월 |
| 개월 | 6개월 | 12개월 | 1.5년 |
| 분기 | 12개월 | 24개월 | 3년 |

'일' 배율에서는 3개월 밖으로 아예 이동할 수 없었고, 먼 미래에 잡힌 단계는 어떤 배율에서도 화면 밖이면 도달할 수 없었다.

### 변경 후

- **모든 배율에서 오늘 기준 과거 24개월 · 미래 24개월**을 기본 보장 (배율은 확대 배율일 뿐, 이동 가능 범위와 무관해짐)
- 표시 대상 단계의 시작·종료일이 기본 범위를 벗어나면 **그 날짜까지 타임라인을 넓힌다** (종료일 당일이 온전히 보이도록 하루 더 확보)

## 2. 구현

`roadmap-window.ts`

- `RangeConfig`에서 `monthsBefore`·`monthsAfter` 제거 → `dayWidth`·`unit`만 남김 (배율 = 확대율이라는 기존 주석의 의도와 코드가 이제 일치)
- `MIN_MONTHS_BEFORE = 24` / `MIN_MONTHS_AFTER = 24` 상수 도입
- `buildTimeline(range, today, bounds?)` — 세 번째 인자로 데이터 구간을 받아 기본 범위와 합집합을 취한다
- `boundsOfStages(stages)` 헬퍼 추가 — 단계 배열에서 최소·최대 날짜를 뽑는다 (문자열 `YYYY-MM-DD` 비교라 TZ 이슈 없음)

호출부

| 화면 | 넘기는 범위 |
|---|---|
| 프로젝트 상세 단계 로드맵 | 해당 프로젝트의 단계 전체 |
| 작업 현황 로드맵 | 표시 중인 섹션의 모든 프로젝트 단계 |

기존 눈금 경계 스냅(주=월요일, 개월/분기=1일) 로직과 `todayOffset`·`scrollToToday`는 그대로 동작한다.

### 성능 확인

가장 촘촘한 '일' 배율에서 범위가 3개월 → 4년으로 늘어 눈금이 약 90개에서 **1,466개**, 폭이 **81,996px**가 된다. 그래서 성능을 검증 항목에 포함했다.

- 배율 전환 5초 이내 ✓ (실측 여유 있음)
- 스크롤 조작 200ms 이내 ✓

가상 스크롤 없이도 현재 규모에서는 문제없다고 판단해 도입하지 않았다. 다만 눈금 수가 선형으로 늘어나는 구조이므로, 기본 범위를 더 넓히거나 '일' 배율을 더 촘촘하게 만들 경우 재검토가 필요하다.

## 5. 알려진 사항 / 후속 과제

- 범위가 넓어진 만큼 '일' 배율에서 초기 진입 시 오늘 위치로 스크롤하는 동작(`scrollToToday`)의 중요도가 커졌다 — 기존 로직 그대로 동작함을 확인했으나, 사용자가 멀리 스크롤한 뒤 배율을 바꾸면 다시 오늘로 점프한다(기존 동작 유지)
- 과거/미래 2년은 상수(`MIN_MONTHS_BEFORE`/`AFTER`)로 한 곳에서 관리한다. 설정화가 필요하면 이 상수만 바꾸면 된다
- 데이터 확장은 **표시 중인 단계 기준**이다. 작업 현황에서 필터로 프로젝트를 빼면 그 프로젝트의 먼 날짜는 범위에 반영되지 않는다(의도된 동작 — 보이지 않는 데이터까지 스크롤 범위를 늘릴 이유가 없음)
