# fix: 헤더 유저 트리거를 아바타+이름 한 줄로 정리

## 작업 요약

글로벌 헤더 우측 유저 메뉴 트리거가 "부자연스럽고 오른쪽으로 치우쳐 보인다"는 지적을 받아, **아바타 + 이름 + 쉬브론 한 줄** 형태로 재구성했다. 역할(직책)은 드롭다운 패널로 일원화했다.

## 원인 — 측정으로 특정

헤더 자체는 대칭이었다. 좌측 첫 컨트롤 `left = 24px`, 우측 트리거 `right = 1416px`(뷰포트 1440 기준 우 여백 24px)로 `px-6` 패딩이 정확히 맞았다. **문제는 트리거 내부였다.**

실제 앱을 띄워 Playwright로 `getBoundingClientRect()`를 측정한 결과:

| 항목 | 측정값 (변경 전) | 문제 |
| --- | --- | --- |
| 텍스트단 높이 | `13 → 50` (37px) | 아바타(`15.5 → 47.5`, 32px)보다 커서 위아래로 2.5px씩 삐져나옴 |
| 이름/역할 우측단 | 이름 `1416`, 역할 `1412` | 좌측 정렬이라 짧은 역할 텍스트가 헤더 우측 경계에서 **4px 파임** |
| 같은 항목 (master01) | 이름 `1416`, 역할 `1399` | 이름이 길수록 악화 — **17px 파임** |
| 트리거 좌측단 | `스탭` 1390 / `마스터` 1377 | 이름 길이에 따라 블록 전체가 좌우로 이동 |

즉 "오른쪽으로 치우쳐 보인다"의 실체는 헤더 정렬 오류가 아니라, **우측 경계에서 2줄 텍스트의 오른쪽 끝이 들쭉날쭉하게 파이면서 생기는 시각적 쏠림**이었다. 여기에 텍스트단이 아바타보다 높아 세로로도 어긋나 있었다.

부가로, 트리거의 이름·역할은 드롭다운 `_Me` 블록(이름 · 역할 · 이메일)과 **정보가 중복**이었다.

## 선택한 방향

사용자와 세 안을 비교해 **아바타 + 이름 한 줄 + 쉬브론**을 채택했다.

| 안 | 내용 | 판단 |
| --- | --- | --- |
| A (채택) | 아바타 + 이름 한 줄 + 쉬브론, 역할은 드롭다운으로 | 로그인 주체가 헤더에 남으면서 높이·우측단이 정리됨 |
| B | 아바타만 남기기 | 가장 깔끔하나 헤더에서 사용자 확인이 불가 |
| C | 2줄 유지 + 정렬만 교정 | 변경은 작으나 2줄 구조의 불안정성이 남음 |

## 변경 파일 내역

### `src/components/layout/user-menu.tsx`

**1. 트리거 — 2줄 → 한 줄**

```tsx
<DropdownMenuTrigger className="flex h-8 items-center gap-2.5 rounded-full outline-none focus-visible:ring-2 focus-visible:ring-ring">
  <span className="flex size-8 shrink-0 items-center justify-center rounded-full bg-primary ...">
    {user.name.charAt(0)}
  </span>
  <span className="max-w-[140px] truncate text-[13px] font-medium text-foreground">
    {user.name}
  </span>
  <ChevronDown aria-hidden className="size-3.5 shrink-0 text-muted-foreground" />
</DropdownMenuTrigger>
```

- `h-8`로 트리거 높이를 아바타와 동일한 32px에 고정 → 세로 어긋남 제거
- 역할 줄 제거 → 우측단 파임의 원인 자체를 제거
- `max-w-[140px] truncate` → 이름이 길어도 트리거 폭이 고정되어 헤더 우측 덩어리가 흔들리지 않음
- `ChevronDown` 추가 → 클릭 가능한 메뉴임을 명시 (기존에는 어포던스가 없었다)
- 아바타·쉬브론에 `shrink-0` → 이름이 길 때 눌리지 않도록

**2. 로딩 스켈레톤 — 트리거와 같은 형태로**

```tsx
<div className="flex h-8 items-center gap-2.5">
  <Skeleton className="size-8 shrink-0 rounded-full" />
  <Skeleton className="h-3.5 w-16" />
</div>
```

2줄 스켈레톤을 그대로 두면 로딩 → 로드 전환 시 헤더 우측이 한 번 튄다. 트리거와 동일한 한 줄·32px로 맞췄다.

**3. 드롭다운 `sideOffset` 22 → 24 (중요)**

트리거 높이를 37px → 32px로 바꾸면서 **패널 위치가 함께 어긋났다.** 기존 `sideOffset={22}`는 2줄 트리거의 하단(50px)을 전제로 "헤더 아래 8px(top 72px)"을 맞춘 값이었다.

- 변경 직후 측정: 패널 `top = 69.7px` (기대 72px, **2.3px 위로 파고듦**)
- 트리거 하단이 48px(`(64-32)/2 + 32`)로 바뀌었으므로 `72 - 48 = 24`
- `sideOffset={24}`로 수정 후 재측정: 패널 `top = 71.7px` ✅

주석에 "트리거 높이를 바꾸면 이 값도 함께 조정해야 한다"를 명시해 다음 사람이 같은 함정을 밟지 않도록 했다. **이 항목은 화면을 직접 띄워 측정하지 않았으면 놓쳤을 회귀다.**

## 알려진 이슈 / 후속

- **`max-w-[140px]`는 경험값**이다. 실제 사용자 이름이 길어지면(예: 4자 이상 + 영문 혼용) truncate가 걸린다. 현재 시드 계정(`스탭`, `마스터`)에서는 여유가 충분하다.
- 검증 중 `[workspace] 불러오기 실패 TypeError: Failed to fetch`가 간헐적으로 관측됐으나, **이번 변경과 무관**하다. 테스트 스크립트가 워크스페이스 fetch가 진행 중일 때 `page.goto`로 페이지를 이동시켜 요청이 중단된 것으로, 동일 스크립트의 다른 실행에서는 발생하지 않았다.
- 알림 배지(`NOTIFICATION_COUNT = 2`)는 여전히 드롭다운 안에만 있다. 헤더 트리거에 미읽음 표시(아바타 위 dot 등)를 노출할지는 알림 도메인 구현 시 함께 검토할 사항이다.
- **docs 번호는 68 → 69로 조정**했다. 작성 시점 main 최대 번호가 67이라 68을 썼으나, 커밋 직후 `상세-헤더-칩을-고정-라벨로-변경`이 먼저 머지되며 68을 가져갔다. "나중에 머지되는 쪽이 번호를 올린다"는 규칙(CLAUDE.md)에 따라 69로 올리고 최신 main 위로 리베이스했다.
- 참고로 현재 main에는 `66-feat-작업-세부-사항에-완료날짜-추가.md`와 `66-refactor-상세-오버레이-헤더-브레드크럼-통일.md`로 **66번이 중복**되어 있다. 이번 작업 범위 밖이라 손대지 않았으나 정리가 필요하다.
