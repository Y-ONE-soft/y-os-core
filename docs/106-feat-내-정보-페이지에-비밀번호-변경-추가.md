# feat: 내 정보 페이지에 비밀번호 변경 추가

요청 사이클 `비밀번호-변경`의 2번(마지막) 태스크. 1번 태스크에서 만든 API를 화면에 연결한다. 이 커밋으로 요청이 완결된다.

## 변경 내용

### 1. `PasswordSection` 컴포넌트 (같은 파일 내 신규)

**프로필 폼과 완전히 분리한 별개의 `<form>`이다.**

```tsx
/**
 * 비밀번호 변경 — 프로필 폼과 저장 버튼을 나눈다.
 * 이름만 고치려는데 현재 비밀번호까지 요구하면 안 된다.
 */
```

하나의 폼으로 합치면 이름만 바꾸려는 사용자도 현재 비밀번호를 입력해야 한다. 저장 버튼이 둘인 것이 오히려 의도를 분명히 한다.

**확인란 일치는 화면에서 막는다.** 서버는 "확인란"이라는 개념을 모른다(요청 본문에 `currentPassword`·`nextPassword`만 보낸다). 불일치를 서버까지 보낼 이유가 없으므로 제출 전에 거른다.

```tsx
// 확인 일치는 서버가 알 수 없는 값이라 화면에서 막는다
if (next !== confirm) {
  setStatus({ kind: "error", text: "새 비밀번호가 서로 다릅니다." });
  return;
}
```

**즉시 불일치 표시는 확인란에만 건다.**

```tsx
// 확인란만 즉시 불일치를 표시한다 — 새 비밀번호 쪽에 띄우면
// 입력하는 도중 내내 빨갛다
aria-invalid={(confirm.length > 0 && confirm !== next) || undefined}
```

`confirm.length > 0` 조건이 있어 비어 있을 때는 경고가 뜨지 않는다.

**성공하면 입력 3개를 비운다.** 비밀번호가 화면에 남아 있을 이유가 없고, 비워야 "처리됐다"는 신호도 된다.

`autoComplete`을 `current-password`/`new-password`로 지정해 브라우저 비밀번호 관리자가 올바로 동작하게 했다.

### 2. 배치 — 아래 행에 전폭 패널

```tsx
{/* 좌: 계정 요약(고정폭) · 우: 편집 폼 …
    비밀번호는 아래 행에 따로 둔다 (저장 버튼이 분리돼야 하므로 폼도 별개). */}
```

계정·프로필 2단 아래에 전폭으로 놓았다. 3개 입력을 `sm:grid-cols-3`으로 한 줄에 배치한다. **직전 사이클(docs/97)에서 지적된 아래쪽 여백이 이 패널로 채워지는 효과도 있다** — 여백을 채우려고 넣은 것은 아니고, 순서상 프로필 다음에 오는 것이 자연스러워서다.

패널·라벨·입력·푸터 밴드는 모두 CLAUDE.md의 디자인 규약(docs/91)을 따랐다.

### 3. 부작용 안내

비밀번호를 바꾸면 다른 기기의 세션이 끊긴다(docs/99). 사용자가 모르면 "왜 폰에서 로그아웃됐지"가 된다. **두 군데에 적었다.**

- 섹션 설명: `변경하면 이 기기를 제외한 다른 기기에서 로그아웃됩니다.` — 누르기 전에 보인다
- 성공 메시지: `비밀번호를 변경했습니다. 다른 기기는 로그아웃됩니다.` — 결과를 확인시킨다

### 4. 페이지 부제 수정

`이름·직책·이메일·연락처를 수정할 수 있습니다` → `프로필과 비밀번호를 수정할 수 있습니다`. 항목을 나열하던 문구가 비밀번호 추가로 부정확해졌다.

## 검증

`npm run build` 성공, `npm run lint` 경고 0.

**헤드리스 Chrome + CDP로 폼을 실제로 채우고 눌렀다.**

```
① 확인 불일치 (1111 / 2222 / 3333)
   상태: 새 비밀번호가 서로 다릅니다.        ← 서버로 안 보내고 화면에서 차단
② 정상 변경 (1111 → 2222)
   상태: 비밀번호를 변경했습니다. 다른 기기는 로그아웃됩니다.
   입력 비워졌나: true
③ 원복 (2222 → 1111)
   상태: 비밀번호를 변경했습니다. 다른 기기는 로그아웃됩니다.
④ 최종 로그인 확인
   1111: 200
```

스크린샷으로 섹션 렌더와 성공 메시지 표시도 확인했다.

### 원상복구

검증에서 비밀번호를 바꿨으므로 **`1111`로 되돌렸고 로그인 200을 확인했다.** 되돌리지 않으면 사용자가 로그인하지 못한다.

## 알려진 이슈 / 주의점

### 최소 4자 규칙이 화면에 안내되지 않는다

입력란에 "4자 이상" 힌트가 없다. 짧게 입력하면 제출 후 서버 메시지로 알게 된다. 규칙이 4자로 느슨한 동안은 부딪힐 일이 드물지만, **운영 전 길이를 올릴 때는 화면 안내도 함께 넣어야 한다.**

### 비밀번호 강도 표시가 없다

`1111` 같은 값이 그대로 통과한다. docs/99의 한계와 같은 사안이다.

### 변경 후 현재 기기의 세션은 유지된다

의도된 동작이다(바꾸자마자 로그아웃되면 불편). 다만 "모든 기기에서 로그아웃" 옵션을 원하는 경우는 지원하지 않는다.

### 다른 기기는 조용히 끊긴다

끊긴 기기는 다음 요청에서 401을 받아 로그인 화면으로 이동할 뿐, 이유를 알려주지 않는다. 세션 만료와 구분되지 않는다.

### 실패 메시지가 입력란 옆이 아니라 푸터에 뜬다

어느 칸이 문제인지 직접 가리키지 않는다. 입력이 3개뿐이고 메시지가 구체적이라(`현재 비밀번호가 올바르지 않습니다`) 혼선은 적다고 판단했다.
