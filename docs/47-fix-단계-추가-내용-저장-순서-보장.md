## 1. 배경 — docs/45(PR #39) 프로덕션 검증에서 발견한 자체 결함

단계 추가 오버레이(docs/45)를 머지하고 **프로덕션에서 검증하다 발견**했다. 로컬 검증 20건은 모두 통과했지만, 프로덕션에서는 **입력한 `내용`(description)이 저장되지 않았다**(기간·단계명은 정상).

```
FAIL — 상세 오버레이에 내용 저장 확인 :: {"desc":"","start":"2026-08-03","end":"2026-08-07"}
```

## 2. 원인 — 두 요청의 순서 미보장 + 조용한 실패

오버레이는 생성 후 상세 필드를 이어 저장하는 구조였다.

```
addStage()  → POST /api/admin/stages        (생성)
updateStage() → PATCH /api/admin/stages/{id} (내용·공동작업자)
```

두 호출은 각각 `cache.persist(promise)`로 넘어가는데, **`persist()`는 에러 처리만 할 뿐 순서를 보장하지 않는다**(`workspace-cache.ts`). 즉 두 요청이 사실상 동시에 출발한다.

- 로컬(dev, 왕복 수 ms): POST가 먼저 도착 → 정상 저장 → **검증 통과**
- 프로덕션(서버리스, 왕복 수백 ms + 콜드 스타트): **PATCH가 먼저 도착** → 대상 행이 아직 없음

여기에 두 번째 문제가 겹친다. 서버의 단계 수정은 **0건 수정도 성공(200)** 으로 응답한다. 그래서 클라이언트는 실패를 감지하지 못하고 `내용 PATCH 200`이 찍힌 채 데이터만 조용히 사라졌다 — 검증 스크립트조차 "PATCH 200"을 보고 통과 처리했다.

## 3. 수정

`addStage`에 **선택 인자 `extra`**(내용·공동작업자)를 추가하고, **생성 완료 후에 patch가 나가도록 스토어 내부에서 체이닝**했다.

```ts
const created = createStageApi({ ... });
cache.persist(
  description || requestedCollaborators
    ? created.then(() => patchStageApi(id, { description, requestedCollaborators }))
    : created,
);
```

- 순서가 코드로 보장되어 호출부(오버레이)는 한 번만 호출하면 된다
- 낙관적 로컬 상태에도 `description`·`requestedCollaborators`를 함께 반영해 화면과 서버가 어긋나지 않게 했다
- `persist()`가 체인 전체를 감싸므로 **어느 단계에서 실패해도 서버 상태로 재동기화**된다 (기존 실패 복구 동작 유지)

### 대안 검토

| 방안 | 판단 |
|---|---|
| 생성 API가 description·collaborators를 받도록 확장 | 요청 1회로 줄어 가장 깔끔하나 `service.ts`·route·api 클라이언트를 동시에 건드려야 한다. 해당 파일들을 다른 세션이 활발히 수정 중이라 이번엔 제외 (docs/45 후속 과제로 유지) |
| 서버 update를 0건일 때 404로 응답 | 조용한 실패를 드러내는 근본 개선이지만 서버 파일 수정이라 위와 같은 이유로 별도 태스크 대상 |
| **클라이언트 체이닝 (채택)** | 서버 무변경 + 순서 확정. 변경 범위 2파일 |

## 5. 검증

1. `npm run lint` ✓ · `npm run build` ✓ (Next.js 16.2.11) — 최신 main(`e731c00`) 반영 후 재확인
2. **요청 순서 자체를 검증하는 스크립트 추가** — 이 버그는 지연이 작은 로컬에서 재현되지 않으므로 "결과"만 봐서는 회귀를 잡을 수 없다. puppeteer로 요청/응답 타임스탬프를 수집해 **`PATCH 요청 시작 시각 ≥ POST 응답 시각`** 을 단언한다.
   - 생성 POST 200 ✓ / 내용 PATCH 200 ✓ / **PATCH는 POST 응답 이후에 시작** ✓ / **새로고침 후 내용 저장 확인** ✓ — ALL PASS
3. 검증용 데이터는 임시 프로젝트째 생성·삭제하고 잔여 0건 확인

### 검증 환경 메모

main의 시드 정책 변경(docs/46 — 그룹 골격만 남기고 프로젝트 시드 제거)으로 공유 DB에 `p-yos` 같은 고정 프로젝트가 더 이상 없다. 그래서 이번 검증은 **임시 프로젝트를 직접 만들어 쓰고 지우는 방식**으로 전환했다. 다른 세션의 테스트 데이터를 건드리지 않는 이점도 있으므로, 이후 보드 검증도 이 방식을 권장한다.

## 6. 알려진 사항 / 후속 과제

- **서버가 0건 수정에도 200을 반환**하는 문제는 그대로다. 이번 순서 보장으로 이 경로에서는 재현되지 않지만, 존재하지 않는 id로 patch하면 여전히 조용히 성공한다 — 서버 파일 동시 수정이 잦아 별도 태스크로 남긴다
- 내용·공동작업자를 입력하면 요청은 여전히 2회(POST → PATCH)다. 생성 API 확장 시 1회로 축소 가능(docs/45 후속 과제와 동일)

## 7. 교훈

**"응답 200"이 "저장됨"을 뜻하지 않는다.** docs/43(보드 단계 추가)에서 *낙관적 UI 갱신 ≠ 저장*을 배워 POST 200 확인을 추가했는데, 이번엔 그 **200조차 신뢰할 수 없는 경우**를 만났다. 저장 검증의 최종 근거는 **새로고침 후 실제 값 확인**이며, 순서 의존이 있는 흐름은 **요청 순서 자체를 단언**해야 한다. 또한 지연이 작은 로컬에서만 검증하면 경쟁 조건은 통과해 버리므로, **머지 후 프로덕션 검증이 형식적 절차가 아니라 실제로 결함을 잡는 관문**임이 이번에 확인됐다.
