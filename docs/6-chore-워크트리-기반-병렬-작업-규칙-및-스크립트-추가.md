# 6. 워크트리 기반 병렬 작업 규칙 및 스크립트 추가

- **예정 커밋 메시지**: `chore: 워크트리 기반 병렬 작업 규칙 및 스크립트 추가`
- **작업일**: 2026-07-22
- **작업 브랜치**: `병렬-작업-환경-구성` (이 태스크부터 신규 전략 적용 — main 직접 커밋 아님)

---

## 1. 작업 내용 요약

여러 세션(에이전트)이 같은 저장소에서 **동시에** 작업할 수 있도록 git worktree 기반 병렬 작업 체계를 도입했다.

1. **`CLAUDE.md` 작업 워크플로우를 2단 사이클로 통합 개정** — **요청 사이클**(러프 요청 1건 = 브랜치 1개 = PR 1개: 분해 → 브랜치+워크트리 → 태스크 반복 → PR 생성(작업 내용 기록) → merge commit 머지 → 로컬 최신화·정리)과 **태스크 사이클**(태스크 1개 = 커밋 1개 = 문서 1개: 작업 → 문서 → 검토 → 커밋). 브랜치명 = 한글 작업명, main 직접 커밋 금지
2. **워크트리 생성 스크립트 `scripts/new-work.ps1` 추가** — 브랜치 + 워크트리 생성 + `npm install`까지 한 번에
3. **`.gitignore`에 `/.claude/worktrees/` 추가** — 워크트리가 저장소 내부 경로에 생성되므로 제외 필요
4. **`next.config.ts`에 `turbopack.root` 고정** — 워크트리에서 빌드 시 워크스페이스 루트 오추론 문제 해결 (아래 5절)
5. 이 태스크 자체를 신규 전략 그대로 워크트리에서 수행하여 체계를 엔드투엔드 검증

## 2. 배경 (왜)

- 사용자 요구: "세션을 하나의 작업자로 가정하고, 브랜치를 새로 만들면 자동으로 워크트리를 반영해 병렬로 작업. 브랜치 전략은 PR 단위 한글 작업명."
- 같은 폴더에서 여러 작업자가 작업하면 미커밋 변경이 뒤섞인다. 실제로 이 작업 시작 시점에 **메인 폴더에는 docs/5(프로젝트 섹션 사이드바) 작업이 미커밋 검토 대기 상태**로 존재했고, 워크트리 격리 덕분에 해당 파일을 전혀 건드리지 않고 이 작업을 진행할 수 있었다 — 도입 필요성이 그대로 입증된 셈.
- worktree는 `.git`(히스토리·브랜치·리모트)을 공유하면서 작업 폴더만 분리하므로, "저장소 하나 + 작업자 N명" 구조에 정확히 부합한다.

## 3. 확정된 전략

| 항목 | 결정 |
|---|---|
| 작업 단위 | 세션(에이전트) 1개 = 작업자 1개 |
| 브랜치 | **PR 1개 = 브랜치 1개, 브랜치명 = 한글 작업명** (공백은 하이픈). 예: `병렬-작업-환경-구성` |
| main | 직접 작업·커밋 금지. 모든 변경은 브랜치 → PR → 머지로만 유입 |
| 워크트리 위치 | `.claude/worktrees/<이름>` (EnterWorktree 기본 경로와 통일, gitignore 한 줄로 처리) |
| 워크트리 생성 | Claude 세션: `EnterWorktree` → `git branch -m <한글-작업명>` / 수동: `.\scripts\new-work.ps1 <한글-작업명>` |
| 완료 흐름 | 마지막 커밋 승인 = 일괄 진행 승인 → push → PR 생성(**본문에 작업 내용 기록**) → 머지 → 메인 폴더 `git pull` → 원격 브랜치·워크트리 제거 |
| 머지 방식 | **merge commit** (`gh pr merge --merge`) — 태스크=커밋=문서 매핑을 main 히스토리에 보존, squash 금지 |
| docs 번호 충돌 | 병렬 작업으로 같은 번호 발생 시 **나중에 main에 머지되는 쪽이 번호를 올려 조정** |

### 사용법

```powershell
# 수동 (터미널)
.\scripts\new-work.ps1 로그인-화면          # 브랜치+워크트리 생성 + npm install
cd .claude\worktrees\로그인-화면
npm run dev -- -p 3001                      # 메인 dev 서버(3000)와 포트 분리

# 다 쓰면 (머지 후)
git worktree remove .claude/worktrees/로그인-화면
```

Claude 세션은 태스크 시작 시 EnterWorktree로 진입하면 되고, 스크립트로 만들어 둔 워크트리는 EnterWorktree의 `path` 파라미터로 이어받을 수 있다.

## 4. 변경 파일 내역

| 구분 | 파일 | 내용 |
|---|---|---|
| 수정 | `CLAUDE.md` | "작업 워크플로우 (필수)"를 요청 사이클(브랜치·PR)/태스크 사이클(커밋·문서) 2단 구조로 통합 개정 — 기존 태스크 워크플로우와 신규 병렬 규칙을 하나의 흐름으로 병합 |
| 수정 | `.gitignore` | `/.claude/worktrees/` 추가 (저장소 내부 워크트리 제외) |
| 수정 | `next.config.ts` | `turbopack.root: __dirname` — 워크트리 빌드 시 루트 오추론 방지 |
| 신규 | `scripts/new-work.ps1` | 브랜치+워크트리+`npm install` 원스텝 스크립트 |
| 신규 | `docs/6-chore-워크트리-기반-병렬-작업-규칙-및-스크립트-추가.md` | 이 문서 |

### 커밋에 포함되지 않는 변경 (히스토리 기록)

- `git config core.quotepath false` — 한글 파일명이 `git status`에서 8진수 이스케이프(`\355\224...`)로 깨져 보이는 것 수정. 저장소 로컬 설정이라 커밋 대상 아님 (`.git/config`는 모든 워크트리가 공유하므로 1회면 충분)
- 세션 메모리 저장 — 사용자 요청("작업 내용을 기억")에 따라 기존 태스크 ①~⑤ 요약과 병렬 작업 규칙을 에이전트 영구 메모리(저장소 외부)에 기록
- 이 워크트리 자체(`.claude/worktrees/parallel-work-setup`) — gitignore 처리되어 커밋에 나타나지 않음

## 5. 주요 결정 이유

- **워크트리 위치를 저장소 내부(`.claude/worktrees/`)로 통일** — Claude의 EnterWorktree 기본 경로와 같아 도구/수동 생성물이 한곳에 모이고, gitignore 한 줄로 관리된다. 저장소 밖 형제 폴더 방식보다 탐색·정리가 쉽다.
- **워크트리 폴더명은 ASCII, 브랜치명은 한글** — EnterWorktree가 워크트리 이름에 ASCII만 허용한다(한글 거부 확인). 브랜치명은 git이 한글을 완전 지원하므로 생성 직후 `git branch -m`으로 한글 작업명을 적용한다. 수동 스크립트는 처음부터 한글 이름으로 만들 수 있다.
- **스크립트를 ASCII 전용으로 작성** — Windows PowerShell 5.1은 BOM 없는 UTF-8 스크립트를 ANSI로 오독해 한글 리터럴이 깨진다. 파라미터로 들어오는 한글(브랜치명)은 런타임 전달이라 무관하다.
- **`turbopack.root: __dirname`** — 워크트리에서 빌드하면 락파일이 2개(메인 + 워크트리) 감지되어 Next.js가 **메인 폴더를 워크스페이스 루트로 오추론**하는 경고가 발생했다. 내장 문서(`node_modules/next/dist/docs/.../turbopack.md`)의 해법대로 각 체크아웃이 자기 자신을 루트로 갖게 절대 경로로 고정 — 메인 폴더에서도 동작이 동일하므로 부작용 없다.

## 6. 검증

이 태스크 자체가 신규 전략의 첫 적용 사례다.

1. **워크트리 생성·격리** — `origin/main`(2dbb5bb) 기준 워크트리 생성, 브랜치 `병렬-작업-환경-구성` rename 확인. 메인 폴더의 미커밋 docs/5 작업과 완전 격리된 상태로 작업 완료 (`git worktree list`로 2개 체크아웃 공존 확인)
2. **`npm install`** ✓ — 신규 워크트리에서 657개 패키지, 38초
3. **`npm run lint`** ✓
4. **`npm run build`** ✓ — 최초 빌드에서 워크스페이스 루트 오추론 경고 발견 → `turbopack.root` 수정 후 재빌드에서 **경고 소멸** 확인, `/`·`/_not-found` Static 프리렌더 정상

## 7. 알려진 사항 / 후속 과제

- **Vercel 프리뷰와 한글 브랜치** — 브랜치 push 시 프리뷰 배포가 자동 생성되는데, 한글 브랜치명이 프리뷰 URL 슬러그로 어떻게 치환되는지는 첫 push 후에 확정된다. 이 브랜치를 push할 때 "사후 검증 결과 (추록)"으로 기록 예정
- `npm audit`이 이 시점 6건(moderate 4, high 2)을 보고 — 기존 방침대로 **`npm audit fix --force` 금지** 유지. Next.js 패치 릴리스 추적으로 해소 예정
- 메인 폴더의 dev 서버가 3000 포트를 상시 점유 중 — 워크트리 dev 서버는 3001부터 사용
