# feat: 행 호버 점메뉴 추가 및 할일 삭제 연결

## 작업 요약

행에 마우스를 올리면 **⋮ 버튼**이 나타나고, 누르면 **삭제 메뉴**가 열리도록 했다. 노션의 행 호버 메뉴와 같은 방식이다.

이번 커밋은 공용 컴포넌트 `RowActions`를 만들고 **할일 행 3곳**에 적용한다. 프로젝트·단계는 다음 태스크에서 같은 컴포넌트로 붙인다.

## 왜 필요한가

지금까지 삭제는 **우클릭 컨텍스트 메뉴로만** 가능했다.

| 화면 | 삭제 방법 |
| --- | --- |
| 프로젝트 상세 백로그 | 우클릭 |
| 내 할일 백로그 | 우클릭 |
| 보드 카드 | 우클릭 |
| 사이드바 프로젝트 | 우클릭 |
| 보드 단계 컬럼 | 헤더 우클릭 |

우클릭은 **화면에 아무 단서가 없다.** 처음 쓰는 사람은 삭제할 방법이 있다는 것 자체를 알 수 없다.

## 우클릭 메뉴는 그대로 둔다

⋮ 메뉴를 넣으면서 기존 컨텍스트 메뉴를 **제거하지 않았다.** 익숙해진 사용자의 조작을 빼앗을 이유가 없고, 노션도 둘 다 지원한다. 같은 동작을 두 진입점이 공유하도록 `onSelect` 콜백만 같은 함수를 가리키게 했다.

## 아이콘 — 점 3개(⋮), 점 6개(⠿) 아님

노션의 실제 손잡이는 점 6개(⠿)지만 **점 3개를 택했다.**

점 6개는 관습적으로 **"끌 수 있다"** 는 신호다. 이 저장소에서 할일 행은 실제로 드래그되지만(백로그 → 단계, 캘린더 날짜 칸) **프로젝트 행은 드래그가 안 된다.** 같은 손잡이 모양을 전부에 붙이면 프로젝트 행에서 거짓 신호가 된다. 점 3개는 "메뉴를 여는 버튼"이라는 뜻이라 드래그 가능 여부와 무관하게 6종에 같은 모양을 쓸 수 있다.

## 어디에 두었나 — `components/ui/`

`RowActions`는 **프로젝트(layout)·단계·할일(projects, my-work)** 이 모두 쓴다. 어느 feature 도메인도 소유하지 않으므로 `components/ui/row-actions.tsx`에 두었다.

CLAUDE.md의 `ui`는 "shadcn/ui 공통 컴포넌트"로 적혀 있어 엄밀히는 shadcn 산출물 자리다. 다만 이 컴포넌트는 **도메인 로직이 없고 `dropdown-menu` 프리미티브를 얇게 조합하기만** 하며, 대안(`features/projects/`에 두고 my-work·layout에서 가져다 쓰기)은 docs/78이 지적한 "호출 위치가 도메인을 정하지 않는다"를 거꾸로 어기는 셈이 된다. 판단이 갈릴 수 있는 지점이라 여기 남긴다.

## 컴포넌트 설계

```tsx
<RowActions
  label={task.name}
  actions={[{ label: "할일 삭제", destructive: true, onSelect: () => ... }]}
/>
```

- `label`은 `aria-label`("○○ 메뉴 열기")에 쓰인다. 행이 여러 개일 때 스크린리더가 어느 행의 메뉴인지 구분할 수 있어야 한다.
- `actions`가 비면 아무것도 렌더하지 않는다.
- `destructive`는 `DropdownMenuItem`의 기존 variant로 넘긴다 — 색을 직접 지정하지 않는다.

### 나타나고 사라지는 규칙

```
opacity-0
group-hover:opacity-100          부모 행에 hover
focus-visible:opacity-100        키보드 포커스
data-[state=open]:opacity-100    메뉴가 열려 있는 동안
```

세 번째가 중요하다. 메뉴를 연 뒤 커서가 행을 벗어나면(메뉴 항목은 팝오버라 행 밖에 그려진다) `group-hover`가 풀려 **버튼이 사라지고 메뉴만 떠 있는** 어색한 상태가 된다. 열린 동안은 붙잡아 둔다.

`display:none`이 아니라 `opacity`를 쓴 것은 **호버 시 레이아웃이 밀리지 않게** 하기 위해서다. 버튼 자리는 항상 차지한다.

부수 효과로 **스크린리더에는 항상 노출된다** — `opacity-0`은 접근성 트리에서 숨기지 않는다. 호버가 불가능한 사용자도 이 동작에 도달할 수 있으니 의도한 대로다.

### 행의 클릭·드래그와 충돌하지 않게

할일 행은 **클릭하면 상세가 열리고, 끌면 드래그가 시작된다.** ⋮ 버튼이 그 안에 있으므로 셋을 갈라야 한다.

```tsx
draggable={false}                                   // 버튼 자체는 끌리지 않는다
onClick={(e) => e.stopPropagation()}                // 행의 상세 열기 방지
onPointerDown={(e) => e.stopPropagation()}          // 행으로 전파 차단
```

`stopPropagation`만 쓰고 `preventDefault`는 쓰지 않았다. Radix의 `asChild`는 자식 핸들러를 먼저 실행하고 **`preventDefault`가 호출됐으면 자기 핸들러를 건너뛴다.** `preventDefault`를 썼다면 메뉴가 아예 열리지 않았을 것이다.

### 미검증 — 호버·클릭 동작

**핵심 동작을 눈으로 확인하지 못했다.** 저장소에 브라우저 자동화 도구가 없고, 백로그·보드 행은 보드 스토어 데이터가 클라이언트에 들어온 뒤 그려져 **SSR HTML에 아예 없다**(응답에서 "메뉴 열기" 0건). 확인이 필요한 것은 다음 넷이다.

1. 행에 올렸을 때만 ⋮ 가 보이는가
2. ⋮ 를 눌러도 할일 상세가 같이 열리지 않는가 (`stopPropagation`)
3. 메뉴가 열린 채 커서를 빼도 버튼이 남는가 (`data-[state=open]`)
4. 행 드래그가 여전히 되는가

포트는 다른 세션과 겹치지 않게 3031을 쓰고 **기동 로그의 `Ready`를 확인한 뒤** 요청했다 — 앞선 작업에서 포트 충돌로 다른 세션 서버 응답을 내 것으로 오인한 적이 있다(docs/92).
