# feat: 세 상세 화면에 작업자 칸 통일 배치

요청 사이클 `작업자-표시`의 2번(마지막) 태스크. 1번 태스크에서 응답에 실은 값을 화면에 그린다. 이 커밋으로 요청이 완결된다.

## 요구사항

대화에서 세 번에 걸쳐 확정됐다.

1. 프로젝트 상세·단계 상세·할 일 상세에 **작업자 칸**을 만들어 **아바타와 이름**이 함께 보이게, 세 곳 **동일 위치**로 통일
2. **셀렉트가 아니라 선택된 작업자가 보이도록** — 할 일 상세에 담당자 셀렉트가 이미 있었지만 "고르는 UI"였지 "지금 누가 작업자인지" 표시가 아니었다
3. **공동 작업자 요청이 수락되면 수락한 사람도 함께** 표시

## "동일 위치"를 어떻게 잡았나

세 화면의 구조가 달라 **문자 그대로 같은 좌표는 불가능**했다.

| 화면 | 구조 |
|---|---|
| 단계 상세 · 할 일 상세 | 오버레이 + 우측 `세부 사항` aside |
| 프로젝트 상세 | 전체 페이지 (헤더 + 탭 + 보드) — **aside 없음** |

그래서 이렇게 잡았다.

- **두 오버레이**: `세부 사항` aside의 **첫 항목** — 서로 완전히 동일한 자리
- **프로젝트 상세**: 헤더 요약줄(진행률 옆) — aside가 없으므로 그 화면에서 "요약 속성"에 해당하는 자리

코드에도 근거를 남겼다.

```tsx
{/* 이 화면에는 세부 사항 aside가 없어, 오버레이들의 "세부 사항 첫 항목"에
    대응하는 자리로 헤더 요약줄을 쓴다. 진행률과 같은 줄에 둔다. */}
```

## 변경 내용

### 1. `src/components/features/projects/assignee-list.tsx` (신규)

세 화면이 같은 모양을 쓰도록 공용 컴포넌트로 뺐다.

```tsx
/**
 * 작업자 표시 — 주 작업자와 수락된 공동 작업자를 아바타+이름으로 나열한다.
 * …
 * 고르는 UI가 아니라 "지금 누가 하고 있는지"를 보여주는 표시 전용이다.
 */
```

- **주 작업자를 맨 앞에**, 이어서 수락된 공동 작업자
- **주 작업자가 공동 작업자 목록에도 있으면 한 번만** 보여준다. 자기 자신에게는 요청할 수 없지만(라우트가 400), 담당자가 나중에 바뀌면 겹칠 수 있다
- 2번째부터 `공동` 표시를 붙여 주 작업자와 구분한다
- 아무도 없으면 `emptyLabel`(기본 "미배정")
- 아바타 색은 기존 `avatarColor(id)` 재사용 — 다른 화면과 같은 색이 나온다

### 2. `src/components/features/projects/stage-detail-overlay.tsx`

`세부 사항` 첫 항목으로 작업자 추가.

**작업자는 프로젝트에서 가져온다.** `Stage`에 담당 필드가 없다는 결정에 따른 것인데, 구현 방식이 하나 갈렸다.

```tsx
// 작업자는 프로젝트에서 가져온다. props로 받으면 호출부 4곳을 모두 고쳐야 한다.
const { groups } = useProjectStore();
const projectOwner = groups
  .flatMap((group) => group.projects)
  .find((project) => project.id === projectId)?.owner;
```

이 오버레이는 `projectName`·`projectColor`를 props로 받고 있어 `projectOwner`도 props로 넘기는 게 자연스러워 보였지만, **호출부가 4곳**(`project-detail-page`·`task-status-page`·`my-work-calendar-panel`·`my-work-timeline-panel`)이라 전부 고쳐야 한다. 이미 `projectId`를 받고 있으므로 스토어에서 직접 찾는 편이 변경 범위가 작다.

### 3. `src/components/features/projects/task-detail-overlay.tsx`

`세부 사항` 첫 항목으로 작업자 표시를 추가하고, **기존 배정 셀렉트는 남겼다.**

```tsx
{/* 작업자는 세부 사항의 첫 항목 — 단계 상세와 같은 자리.
    아래 배정 셀렉트는 "고르는 UI", 여기는 "지금 누가 하는지" 표시. */}
```

셀렉트를 없애면 담당자를 바꿀 수단이 사라진다. 둘의 역할이 다르므로 라벨로 구분했다 — 표시는 **`작업자`**, 셀렉트는 **`작업자 배정`**.

기존 셀렉트 라벨이 "담당자"였는데 **"작업자 배정"으로 바꿔** 용어를 통일했다.

### 4. `src/components/features/projects/project-detail-page.tsx`

헤더의 진행률 줄에 작업자를 함께 배치했다. 프로젝트에는 공동 작업자 개념이 없어(`Request` 대상이 할일·단계뿐) `assignee`만 넘긴다. 미지정일 때 문구는 "미배정"이 아니라 **"미지정"** — 프로젝트는 배정 대상이 아니라 소유자 개념이다.

## 검증

`npm run build` 성공(타입 검사 포함), `npm run lint` 경고 0.

헤드리스 Chrome + CDP로 **세 화면을 실제로 열어** 확인했다. 1번 태스크에서 만든 픽스처(마스터 소유 프로젝트 + 단계 + 할일, 스탭이 지정 요청 2건 수락)를 그대로 썼다.

```
① 프로젝트 상세 헤더 작업자: 노노윤기            ← 아바타 "노" + 이름
② 단계 상세 세부사항 첫 항목: 작업자 노노윤기 김김주웅 공동
③ 할일 상세 세부사항 첫 항목: 작업자 노노윤기 김김주웅 공동
   배정 셀렉트 라벨 남아있나: true
```

스크린샷으로 확인한 것.

- **단계 상세**: `세부 사항` → **작업자**(노윤기, 김주웅 공동) → 프로젝트 → 시작일 → 종료일
- **할 일 상세**: `세부 사항` → **작업자**(노윤기, 김주웅 공동) → 프로젝트 → 단계 → **작업자 배정**(셀렉트) → 예정일 → 완료날짜
- **프로젝트 상세**: 제목 아래 `진행률 0% · 작업자 [노] 노윤기`

두 오버레이에서 작업자가 **같은 자리(세부 사항 첫 항목)** 에 있는 것을 확인했다.

### 검증 중 겪은 것

첫 두 번의 실행이 "프로젝트를 찾을 수 없습니다"로 실패했다. DB를 확인하니 데이터는 멀쩡했고(그룹 3·프로젝트 4·`p-asg` 존재), **파일 수정 직후 dev 서버 재컴파일로 첫 로드가 느려** 스토어가 로드되기 전에 스크립트가 DOM을 읽은 것이었다. 대기 시간을 늘려 해소했다.

덧붙여 **프로젝트 상세는 로딩 중에도 "프로젝트를 찾을 수 없습니다"를 보여준다**(로딩 상태 구분 없음). 이번 변경과 무관한 기존 동작이라 손대지 않았다.

### 정리

픽스처 프로젝트를 삭제했다. 잔여 프로젝트 0건·요청 0건을 DB에서 확인했다(요청은 대상 삭제 시 cascade).

## 알려진 이슈 / 주의점

### 프로젝트 상세만 위치가 다르다

aside가 없어 헤더에 뒀다. 세 화면을 오가면 작업자 칸의 위치가 오른쪽(오버레이) ↔ 위쪽(페이지)으로 바뀐다. 완전히 같은 자리로 맞추려면 프로젝트 상세에도 세부 사항 aside를 만들어야 하는데, 그 화면 레이아웃을 크게 바꾸는 일이라 이번 범위 밖으로 두었다.

### 단계 작업자는 프로젝트에서 빌려온 값이다

단계마다 주담당이 다른 운영 방식이라면 맞지 않는다. 화면상 프로젝트 상세와 단계 상세의 작업자가 **항상 같아 보인다** — 공동 작업자만 단계별로 다르다. `Stage.assigneeId` 추가가 정확한 해법이다.

### "공동" 표시는 주 작업자가 있을 때만 붙는다

주 작업자가 없고 공동 작업자만 있으면 전부 구분 없이 나열된다. 첫 번째가 주 작업자라는 규칙이 깨지는 경우라 의도적으로 뺐지만, 목록만 보면 누가 주담당인지 알 수 없다.

### 담당자 보드와 용어가 어긋난다

화면 표기는 "작업자"로 통일했으나 작업 현황의 **"담당자 보드"** 는 그대로다. `assigneeId` 기준으로 컬럼을 묶는 같은 개념이므로, 다음에 그 화면을 손볼 때 함께 맞추는 편이 낫다.
