# feat: 내 할일에 프로젝트 생성 버튼과 프리셋 적용 모드 추가

## 작업 요약

`내 할일` 헤더에 **프로젝트 생성** 버튼을 달고, 프리셋을 골라 시작일만 정하면 프로젝트가 만들어지는 다이얼로그를 붙였다. 태스크 1의 API와 태스크 2의 미니 캘린더를 결합한 것이다.

**직접 만들기 모드는 후속 태스크(4)다.** 이번 커밋에는 모드 전환 탭을 만들지 않았다 — 한쪽만 동작하는 탭은 그 자체로 죽은 UI이고, 이 저장소에서 방금 두 번(docs/81의 기간 세그먼트, 그 전의 뷰 탭) 걷어낸 패턴이기 때문이다. 태스크 4에서 두 모드가 모두 생길 때 전환 UI를 함께 넣는다.

## 변경 파일 내역

### 신규 `src/components/features/projects/project-create-dialog.tsx`

프리셋 목록 → 선택 → 이름 → (마스터만) 그룹 → 시작일 → 생성.

- **이름 기본값**: 프리셋을 고르면 프리셋 이름이 채워진다. 단 사용자가 이미 입력한 이름이 있으면 덮어쓰지 않는다.
- **그룹 선택은 마스터에게만** 노출한다. 스탭은 서버가 세션 그룹으로 강제하므로(`resolveProjectGroupId`) 물어볼 필요가 없고, 물어보면 오히려 남의 그룹을 고를 수 있는 것처럼 보인다.
- **색**: 스토어의 `addProject`와 같은 순번 규칙(`PROJECT_COLORS[전체 개수 % 8]`)을 따른다.
- **열 때마다 초기화**된다. 이전 입력이 남아 있으면 실수로 그대로 만들게 된다. 다만 이펙트로 리셋하지 않는다 — 아래 "린트가 잡은 설계 문제" 참고.
- 프리셋이 하나도 없으면 목록 대신 안내 문구를 보여준다.

#### 린트가 잡은 설계 문제 — 이펙트 리셋 대신 마운트 주기로

처음에는 `open`이 바뀔 때 `useEffect`에서 상태를 하나씩 되돌렸는데 ESLint가 막았다.

```
react-hooks/set-state-in-effect  (setGroupId(""))
```

규칙을 우회하는 대신 구조를 바꿨다. **Radix Dialog는 닫힐 때 `DialogContent`를 언마운트**하므로, 폼 본문을 `CreateFromPresetForm` 내부 컴포넌트로 빼면 다시 열 때 `useState` 초기값이 그대로 다시 잡힌다. 리셋 이펙트가 통째로 사라졌고, "닫았다 열면 깨끗해야 한다"는 요구를 React의 마운트 주기가 대신 보장한다.

남은 이펙트는 프리셋 목록 fetch 하나뿐이고, 여기서의 `setState`는 비동기 콜백 안이라 규칙 대상이 아니다.

> 이 건은 **검증 명령의 종료코드를 잘못 읽어 하마터면 놓칠 뻔했다.** `npm run lint | tail`의 `$?`는 `tail`의 것이라 항상 0이다. 파일로 받아 `$?`를 직접 확인하도록 바꿨다.

## 검증 중 잡은 자체 결함 — 생성 후 다이얼로그가 닫히지 않았다

첫 검증에서 프로젝트는 만들어졌는데 다이얼로그가 열린 채 남았다. 원인은 순서였다.

```ts
await cache.refresh();     // ← 여기서 실패하면
onOpenChange(false);       // ← 여기에 도달하지 못한다
```

목록 갱신이 실패하면 **이미 생성이 끝난 상태인데도 창이 남아, 사용자가 다시 눌러 같은 프로젝트를 중복 생성**하게 된다. 생성 성공과 목록 갱신은 성패를 함께 묶을 이유가 없다.

```ts
onOpenChange(false);            // 생성이 끝났으면 먼저 닫는다
cache.refresh().catch(() => {}); // 갱신 실패는 생성 결과에 영향이 없다
```

수정 후 재검증에서 정상 종료를 확인했다.

## ⚠️ 검증이 막힌 부분 — 공유 개발 DB의 선반영 마이그레이션

**단계 생성을 검증하지 못했다. 이 브랜치의 결함이 아니다.**

`stage-order` 워크트리 세션이 아직 머지도 PR도 하지 않은 마이그레이션을 **공유 개발 DB에 이미 적용**해 두었다.

```sql
ALTER TABLE "Stage" ADD COLUMN "order" INTEGER NOT NULL DEFAULT 0;
UPDATE ...                                              -- 기존 행 번호 매기기
ALTER TABLE "Stage" ALTER COLUMN "order" DROP DEFAULT;  -- ← 이 줄
CREATE UNIQUE INDEX "Stage_projectId_order_key" ON "Stage"("projectId", "order");
```

기본값을 떨어뜨렸기 때문에 `order`를 넣지 않는 **모든** 단계 생성이 실패한다.

```
P2011  null value in column "order" of relation "Stage" violates not-null constraint
```

**이 브랜치만의 문제가 아니다.** 진단용 라우트로 확인한 결과 트랜잭션 밖의 평범한 `db.stage.create`(= main의 기존 `createStage`)도 똑같이 실패한다. 지금 개발 DB에서는 **어느 세션이든 단계를 만들 수 없다.**

- 프로덕션은 영향 없다. 해당 마이그레이션이 머지되지 않아 운영 DB에는 `order` 컬럼 자체가 없다.
- 이쪽에서 손쓸 수 없다. `schema.prisma`에 `Stage.order`가 없으므로 Prisma 클라이언트가 그 필드를 받지 않는다. **그쪽 스키마 변경이 머지돼야 대응할 수 있다.**

사용자에게 보고했고 "보고만 하고 대기"로 결정됐다. `stage-order` 머지 후 이 브랜치를 리베이스해 `compose.ts`에 `order`를 넣고(프로젝트별 `1..N`, 유니크 인덱스 `(projectId, order)` 준수) 단계 생성까지 재검증한다.

## 알려진 이슈 / 후속

- **단계 생성 재검증이 남아 있다.** `stage-order` 머지 후 리베이스 → `compose.ts`에 `order` 반영 → 재검증.
- 남은 태스크: (4) 직접 만들기 모드 + 모드 전환 UI.
- 진단 과정에서 임시로 만든 `/api/zz-probe` 라우트는 원인 특정 후 삭제했다. 최종 diff에 없다.
- 생성 직후 새 프로젝트로 이동하지 않는다. 사이드바에는 바로 나타나므로 사용자가 눌러 들어가는 흐름이다. 자동 이동이 필요한지는 사용해 보고 판단할 사항이다.
- docs 번호는 작성 시점 기준 88이다. 병렬 작업이 많아 머지 순서에 따라 조정이 필요할 수 있다.
