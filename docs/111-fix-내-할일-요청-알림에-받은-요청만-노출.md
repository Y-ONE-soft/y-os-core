# 111. fix: 내 할일 요청 알림에 받은 요청만 노출

## 작업 요약

내 할일 화면 하단의 `요청 알림` 띠가 **보낸 요청까지 함께** 보여주던 것을 **받은 요청만** 보여주도록 좁혔다.

## 문제

이 띠는 "나한테 온 것"을 확인하는 자리인데 내가 보낸 요청도 섞여 나왔다. 실제 화면에서 3건 중 2건이 보낸 요청이었고, 그 카드에서 할 수 있는 동작은 `요청 취소`뿐이라 알림으로서 할 일이 없었다.

```
요청 알림 · 3
[할일 요청 · 받은 요청] 마스터 → 나          수락 / 거절
[도움 요청 · 보낸 요청] 나 → 마스터          요청 취소   ← 알림으로서 불필요
[할일 요청 · 보낸 요청] 나 → 마스터          요청 취소   ← 알림으로서 불필요
```

## 변경 내용

### `src/components/features/my-work/my-work-requests.tsx`

- `direction === "received"`로 거른 뒤 정렬·집계한다. `요청 알림 · N`의 N도 **받은 요청 중 대기 중**만 센다
- 빈 상태 문구를 `주고받은 요청이 없습니다` → `받은 요청이 없습니다`로
- 헤더 주석을 실제 역할에 맞게 고침

**보낸 요청은 없어지지 않는다.** 이 띠 옆의 `전체 보기` 링크가 알림 페이지(`/notifications`)로 가고, 거기서 방향 필터와 함께 전체를 볼 수 있다. 보낸 요청의 취소도 그 화면에서 한다.

서버는 그대로 두고 화면에서만 걸렀다. `/api/admin/requests`는 보낸 요청·받은 요청을 모두 돌려주고 알림 페이지가 그것을 쓰기 때문이다.

## 진행 중 바로잡은 오판

요청 데이터를 지워 달라는 요청을 받고 **"화면의 요청 카드는 하드코딩 자리표시"라고 잘못 단정**했다. 근거는 개발 DB의 `Request` 테이블이 0건이고 개발·프로덕션 API가 모두 빈 배열을 돌려준 것이었는데, 사용자 화면에는 실제 단계·할일(`123`, `서비스 시스템 설계`, `프로젝트 정의서 작성하기`)을 참조하는 카드가 3건 떠 있었다.

코드를 다시 보니 `my-work-requests.tsx`는 `use-requests` → `/api/admin/requests`로 **실데이터를 읽는다.** 자리표시가 아니다. 내가 접근한 DB(개발 env)와 사용자가 보던 화면의 DB가 달랐던 것으로 보인다 — 개발 DB의 단계는 `123`·`서비스 시스템 설계`·`123`인데 프로덕션은 `담당자보드단계`·`프로젝트 정의`·`123`으로 내용이 달랐다.

조회가 비었다는 사실만으로 "존재하지 않는다"고 결론 내린 것이 잘못이었다. 결과적으로 요청의 본뜻은 데이터 삭제가 아니라 **보낸 요청을 알림에서 빼는 것**이었다.

### 요청 데이터 정리

사용자 요청으로 `Request` 테이블의 남은 행을 비웠다 — `HELP/PENDING 마스터→김주웅 대상=서비스 설계서 작성` 1건. 삭제 후 프로덕션 API가 `{"requests":[]}`를 돌려주는 것까지 확인했다.

**앞서 "DB가 다르다"고 본 것도 오판이었다.** 개발 env로 받은 `DATABASE_URL`과 프로덕션은 **같은 DB**였고, 조회 결과가 달랐던 것은 그 사이 데이터가 만들어지고 바뀌었기 때문이다(단계 이름도 다른 세션들이 계속 바꾸고 있었다). 같은 행 `rq-97789124-…`가 양쪽에서 동일하게 조회되는 것으로 확인했다.

`Request`에는 DELETE 엔드포인트가 없어 Prisma로 직접 지웠다. 요청을 지우는 화면 동선이 필요하면 별도 과제다(현재는 수락·거절·취소로 상태만 바뀐다).
