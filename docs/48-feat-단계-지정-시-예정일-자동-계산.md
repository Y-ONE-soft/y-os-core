## 1. 작업 요약

작업을 프로젝트·단계에 지정하면 **예정일이 자동으로 잡힌다** — 단계 시작일과 오늘 중 **더 늦은 날짜**. 이미 시작한 단계에 뒤늦게 넣은 작업이 과거 날짜에 놓이지 않게 하려는 규칙이다.

사용자 확인에 따라 **"시작일 없는 단계"라는 상태를 없앴다** — 단계 추가 시 날짜를 비워두면 시작·종료 모두 오늘로 잡힌다.

## 2. 규칙

| 상황 | 예정일 |
|---|---|
| 단계 시작일이 **미래** | 단계 시작일 |
| 단계 시작일이 **과거·오늘** | 오늘 |
| 백로그·미배정으로 이동 | **해제(null)** — 일정 미정 상태로 되돌린다 |

`scheduleFor(stageStartDate)` 한 함수로 표현했고, `assignTask`가 소속을 바꿀 때 함께 계산해 **한 번의 PATCH**로 저장한다(`projectId`·`stageId`·`scheduledDate`).

### 날짜를 클라이언트에서 계산하는 이유

`todayISO()`는 **브라우저 로컬 기준**으로 오늘을 만든다. 서버(Vercel = UTC)에서 계산하면 한국 시간대 기준으로 하루가 어긋나는 구간이 생긴다. 프로젝트가 날짜를 `YYYY-MM-DD` 문자열로 다루는 이유("TZ 이슈 회피", `Stage.startDate` 주석)와 같은 맥락이다.

## 6. 알려진 사항 / 후속 과제

- **캘린더에는 아직 반영되지 않는다** — 캘린더가 자리표시 상수(`CAL_OVERLAYS`) 기반이라, 예정일이 저장돼도 화면의 날짜 칸으로 이동하지 않는다. 캘린더 실데이터 전환은 사용자 결정에 따라 이번 요청 범위 밖(별도 요청)
- 기존에 날짜 없이 만들어진 단계가 DB에 남아 있으면 `scheduleFor`가 오늘로 처리한다 — 소급 보정(백필)은 하지 않았다
- 단계의 시작일을 나중에 바꿔도 이미 배정된 작업의 예정일은 따라가지 않는다 — 재계산 정책은 별도 논의 대상
