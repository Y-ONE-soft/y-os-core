# feat: 기존 프로젝트에 프리셋 적용 API 추가

- **예정 커밋 메시지**: `feat: 기존 프로젝트에 프리셋 적용 API 추가`
- **작업일**: 2026-07-22
- **작업 브랜치**: `프로젝트-상세-프리셋-버튼` (워크트리 `.claude/worktrees/project-detail-preset`)

---

## 1. 작업 내용 요약

**이미 만들어진 프로젝트에 프리셋을 적용하는 경로**를 서버에 추가했다. 화면 연결은 다음 태스크다.

기존에는 프리셋을 쓰는 길이 **프로젝트 생성 시점 하나뿐**이었다 (`POST /api/admin/projects/from-preset` — 프로젝트를 새로 만들면서 단계·할일을 함께 채운다). 이미 만든 빈 프로젝트에 나중에 프리셋을 채워 넣을 수는 없었다.

## 2. 적용 범위 — 단계가 없는 프로젝트만

**단계가 하나라도 있으면 적용을 거부한다** (사용자 결정: "비활성화 해줘").

이어붙이기(append)나 덮어쓰기(replace) 중 어느 쪽이든 사용자가 만든 내용을 건드리게 된다. 화면에서도 단계가 있으면 버튼을 비활성화할 것이므로, 서버는 같은 규칙을 **최종 판정으로 다시 확인**한다 (화면 상태만 믿지 않는다).

빈 프로젝트 판정은 **트랜잭션 안에서** 한다. 밖에서 세면 동시 요청 두 건이 모두 "비어 있음"을 보고 함께 통과해 단계가 두 벌 생길 수 있다.

## 4. 설계 결정

| 결정 | 선택 | 이유 |
|---|---|---|
| 경로 | `POST /api/admin/projects/[projectId]/apply-preset` | 대상 프로젝트에 하는 동작이므로 프로젝트 하위 경로. `from-preset`(생성)과 헷갈리지 않는다 |
| 소유자 검증 | `getPreset(ownerId, presetId)` 그대로 재사용 | 남의 프리셋 id는 not found로 떨어진다 — `from-preset`과 동일한 방식 |
| 단계 생성 규칙 | `createProjectFromPreset`과 동일 | 날짜 오프셋(`presetStageSpan`)·`order` 1..N·할일 정렬 id(`orderedId`)를 그대로 따른다. 두 경로가 다른 결과를 내면 안 된다 |
| 실패 응답 | `ProjectNotEmptyError` → 400 + 메시지 | 화면이 그대로 보여줄 수 있는 한국어 메시지를 담았다 |
| 트랜잭션 | 단계·할일 전체를 한 트랜잭션 | 중간 실패로 절반만 채워지면 "빈 프로젝트"가 아니게 되어 재시도도 막힌다 |

## 5. 검증

`npm run lint` ✓ · `npx tsc --noEmit` ✓ · `npm run build` ✓

API 동작은 화면이 붙는 태스크 2에서 함께 검증했다 (docs/115). 요약하면:

| 검증 | 결과 |
|---|---|
| 빈 프로젝트에 적용 | **200** — 단계·할일 생성됨 |
| 같은 프로젝트에 재적용 | **400** `이미 단계가 있는 프로젝트에는 프리셋을 적용할 수 없습니다.` |

## 6. 알려진 사항 / 후속 과제

- **이어붙이기·덮어쓰기는 지원하지 않는다.** 단계가 있는 프로젝트에 프리셋을 얹는 요구가 생기면 별도 결정이 필요하다 (기존 데이터 처리 방식).
- `createProjectFromPreset`과 단계 생성 로직이 거의 같다. 지금은 트랜잭션 참여 방식이 달라(하나는 프로젝트도 함께 만든다) 공유하지 않았는데, 세 번째 사용처가 생기면 공통 함수로 뽑는 게 맞다.
