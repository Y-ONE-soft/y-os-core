# fix: 아바타 드롭다운에서 죽은 항목 제거하고 알림을 실제 페이지로 연결

## 작업 요약

헤더 아바타 드롭다운을 정리했다.

- **"프리셋 관리" 제거** — 사용자 요청. `onSelect`가 없어 눌러도 아무 일도 없었고, 실제 프리셋 화면은 사이드바 "프리셋"(`/projects/presets`)에 이미 있었다.
- **"설정" 제거** — 사용자 요청. 같은 이유로 동작하지 않는 자리표시였다.
- **"알림"을 `/notifications`로 연결** — 이전 태스크(docs/107)에서 만든 페이지로 보낸다.
- **배지를 실데이터로 교체** — 하드코딩 `2` → 받은 요청 중 대기 건수. 0건이면 배지를 숨긴다.

결과적으로 메뉴는 `알림 · 내 정보 · 로그아웃` 세 항목이 되고, **모두 실제로 동작한다.**

## 배경 — 세 항목 모두 거짓말을 하고 있었다

변경 전 [user-menu.tsx](../src/components/layout/user-menu.tsx)는 이랬다.

```tsx
const NOTIFICATION_COUNT = 2;   // 실제 건수와 무관한 고정값
...
<DropdownMenuItem className={ITEM_CLASS}>          {/* onSelect 없음 */}
  <span className="flex-1">알림</span>
  <Badge>{NOTIFICATION_COUNT}</Badge>              {/* 항상 2 */}
</DropdownMenuItem>
<DropdownMenuItem className={ITEM_CLASS}>프리셋 관리</DropdownMenuItem>  {/* onSelect 없음 */}
<DropdownMenuItem className={ITEM_CLASS}>설정</DropdownMenuItem>        {/* onSelect 없음 */}
```

`NOTIFICATION_COUNT` 위 주석은 "알림 도메인이 아직 없어 Figma 예시 값을 자리표시로 둔다"였는데, 이는 사실과 달랐다. 요청 도메인(`Request`)이 이미 알림 데이터를 갖고 있었고 대기 건수도 `my-work-requests.tsx`가 이미 계산하고 있었다. docs/107에서 이를 확인해 페이지를 만들었고, 이 태스크가 배지와 연결을 실데이터로 맞춘다.

## 변경 내용

### 배지 — 받은 요청 중 대기 건수

```tsx
// 배지는 "내가 처리할 것"만 센다 — 받은 요청 중 아직 응답하지 않은 건수.
// 보낸 요청은 상대가 처리할 몫이라 제외한다.
const { requests } = useRequests();
const pendingCount = requests.filter(
  (item) => item.status === "PENDING" && item.direction === "received",
).length;
```

`direction === "received"` 조건이 핵심이다. 내가 보낸 대기 요청은 상대가 처리할 몫이라 내 배지에 세면 안 된다. 알림 페이지 헤더의 대기 건수와 같은 기준이라 두 숫자가 어긋나지 않는다.

내 할일 띠(`my-work-requests.tsx:135-137`)의 `요청 알림 · N`은 **방향 구분 없이 전체 대기 건수**를 세는데, 이건 그 섹션이 주고받은 요청을 함께 보여주기 때문이라 그대로 뒀다.

### 0건이면 배지를 숨긴다

```tsx
{/* 처리할 게 없으면 배지를 숨긴다 — 0이 붙어 있으면 볼 게 있는 것처럼 읽힌다 */}
{pendingCount > 0 && <Badge …>{pendingCount}</Badge>}
```

### 연결

`알림` → `router.push("/notifications")`. `내 정보`가 이미 쓰던 방식을 그대로 따랐다.

`useRequests()`는 모듈 수준 스토어라 헤더가 구독해도 요청이 중복 발생하지 않는다(`use-requests.ts:37-54`의 `inflight` 병합). 헤더는 로그인 상태에서 항상 떠 있으므로, 이제 앱 어디서든 요청 목록이 한 번은 로드된다.

### 실제 화면 (프로덕션 빌드 + 헤드리스 Chrome)

검증용 요청 3건(master01이 받은 것 2·보낸 것 1)을 만든 상태에서:

```
[master01] 메뉴 항목: [ '알림 2', '내 정보', '로그아웃' ]
알림 클릭 후 URL: http://localhost:3018/notifications   페이지 h1: 알림
내 정보 클릭 후 URL: http://localhost:3018/me
```

- **프리셋 관리·설정이 목록에 없다** — 제거 확인.
- **배지가 `2`** — 받은 대기 2건과 일치. 보낸 대기 1건은 세지 않았다(의도대로).
- 알림·내 정보 모두 실제로 이동.

검증 데이터를 원복해 대기 0건으로 만든 뒤 다시 확인:

```
메뉴 항목(대기 0건): [ '알림', '내 정보', '로그아웃' ]
```

숫자가 사라져 **배지 숨김도 확인**했다.

`pageerror` 없음.

### 검증 하네스 메모

docs/107에 적은 대로 이 환경의 헤드리스 Chrome은 앱 셸 안쪽으로 CDP 마우스 이벤트를 전달하지 못한다(기존 페이지에서도 재현). 추가로 이 태스크에서 알아낸 것:

- Radix `DropdownMenu`는 `pointerdown`으로 열리므로 DOM `click()`으로는 열리지 않는다.
- 트리거에 포커스 후 Enter도 통하지 않았다.
- **합성 `PointerEvent("pointerdown")`을 디스패치하면 정상적으로 열린다.** 이 방법으로 메뉴를 열어 검증했다.

즉 메뉴 항목·배지·이동은 실제 React 핸들러를 통해 확인했고, 물리적 마우스 클릭만 하네스 한계로 재현하지 못했다.

### 공유 개발 DB 원복

docs/107과 동일한 절차로 되돌렸다 — 만든 요청 3건 삭제, 원복 후 `Request` 0건·`requestedCollaborators` 보유 단계 0건 확인. 이번 검증에서는 수락을 하지 않아 공동작업자 부수 효과는 발생하지 않았다.

검증 중 전체 단계 수가 7 → 4로 줄어든 것을 관측했는데, **원복 스크립트는 단계를 삭제하지 않는다**(요청 삭제와 `requestedCollaborators` 배열 수정만 한다). 공유 개발 DB를 쓰는 다른 세션의 변경으로 판단된다.

## 알려진 이슈

- **사이드바 "문의함" 배지 `4`는 여전히 하드코딩이다** (`context-nav.tsx:33`). 같은 종류의 거짓 배지지만 `/inbox` 페이지 자체가 없어 연결할 대상이 없다. 이번 요청 범위 밖이라 두었다.
- **`/tasks`·`/inbox`·`/wiki`·`/customers` 사이드바 링크는 페이지가 없어 404다.** Next.js가 프리페치하면서 콘솔에 404가 찍힌다. 이번 작업과 무관한 기존 결함이다.
- 알림 배지는 폴링하지 않는다 — 새 요청이 와도 스토어가 갱신되기 전까지 숫자가 그대로다(docs/107의 알려진 이슈와 동일).
- 헤더가 `useRequests()`를 구독하면서, 로그인 후 어느 화면에서든 요청 목록을 한 번 조회한다. 조회는 가볍고 스토어가 중복 요청을 병합하지만, 알림이 필요 없는 화면에서도 호출이 나가는 것은 사실이다.
