# 70. feat: 프로젝트 상세 프리셋 저장 버튼 및 저장 다이얼로그

## 작업 요약

프로젝트 상세 화면에 **프리셋 저장 버튼**과 **저장 다이얼로그**를 붙였다. docs/69에서 만든 도메인 API를 실제로 호출하는 첫 화면이다.

Figma 기준: Preset Save · New `161:511`, Overwrite `161:1573`, 버튼 `180:901`.

## 버튼 위치가 디자인에서 바뀐 지점

docs/18에서는 프리셋 저장 버튼을 **로드맵 카드 헤더**에 뒀다가, 이후 `82b0558`(로드맵 타임라인·드래그 전환)에서 헤더가 재구성되며 사라진 상태였다.

현재 디자인(`180:901`)은 **페이지 헤더 제목 줄 오른쪽**이다. 그래서 이번에는 `project-roadmap.tsx`가 아니라 `project-detail-page.tsx`에 넣었다. 제목 줄을 `justify-between`으로 바꾸고 오른쪽에 버튼을 놓는다.

## 다이얼로그 구성

| 영역 | 내용 |
| --- | --- |
| 제목 | 단계 프리셋 저장 |
| 설명 | 현재 단계와 할 일 구성이 그대로 저장됩니다. |
| 포함 구성 | `단계 N개 · 할 일 M개` — muted 박스 |
| 모드 스위처 | `새로 만들기` / `기존에 덮어쓰기` |
| 새로 만들기 | 프리셋 이름 입력 (`예: 표준 개발 프로세스`) |
| 덮어쓰기 | 대상 프리셋 Select + "선택한 프리셋이 현재 구성으로 대체됩니다." |
| 푸터 | 취소 · 저장 |

**포함 구성은 자리표시가 아니라 실제 값이다.** `useProjectBoard(projectId)`로 현재 보드의 단계 수와 할일 수를 세어 보여준다.

덮어쓰기 모드는 마운트 시 `fetchPresets()`로 내 프리셋 목록을 읽어 Select를 채운다. 프리셋이 하나도 없으면 Select 대신 "저장된 프리셋이 없습니다. 먼저 새로 만들어 주세요." 안내를 보여준다 — 빈 Select를 열게 두면 막다른 길이 된다.

이름 중복(409)이나 네트워크 오류는 다이얼로그 안에 메시지로 띄우고 닫지 않는다. 공통 래퍼 `ApiError`가 서버의 `{error}` 문구를 그대로 들고 오므로 별도 매핑이 없다.

## 조건부 마운트로 바꾼 이유

처음에는 `open`/`onOpenChange` prop을 받고 열릴 때마다 `useEffect`에서 입력을 초기화하려 했는데, `react-hooks/set-state-in-effect` 린트 규칙에 걸렸다.

부모가 **열려 있는 동안만 마운트**하도록 바꿔 초기화 자체를 없앴다(`{presetDialogOpen && <PresetSaveDialog … />}`). 닫으면 컴포넌트가 통째로 사라지므로 이전 입력이 남아 **잘못된 대상에 덮어쓸 여지가 없다.** 규칙을 우회한 게 아니라 상태 수명이 실제로 짧아진 것이다.

같은 화면의 `StageDetailOverlay`도 부모에서 조건부로 렌더하는 방식이라 패턴이 어긋나지 않는다.

## 변경 파일

### `src/components/features/projects/preset-save-dialog.tsx` (신규)

- props는 `projectId`·`onClose` 둘뿐. 열림 상태는 마운트 여부가 대신한다
- 모드 스위처는 Figma의 세그먼트 컨트롤 그대로 — 활성 탭만 `bg-background` + 미세 그림자
- 저장 중에는 버튼이 `저장 중…`으로 바뀌고 비활성화된다
- 저장 가능 조건: 새로 만들기는 이름이 비지 않을 것, 덮어쓰기는 대상이 선택됐을 것

### `src/components/features/projects/project-detail-page.tsx`

- 제목 줄을 `justify-between`으로 바꾸고 오른쪽에 `프리셋 저장` 버튼(`variant="outline"`)
- `presetDialogOpen` 상태 추가, 다이얼로그를 조건부 렌더

## 미검증 항목

**브라우저에서 버튼을 눌러 다이얼로그가 열리고 저장되는 흐름은 확인하지 못했다.** 저장소에 브라우저 자동화 도구가 없다.

SSR HTML로도 확인이 안 되는데, 워크스페이스 캐시의 `getServerSnapshot()`이 항상 빈 값을 돌려주기 때문이다(하이드레이션 안정용). 서버에서는 `groups`가 비어 프로젝트를 못 찾고 "프로젝트를 찾을 수 없습니다" 분기가 렌더된다 — 페이지 200과 이 폴백까지만 확인했다.

## 진행 중 겪은 것

dev 서버가 `Jest worker encountered 2 child process exceptions`로 500을 뱉었다. 코드 문제가 아니라 워커 크래시였고, `.next` 삭제 후 재기동하니 정상화됐다. 워크트리 여러 개에서 dev 서버가 동시에 도는 상황이라 메모리 압박으로 보인다. 포트도 3037이 이미 쓰이는 일이 있어 3041로 옮겼다.
