# feat: 프로젝트 생성 직접 만들기 모드와 생성 방식 전환 구현

## 작업 요약

프로젝트 생성 다이얼로그에 **직접 만들기** 모드를 추가하고, 두 생성 방식을 오가는 전환 UI를 넣었다. 요청 사이클의 마지막 태스크로, 이로써 기능이 완성됐다.

| 모드 | 입력 | 결과 |
| --- | --- | --- |
| 프리셋 적용 | 프리셋 + 시작일 | 프리셋 구성 그대로 (단계 + 할일) |
| **직접 만들기** | **드래그로 기간 + 단계 수** | **기간을 균등 분할한 단계** |

## 전환 UI를 이번에 넣은 이유

태스크 3(docs/102)에서는 모드 탭을 만들지 않았다. 직접 만들기가 없는 상태에서 탭을 만들면 한쪽이 눌러도 아무 일 없는 죽은 UI가 되고, 이 저장소에서 최근 두 번 걷어낸 패턴이기 때문이다. 두 모드가 모두 실제로 동작하는 지금 전환 UI를 함께 넣었다.

`role="tablist"` / `role="tab"` / `aria-selected`를 붙여 실제 상태를 노출한다 — 전에 걷어낸 세그먼트들이 `aria-pressed`만 하드코딩해 두고 조작은 불가능했던 것과 대비된다.

## 미리보기가 서버와 어긋나지 않는 이유

직접 만들기는 "만들기 전에 어떻게 쪼개지는지" 보여준다. 이 미리보기는 서버 생성과 **같은 모듈**을 쓴다.

```ts
import { evenSplitError, splitRangeEvenly, inclusiveDays } from "@/lib/stage-plan";
```

태스크 1에서 이 순수 모듈을 `src/components`가 아니라 `@/lib`에 둔 것이 여기서 회수된다. 서버(`compose.ts`)와 화면이 같은 함수를 호출하므로 미리보기와 실제 결과가 구조적으로 어긋날 수 없다. 검증에서도 미리보기 4행과 생성 결과 4단계의 날짜가 정확히 일치했다.

거부 사유도 `evenSplitError` 하나를 공유한다. 화면이 보여주는 문구와 서버가 400으로 돌려주는 문구가 같은 문자열이다.

## 변경 파일 내역

### `src/components/features/projects/project-create-dialog.tsx`

`CreateFromPresetForm` → `ProjectCreateForm`으로 확장.

- **모드 상태**(`preset` | `even`)와 두 모드가 공유하는 값(이름, 그룹, 제출 상태)을 한 곳에서 관리한다. 모드별 값(프리셋 선택·기준일 / 기간·단계 수)은 각각 따로 둔다.
- 모드를 바꾸면 이전 모드에서 난 에러 문구를 지운다. 다른 모드의 실패 메시지가 남아 있으면 혼란스럽다.
- **미리보기**: `1단계 2026-12-01 ~ 2026-12-05 (5일)` 형태로 각 단계의 기간과 일수를 나열한다.
- **검증 실패 시** 만들기를 막고 사유를 그대로 보여준다. (예: `기간이 3일이라 단계를 9개로 나눌 수 없습니다. 단계 수를 3개 이하로 줄이세요.`)
- 다이얼로그 설명 문구도 모드에 따라 바뀐다.

제출은 모드에 따라 `createProjectFromPresetApi` / `createProjectWithEvenStagesApi`로 갈리고, 공통 입력(id·그룹·이름·색)은 한 객체로 묶어 중복을 없앴다.

### 직접 만들기 모드 19건 — 전부 통과

| 묶음 | 확인 내용 |
| --- | --- |
| 모드 전환 | 기본값 프리셋 적용, 직접 만들기로 전환됨, 설명 문구 변경, 프리셋 목록 숨김 |

| 경계 | 단계 수 30(> 20일)이면 만들기 비활성 + 거부 사유 노출, 되돌리면 다시 활성 |

- `npm run lint` 통과 (종료코드 0)
- `npm run build` 통과 (종료코드 0)

### 검증 도구에서 고친 것 세 가지

모두 **테스트 쪽 문제였고 제품 결함이 아니었다.** 기록해 두는 이유는 같은 함정을 다시 밟지 않기 위해서다.

1. **로그인이 네이티브 GET으로 새어 나갔다.** `domcontentloaded` 직후 클릭하면 하이드레이션 전이라 `onSubmit` 핸들러가 없어 폼이 `/login?username=...&password=...`로 이동해 버린다. `networkidle` 대기 + `#login-username` 같은 id 선택자로 바꿨다.
2. **이월 칸 때문에 월 이동이 조기 종료됐다.** 날짜 칸 존재로 대상 월을 판정했더니 11월 그리드의 트레일링 `12/1`에서 멈춰 같은 달의 `12/20`을 찾지 못했다. 캘린더가 이월 칸도 실제 날짜로 그리는 것은 의도된 동작(월 경계 드래그 지원)이므로, **월 제목**으로 판정하도록 고쳤다.
3. **`innerText()` 줄 수로 항목을 셌다.** `1단계`와 날짜가 별도 줄로 나와 4단계가 8행으로 세어졌다. `li` 단위로 세도록 바꿨다. 값 자체는 처음부터 정확했다.

또한 스크립트가 중간에 예외로 죽으면 `ok()` 호출이 하나도 없어 **"전부 통과"로 오보**되던 문제를 고쳤다(`catch`에서 실패로 계상).

## 알려진 이슈 / 후속

- **단계 수 입력이 `type="number"`라 빈 값이 될 수 있다.** 지우면 `Number("") || 0` → 0이 되어 검증에 걸리고 만들기가 막힌다. 안내 문구가 뜨므로 막다른 길은 아니지만, 스피너 대신 명시적인 증감 UI가 더 나을 수 있다.
- **직접 만들기는 단계 이름이 `1단계`, `2단계`… 고정**이다. 생성 후 보드에서 바꾼다.
- **생성 직후 새 프로젝트로 이동하지 않는다.** 사이드바에 바로 나타나므로 눌러 들어가는 흐름이다. 자동 이동이 필요한지는 써 보고 판단할 사항.
- 미니 캘린더의 키보드 조작은 여전히 없다(docs/101에 기록). 접근성 후속 후보.
- docs 번호는 병렬 머지로 여러 번 밀렸다. 이 브랜치의 다섯 문서는 push 직전에 한 번에 재조정한다.
