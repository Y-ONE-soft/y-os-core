# feat: 프리셋 상세 구성 보기 추가

## 작업 요약

프리셋 관리 페이지에서 행을 펼치면 **그 프리셋에 담긴 단계·할일 구성**을 볼 수 있게 했다. 목록만으로는 "단계 3개 · 할일 7개"라는 숫자뿐이라 어떤 프리셋인지 구분할 수 없었다.

서버 변경 없이 기존 `GET /api/admin/presets/[presetId]`만 쓴다.

## 상대 일정을 어떻게 보여줄 것인가

프리셋은 절대 날짜를 담지 않는다. `types/preset.ts`의 정의부터가 그렇다.

> 날짜는 기준일 기준 상대 오프셋으로만 담는다 (적용 시 지정한 시작일부터 재현).

그래서 화면 표기도 상대값이어야 한다. 필드가 셋(`offsetDays`·`durationDays`, 할일의 `offsetDays`)이고 각각 `undefined`가 의미를 갖기 때문에 경우가 여러 개다.

| 데이터 | 표기 | 의미 |
| --- | --- | --- |
| `offsetDays: 0` | `기준일` | 프로젝트 시작일 당일 |
| `offsetDays: 8` | `기준일 +8일` | 8일 뒤 |
| `offsetDays: undefined` | `일정 미정` | 저장 당시 날짜가 없던 단계·할일 |
| `durationDays: 19` | `· 19일간` | |
| `durationDays: undefined` | `· 종료일 없음` | 저장 당시 진행형 막대였던 단계 |

`offsetDays: 0`을 `기준일 +0일`로 쓰지 않고 그냥 `기준일`로 둔 것은, 대부분의 첫 단계가 0이라 `+0일`이 반복되면 시각적 잡음이 되기 때문이다.

**0을 falsy로 다루는 함정을 피했다.** `offsetLabel()`은 값이 없을 때만 `null`을 돌려주고 `0`이면 `"기준일"`이라는 문자열을 돌려준다. 호출부의 `if (!start)` 검사가 문자열을 보므로 `offsetDays: 0`이 "일정 미정"으로 새지 않는다. 숫자를 그대로 검사했다면 첫 단계가 전부 미정으로 표시됐을 것이다.

펼친 패널 아래에 **`적용할 때 정하는 시작일이 기준일이 됩니다`** 를 한 줄 붙였다. "기준일"이라는 말이 처음 보는 사람에게는 뜻이 통하지 않는다.

## 상호작용

- **펼치기** — 행 왼쪽(이름·요약)이 토글 버튼. `ChevronRight`/`ChevronDown`으로 상태를 보인다(`workload-roadmap`의 그룹 접기와 같은 아이콘 규약). `aria-expanded`를 준다.
- **여러 개 동시에 펼칠 수 있다.** 프리셋을 비교하는 화면이므로 아코디언(하나만)으로 제한하지 않았다.
- **상세는 펼칠 때만 읽는다.** 목록 API는 요약만 주므로 구성은 별도 호출이다. 한 번 읽은 것은 캐시해 다시 접었다 펴도 재요청하지 않는다. 실패한 것은 다시 시도한다.
- **로딩** — 펼친 자리에 `Skeleton` 한 줄.

### 버튼 중첩을 피한 구조

행 전체를 토글 버튼으로 만들면 **삭제 버튼이 버튼 안에 들어가** 유효하지 않은 HTML이 된다. 그래서 행을 `<li>`로 두고 그 안에 토글 버튼과 삭제 버튼을 형제로 배치했다. 토글 버튼이 `flex-1`을 먹어 이름 영역 전체가 클릭 대상이 된다.

### setState 업데이터 안에서 fetch하지 않는다

처음에는 `setDetails` 업데이터 안에서 캐시 여부를 확인하고 그 자리에서 `fetchPreset()`을 불렀다. **업데이터는 StrictMode에서 두 번 실행되므로 요청이 두 번 나간다.**

캐시 확인과 요청을 업데이터 밖으로 옮겨 해결했다. 이벤트 핸들러이므로 `openIds`·`details`를 클로저로 읽어도 안전하다.

```
const willOpen = !openIds.has(presetId);
setOpenIds(...)                       // 토글은 항상
if (!willOpen) return;                // 접을 때는 요청하지 않는다
const cached = details[presetId];
if (cached !== undefined && cached !== "error") return;
setDetails(prev => ({ ...prev, [presetId]: "loading" }));
void fetchPreset(presetId).then(...).catch(...)
```

docs/85의 `set-state-in-effect`와 원인은 다르지만(effect vs 업데이터) **"React가 여러 번 부를 수 있는 자리에 부수효과를 두지 않는다"** 는 같은 규칙이다.

## 변경 파일

- `DetailState` 타입 추가 — `"loading" | "error" | PresetDetail`. 미조회(`undefined`)와 로딩을 구분한다.

단계가 0개인 프리셋은 `단계가 없는 프리셋입니다`로 표시한다.

## 검증

### 상대 일정 표기 — 실데이터로 전 분기 확인

실 DB의 프로젝트로 프리셋을 만들어 상세를 읽고, 화면에 나갈 문구를 그대로 계산해 확인했다.

```
[단계] 서비스 시스템 설계  | 기준일 · 26일간
   - 프로젝트 정의서 작성하기 | 기준일 +16일
[단계] 프로젝트 정의       | 기준일 +8일 · 16일간
   - 내작업에서 만들기      | 기준일 +15일
   - 안녕하세요            | 일정 미정        ← offsetDays undefined 경로
[단계] 날짜기본값 단계      | 기준일 +23일 · 19일간
[단계] 개발 구현           | 기준일 +6일 · 19일간
```

`offsetDays: 0`(→ `기준일`), 양수(→ `기준일 +N일`), `undefined`(→ `일정 미정`) 세 경로가 모두 실제로 나왔다. 특히 **0이 "일정 미정"으로 새지 않는 것**을 실데이터로 확인한 것이 핵심이다.

확인 후 프리셋을 삭제해 **개발 DB를 원래 상태(0건)로 되돌렸다** — 전 세션 공유이기 때문이다.

| --- | --- |
| `npm run lint` | 통과 |
| `npx tsc --noEmit` | 통과 |
| `npm run build` | 통과 |

### 미검증

**펼침 동작 자체는 육안 확인을 못 했다.** 저장소에 브라우저 자동화 도구가 없고, 목록 행은 클라이언트에서 데이터를 받은 뒤 그려지므로 SSR HTML에 나오지 않는다. 확인한 것은 표기 계산과 API 왕복이고, 클릭 → 펼침 → 스켈레톤 → 구성 표시의 시각적 흐름은 사용자 몫으로 남는다.

## 알려진 이슈

- **삭제해도 `openIds`·`details`에 항목이 남는다.** 프리셋 id는 UUID라 같은 id가 다시 생길 수 없어 실질적인 문제는 없다. 정리 코드를 넣으면 삭제 경로가 복잡해져 두었다.
- **적용 기능은 여전히 없다.** 구성을 볼 수 있게 됐으니 "이 프리셋 적용" 버튼이 자연스러운 다음 단계지만, 서버에 적용 로직 자체가 없어 별도 작업이 필요하다(docs/85 참조).
- 이름 변경 API도 없다.
