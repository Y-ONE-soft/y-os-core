# 112. fix: 사용자가 바뀌면 요청 캐시를 초기화하고 다시 받아오기

## 증상

step01로 master를 지정해 요청을 보냈는데,

- master 화면에서 그 요청이 **보이지 않고**
- 대신 **"보낸 요청"으로 표시**됐다

## 원인

요청 목록은 **보는 사람에 따라 내용이 달라진다.** 서버가 `direction`(받은/보낸)을 뷰어 기준으로 계산해 내려준다.

```ts
// src/server/requests/service.ts
direction: row.toUserId === viewerId ? "received" : "sent",
```

그런데 `use-requests.ts`의 스토어는 **모듈 수준 전역**이고, 적재 여부를 `loaded` 불리언 하나로만 판단했다.

```ts
let requests: WorkRequest[] = [];
let loaded = false;
…
useEffect(() => {
  if (!loaded) void refresh();   // 사용자가 바뀌어도 loaded가 true면 그대로
}, []);
```

로그인·로그아웃은 `router.replace`라 **전체 새로고침이 없다.** `SessionProvider`는 `(main)` 레이아웃에 있어 다시 마운트되지만, 모듈 스토어는 그와 무관하게 살아남는다.

그래서 이런 순서가 된다.

1. step01 로그인 → 스토어에 step01 기준 목록 적재 (`loaded = true`)
2. 로그아웃 → 스토어는 그대로 남음
3. master 로그인 → `loaded`가 여전히 `true` → **재조회하지 않음**
4. master 화면에 step01의 목록이 뜨고, `direction`도 step01 기준(`sent`)으로 표시

받은 요청이 안 보인 것도, 남의 요청이 "보낸 요청"으로 보인 것도 같은 원인이다.

## 변경 내용

### `src/hooks/use-requests.ts`

**1. 적재 기준 사용자를 기록**

`loaded` 불리언만으로는 "누구 기준으로 담겼는지"를 알 수 없다. `loadedFor`에 사용자 id를 남긴다.

**2. 사용자가 바뀌면 즉시 비우고 다시 받는다**

```ts
const { user } = useSession();
const userId = user?.id ?? null;

useEffect(() => {
  if (loadedFor === userId && loaded) return;
  // 사용자가 바뀌었으면 이전 목록부터 버린다 — 남의 요청이 잠깐이라도 보이면 안 된다
  if (loadedFor !== userId) clear();
  if (!userId) return;
  loadedFor = userId;
  void refresh();
}, [userId]);
```

비로그인 상태(`userId === null`)면 비운 채 두고 조회하지 않는다.

**3. 세대 번호로 늦은 응답을 버린다**

사용자를 바꾸는 순간 이전 조회가 이미 날아가 있을 수 있다. 그 응답이 나중에 도착해 새 사용자의 목록을 덮으면 같은 증상이 재발한다.

`clear()`가 `generation`을 올리고, `refresh()`는 시작 시점의 세대를 기억해 두었다가 응답 처리 직전에 대조한다. 세대가 다르면 그 응답은 남의 것이므로 버린다. `finally`의 `inflight = null`도 같은 조건으로 막아, 새로 시작된 조회를 지우지 않게 했다.

### 서버 계약 확인

이 버그는 브라우저 클라이언트 캐시에서 나므로 저장소의 도구로는 재현할 수 없다. 대신 **서버가 사람별로 `direction`을 올바르게 내려주는지**를 개발 서버에 실제 요청을 넣어 확인했다 — 원인이 서버가 아니라 캐시임을 못 박기 위해서다.

step01로 master에게 요청을 만든 뒤 같은 행을 두 사람 시점으로 조회했다.

```
--- step01 시점 ---
  김주웅→노윤기  direction=sent
--- master 시점 ---
  김주웅→노윤기  direction=received
```

같은 행이 보는 사람에 따라 바르게 뒤집힌다. **서버는 정상이고 캐시만 문제였다.**

검증용으로 만든 행(`rq-verify-…`)은 삭제했다. 사용자가 테스트 중이던 요청 2건은 그대로 뒀다.

**미검증** — 실제 로그인 전환 동작은 프리뷰에서 사용자가 확인해야 한다. 브라우저 자동화 도구가 없다.

## 같은 구조의 다른 캐시

`workspace-cache.ts`도 모듈 전역에 `status` 플래그로 한 번만 적재하는 같은 방식이다. 다만 워크스페이스 응답은 **보는 사람과 무관한 전체 트리**라 사용자가 바뀌어도 내용이 틀리지 않는다(오래된 값일 수는 있다). 이번 범위에서는 손대지 않았다.

뷰어 기준으로 가공된 값을 내려주는 API가 앞으로 늘어나면 같은 함정이 반복될 수 있다. 캐시를 모듈 전역에 두려면 **"이 값이 보는 사람에 따라 달라지는가"를 먼저 확인**해야 한다.
