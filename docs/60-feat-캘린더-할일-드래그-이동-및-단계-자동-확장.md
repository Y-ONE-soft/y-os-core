## 1. 작업 요약

캘린더의 **할일 칩을 끌어 다른 날짜로 옮길 수 있다.** 옮긴 자리가 소속 단계의 기간 밖이면 **단계가 늘어나 할일을 덮는다** — 요청의 커버 규칙을 단계↔할일 방향으로 한 번 더 적용한 것이다.

## 2. 동작 규칙

| 상황 | 결과 |
|---|---|
| 단계 기간 **안**으로 이동 | 할일 예정일만 바뀌고 단계는 그대로 |
| 단계 **오른쪽 밖**으로 이동 | 단계 종료일이 그 날짜까지 늘어난다 |
| 단계 **왼쪽 밖**으로 이동 | 단계 시작일이 그 날짜까지 늘어난다 |
| 저장 | 손을 뗄 때 한 번. 단계가 늘어난 경우에만 단계도 함께 저장한다 |

프로젝트 박스는 단계들의 합집합이라 **단계가 늘어나면 박스도 자동으로 따라 늘어난다** — 요청의 "프로젝트도 하위를 커버" 규칙이 이 단계에서는 파생으로 지켜진다(자체 기간은 태스크 4).

## 3. 변경 내용

드래그 기반을 **단계·할일 공용으로 일반화**했다. 태스크 2에서 만든 포인터 처리(임계값 판정, 루트 캡처, 주 행 기준 일자 계산)를 그대로 쓰고, 대상만 판별식으로 나눴다.

```ts
type DragTarget =
  | { kind: "stage"; stageId: string; mode: DragMode }
  | { kind: "task"; taskId: string };
```

`onStageDrag` → `onDrag(target, deltaDays, phase)` 한 갈래로 합쳐 콜백이 늘어나지 않게 했다.

## 6. 알려진 사항 / 후속 과제

- 할일 칩은 **클릭 동작이 없다** — 드래그만 한다. 칩 클릭 → 작업 상세 열기는 별도 과제(단계 막대는 클릭 시 상세가 열린다)
- 할일을 단계 밖으로 옮기면 단계가 늘어날 뿐, **다른 단계로 옮겨지지는 않는다**. 단계 간 이동은 백로그 라벨·티켓에서 한다
- 완료된 할일도 동일하게 이동할 수 있다
- 프로젝트 자체 기간과 박스 드래그는 태스크 4 — 다른 세션의 스키마 PR 머지 후 진행
