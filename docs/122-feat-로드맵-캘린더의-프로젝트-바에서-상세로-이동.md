# feat: 로드맵·캘린더의 프로젝트 바에서 상세로 이동

## 작업 요약

작업 현황 **로드맵의 프로젝트 막대**와 **캘린더의 프로젝트 이름**을 눌러 해당 프로젝트 상세(`/projects/<id>`)로 이동할 수 있게 했다.

`MyWorkCalendar`는 `내 할일`과 `작업 현황 캘린더 뷰`가 공유하므로 한 곳을 고쳐 두 화면에 모두 반영된다.

## 변경 파일 내역

### `src/components/features/projects/workload-roadmap.tsx`

프로젝트 행의 `RoadmapBar`에 `onClick`과 `title`을 넘겼다.

```tsx
title={`${project.name} 상세 열기`}
onClick={() => router.push(`/projects/${project.id}`)}
```

`RoadmapBar`는 이미 `onClick`을 받으면 `<button>`으로 렌더하고, **드래그와 클릭을 `DRAG_THRESHOLD`(3px)로 구분**해 드래그로 막대를 옮긴 뒤에는 클릭을 한 번 삼킨다. 따라서 별도 처리 없이 "끌어서 기간 조절"과 "눌러서 상세 열기"가 공존한다.

### `src/components/features/my-work/my-work-calendar.tsx`

프로젝트 박스 **전체**를 `Link`로 만들었다. 박스의 빈 영역 어디를 눌러도 상세로 간다.

박스 전체를 클릭 대상으로 만들려면 두 가지 충돌을 해결해야 했다.

**충돌 1 — 안쪽 단계·할일의 클릭·드래그.** 박스가 그것들을 덮고 있어, 박스에 포인터 이벤트를 살리면 가로챌 것 같다. 하지만 실제로는 **단계 막대·할일 칩이 박스보다 DOM에서 나중에 그려지는 형제**라 쌓임 순서에서 위에 있다. 그래서 그것들 위를 누르면 그것들이 클릭을 받고, **박스의 빈 영역만** 박스 링크가 받는다. 별도 처리가 필요 없었다 — 검증에서 단계 클릭이 상세로 튀지 않고 단계 상세가 열리는 것을 확인했다.

**충돌 2 — 백로그 HTML5 드롭 (핵심).** 이건 그냥 두면 깨진다. 박스가 `pointer-events: auto`이면 백로그를 박스가 덮은 날짜에 떨어뜨릴 때 **박스가 드롭을 가로채** 아래 날짜 칸의 `onDrop`이 불리지 않는다. 내 할일의 핵심 기능이 막힌다.

해결은 **백로그를 드래그하는 동안만 박스를 투명하게** 만드는 것이다. 기존 드롭 로직은 한 줄도 건드리지 않는다.

```tsx
// 캘린더 최상위 컨테이너
onDragOverCapture={(e) => { if (isTaskDrag(e) && !draggingBacklog) setDraggingBacklog(true); }}
onDropCapture={() => setDraggingBacklog(false)}
onDragLeave={(e) => { if (밖으로 이탈) { setDraggingBacklog(false); setDropDate(null); } }}

// 박스
draggingBacklog ? "pointer-events-none" : "pointer-events-auto"
```

- **`onDragOverCapture`** — 캡처 단계에서 켠다. 박스가 아직 `auto`인 드래그 진입 첫 순간에도 최상위가 먼저 잡아 즉시 `draggingBacklog`를 켜고, 리렌더로 박스가 투명해진다.
- **끄는 신호는 드롭(성공)과 캘린더 밖 이탈(취소) 둘 다.** `dragend`는 드래그 **소스**(백로그 아이템)에서 발생해 캘린더로 오지 않으므로 의존하지 않는다.

평소엔 박스가 클릭 대상, 백로그 드래그 중엔 투명 → 기존처럼 날짜 칸이 드롭을 받는다.

### 부수 정리 — `aria-hidden` 제거

프로젝트 박스에 `aria-hidden`이 걸려 있었다. 박스 자체가 포커스 가능한 링크가 되었으므로 **`aria-hidden` 요소가 포커스를 받는 접근성 위반**이 된다. 걷어냈다.

## 검증

dev 서버(`localhost:3024`, 본 워크트리 전용 포트)에서 실제 브라우저로 조작했다. 오늘이 포함된 기간의 2단계 프로젝트와 백로그 할일 하나를 만들어 로드맵·캘린더 양쪽에 확실히 나타나게 한 뒤 조작했다. **전부 통과.**

| 항목 | 결과 |
| --- | --- |
| 프로젝트 박스가 `<a>`, `href`가 `/projects/<id>` | ✅ |
| **박스 빈 영역 클릭 → 상세로 이동** | ✅ |
| 회귀: 단계 막대 클릭이 상세로 튀지 않고 단계 상세가 열림 | ✅ |
| **회귀(핵심): 드래그 중 박스가 투명해져 아래 날짜 칸이 드러남** | ✅ (`elementFromPoint`가 링크가 아닌 날짜 칸을 반환) |
| **회귀(핵심): 백로그를 박스가 덮은 날짜에 드롭 → 예정일 배정** | ✅ (`2026-07-03`) |

백로그 드롭 회귀는 HTML5 DnD를 `dataTransfer`를 공유한 `dispatchEvent`로 재현했다. 신뢰도를 위해 **대조군**을 함께 돌렸다 — 프로젝트 박스가 없는 빈 날짜 칸에 같은 방식으로 드롭했을 때도 같은 날짜로 배정되어(`2026-07-03`), 시뮬레이션이 실제 `onDrop`을 태우고 있음과 박스 위 드롭이 빈 칸과 똑같이 동작함을 함께 확인했다.

## 검증 중 잡은 결함 (첫 접근에서)

처음에는 **이름 라벨만** 링크로 만들었는데, 라벨이 화면에 보이는데도 클릭이 타임아웃났다. `elementFromPoint`로 찍어 보니 단계 막대가 라벨 위를 덮고 있었다(`{ "top": "BUTTON", "title": "1단계" }`). 이후 요구가 "박스 전체 클릭"으로 확장되면서 박스를 통째로 링크로 바꿨고, 이 라벨-덮임 문제는 자연히 사라졌다(빈 영역이 넓어 라벨 외 지점으로 누른다). **href만 확인하고 넘어갔으면 라벨이 안 눌리는 채로 나갈 뻔했다** — 실제로 클릭해 본 것이 결정적이었다.

## 알려진 이슈 / 후속

- 라벨 폭을 `45%`로 제한해 프로젝트 이름이 길면 잘린다. `title` 속성에 전체 이름이 있어 호버로 확인은 된다.
- 백로그 드래그 종료를 드롭·이탈 두 신호로만 감지한다. 캘린더 안에서 드롭하지 않고 `Esc`로 드래그를 취소하는 드문 경우엔 `draggingBacklog`가 `true`로 남을 수 있다(박스가 잠시 클릭 불가). 다음 드래그의 `onDragLeave`나 드롭에서 곧 풀린다. 실사용 빈도가 낮아 이번엔 별도 리셋을 넣지 않았다.
- 이 작업은 처음에 **메인 폴더에서 편집을 시작하는 실수**가 있었다. 즉시 `git checkout`으로 되돌리고 워크트리를 만들어 패치를 옮겼다. 메인에는 다른 세션의 미커밋 작업이 없어 피해는 없었다.
