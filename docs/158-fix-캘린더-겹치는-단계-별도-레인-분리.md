# fix: 캘린더에서 겹치는 단계를 별도 레인으로 분리해 가려짐·드래그 불가 해결

## 요청

> Y.OS Core 프로젝트의 2단계가 캘린더 뷰에서 7월 20일 → 7월 21일 이상으로 드래그해도 안 넘어간다. (앞선 두 차례 수정에도 그대로) 새 환경에 맞게 해결해달라.

## 진짜 원인 (실제 앱 재현으로 확정)

내 할일 캘린더는 최근 **일/주/월 보기로 통째로 재구성**됐다(별도 세션 머지). 앞선 두 수정(#134·#139, 델타 대칭화·이동 시 할일 동반)은 그 새 컴포넌트에 통합돼 있었지만, 그게 문제의 원인이 아니었다. **로직만 보고 고쳐 두 번 헛짚었고, 실제 데이터로 앱을 띄워 드래그해 보고서야 진짜 원인을 찾았다.**

**원인: 한 프로젝트의 모든 단계 막대가 캘린더에서 한 줄(레인)에 겹쳐 그려진다.** 레이아웃 규칙이 `STAGE_LANES = 1`로 단계는 무조건 1줄이었고, 기간이 겹쳐도 나누지 않고 겹쳐 그렸다.

- Y.OS core는 1·2·3단계 기간이 겹친다(2단계 07-20~07-26, 3단계 07-22~08-01 등).
- 한 줄에 겹쳐 그리니 **뒤에 그려진 단계(3단계)가 앞 단계(2단계)를 덮는다.** 실제 측정: 2단계 막대는 07-20~07-25에 걸쳐 있으나 **07-22부터 3단계가 위에 덮여**, 2단계를 클릭·드래그로 잡을 수 있는 곳은 07-20~07-21(약 2칸)뿐이었다.
- 사용자가 "2단계"의 잘 보이는(넓은) 부분을 잡으면 실제로는 그 위의 3단계가 잡힌다. 그래서 2단계가 "안 움직인다".

> 참고: 드래그 후 저장은 원래 정상이었다(단계 PATCH가 나가 DB에 반영됨을 실제 커밋 드래그로 확인). 검증 도중 낡은 로컬 dev 서버에서 "저장 안 됨"처럼 보였으나, 깨끗한 빌드에서는 단계·할일 모두 정상 저장됐다.

## 수정 내용

`src/components/features/my-work/my-work-calendar-layout.ts` — **겹치는 단계를 할일과 같은 방식으로 별도 레인에 쌓는다.**

- `assignStageLanes()` 추가: 프로젝트별로 단계를 **그리드 전체 구간** 기준 그리디 구간 분할해 레인을 배정한다. 겹치는 단계는 서로 다른 줄로 간다. 전역 배정이라 **한 단계가 여러 주에 걸쳐도 같은 줄에 머문다**(주마다 배정하면 주 경계에서 막대가 위아래로 튄다).
- `layoutWeek()`: 단계 줄 수를 `stageLaneCount`(겹침 수, 최소 1)로 잡아 박스 높이에 반영하고, 각 단계 조각을 배정된 줄에, 할일 칩은 단계 줄들 **아래**에 놓는다.
- `buildWeekLayouts()`가 `assignStageLanes`를 한 번 계산해 각 주 배치에 넘긴다.
- 레이아웃 규칙 주석(3·4·5)도 새 동작에 맞게 갱신.

효과: 겹치는 단계가 각자 온전한 줄에 그려져 **어느 단계든 클릭·드래그로 잡을 수 있다.** 겹치면 프로젝트 박스가 그만큼 세로로 커진다(캘린더가 콘텐츠 크기로 늘고 페이지가 스크롤).

이 레이아웃(`buildWeekLayouts`)은 **내 할일 캘린더와 작업 현황 캘린더가 공유**하므로 두 캘린더 모두 겹침이 해소된다.

## 결정 이유

- 기존 규칙("단계는 한 줄, 겹쳐 그린다")은 화면을 컴팩트하게 두려는 의도였으나, 드래그가 도입된 뒤로는 **가려진 단계를 조작할 수 없게** 만든다. 사용자가 원하는 "단계를 끌어 옮기기"를 성립시키려면 각 단계가 잡혀야 하고, 그러려면 겹치는 것끼리 줄을 나눠야 한다(캘린더 앱이 겹치는 일정을 다루는 표준 방식).
- 할일 칩은 이미 겹칠 때 줄을 나누고 있었다 — 단계에도 같은 규칙을 적용해 일관되게 했다.
- 레인 배정을 **전역**으로 한 이유: 단계는 여러 주에 걸치므로, 주마다 따로 배정하면 같은 단계가 주 경계에서 다른 줄로 튄다.

## 검증 (실제 데이터 + 깨끗한 빌드)

로컬 dev(최신 main 기준 워크트리)에서 실사용자(김주웅, Y.OS core 소유) 세션으로 실제 드래그:

- **겹침 해소**: 2단계 막대 가로 전 구간을 짚어 `elementFromPoint`로 확인 → 6/6 지점 모두 "2 서비스·기술 설계"가 잡힘(수정 전에는 07-22부터 3단계가 잡혔다).
- **드래그·저장**: 2단계 중앙을 잡아 +2칸 드래그 → 미리보기가 따라오고(+279px), 손을 떼자 **STAGE PATCH(200) + 할일 PATCH들**이 나가고 DB `startDate`가 07-20 → 07-22로 반영됨. (검증 후 07-20으로 원복.)
- `npm run lint` 통과 · `npm run build` 성공(exit 0).

> 검증을 위해 실 데이터(Y.OS core 2단계와 그 할일 3개)를 드래그했다 원복했다. 예정일·단계 날짜는 원래대로 복구했고, 마감일은 서버의 `withDeadline` 규칙(예정일=마감일) 때문에 예정일과 같은 값이 됐다(원래 일부 null·07-20). 화면 위치에는 영향 없음.
