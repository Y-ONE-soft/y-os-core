## 1. 배경 — 반복된 병렬 충돌 3건

| 사례 | 내용 | 비용 |
|---|---|---|
| main 빌드 깨짐 이중 수정 | #6·#7 교차 의존으로 main이 깨졌고, 두 세션이 **같은 수정을 각자 구현** (docs/14의 수정 vs #8) — 리베이스 충돌로 수렴 | 중복 구현 + 충돌 해소 |
| DB 전환 이중 구현 | `할일-DB-전환` 세션이 workspace 보드 DB화를 진행 중인 걸 모른 채 이 세션이 동일 범위(스키마·시드)를 구현 → **한쪽 전체 폐기** | 작업 1건 폐기 |
| DB 전환 3중 시도 | 정리 직후 `프로젝트-데이터-DB-영속화` 세션이 또 겹치는 범위로 시작됨 (사용자에게 중단 권고) | 감지로 예방 |

공통 원인: **세션(작업자)들이 서로의 진행 상황을 확인하지 않고 시작**한다. 공유 자원(개발 DB 스키마)은 충돌이 즉시 전파된다.

## 2. 추가된 규칙 (`CLAUDE.md` › 병렬 작업 주의)

1. **범위 중복 확인 (요청 사이클 시작 전 필수)** — `git worktree list` + `gh pr list`로 진행 중 브랜치·PR 확인, 겹치면 시작하지 말고 사용자 보고
2. **DB 스키마 변경은 동시에 한 브랜치만** — 공유 DB 전제. 스키마 브랜치가 있으면 머지 후 시작. **`prisma db push` 금지, 항상 `migrate dev`** (마이그레이션 파일 미존재 드리프트가 이번 감지의 단서였음)
3. **부수 버그픽스는 초소형 PR 분리** — 기능 브랜치에 섞지 않고 즉시 머지 (이중 수정 방지)
4. **머지 직전 최신 main 리베이스 + 빌드 확인** — 각자 통과 ≠ 합쳐서 통과. base=최신 main이면 로컬 빌드로 갈음
