# feat: 내 작업 할일 상세 열기 및 완료 음영 통일

요청 사이클 `내-작업-할일-상세-완료표시`. "내 작업에서 할일도 누르면 상세 보기 나오게, 체크박스 선택하면 음영처리되면서 누가 봐도 완료된 것처럼" 요청을 처리한다.

두 태스크를 한 커밋으로 묶었다. 둘 다 같은 화면의 같은 요소(할일)를 다루고, 분리하면 "클릭은 되는데 완료 표시는 그대로"인 어중간한 상태가 커밋으로 남는다.

## 배경 — 무엇이 실제로 빠져 있었나

### ① 할일 칩만 클릭이 안 됐다

내 작업 캘린더에서 **단계 칩은 클릭하면 상세가 열린다**(`onClick={() => onOpenStage?.(...)}`). 그런데 **할일 칩은 `<div>`였고 `onPointerDown`(드래그)만 있었다.** 클릭 핸들러가 없어 눌러도 아무 일이 없었다. 요청의 "할일**도** 누르면"이 이 비대칭을 정확히 가리킨다.

내 작업 백로그 목록은 이미 이름 클릭으로 상세가 열린다. 빠진 곳은 캘린더 칩뿐이었다.

### ② 완료 표시가 화면마다 달랐다

| 위치 | 변경 전 |
|---|---|
| 캘린더 칩 | 배경 음영 + ✓ + 취소선 (충실) |
| 내 작업 백로그 행 | **글자만** 흐림 + 취소선 — 행 배경 그대로 |
| 프로젝트 백로그 행 | **글자만** 흐림 + 취소선 |
| 보드 카드 | **글자만** 흐림 + 취소선 — 흰 카드 그대로 |

캘린더만 제대로 돼 있고 나머지 세 곳은 글자만 바뀌어, 목록에서 완료 항목이 한눈에 구분되지 않았다.

## 변경 내용

### 1. `my-work-calendar.tsx` — 할일 칩을 버튼으로

`<div>` → `<button type="button">`으로 바꾸고 `onClick`을 붙였다. 단계 칩과 동일한 구조다.

```tsx
// 단계 칩과 동일하게 클릭 시 상세를 연다. 드래그와의 구분은 캘린더 루트의
// moved 플래그가 처리한다 — 움직이지 않았을 때만 onClick이 살아난다.
<button
  type="button"
  onPointerDown={(event) => onDragStart?.(event, { kind: "task", taskId: overlay.taskId })}
  onClick={() => onOpenTask?.(overlay.taskId)}
  …
>
```

**드래그/클릭 구분 로직은 새로 만들지 않았다.** `handlePointerUp`이 이미 `moved` 플래그로 판단해, 움직이지 않았으면 미리보기를 되돌리고 버튼의 `onClick`에 맡긴다. 단계 칩이 쓰던 그 장치를 그대로 태웠다.

`OverlayItem`과 `MyWorkCalendar`에 `onOpenTask?: (taskId: string) => void` prop을 추가해 연결했다.

### 2. `my-work-calendar-panel.tsx` — 상세 오버레이 연결

`detailTaskId` 상태와 `TaskDetailOverlay`를 추가했다. 단계 상세(`StageDetailOverlay`)가 `projectId`+`stageId`를 필요로 하는 것과 달리, **할일 상세는 `taskId` 하나면 된다** — `TaskDetailOverlay`가 스토어에서 위치를 직접 찾는다.

```tsx
// 할일 상세는 taskId만 있으면 된다 — TaskDetailOverlay가 스토어에서 위치를 찾는다
const [detailTaskId, setDetailTaskId] = useState<string | null>(null);
```

### 3. 완료 음영 — 세 곳 통일

**보드 카드** (`project-board.tsx`)

```tsx
{/* 완료 카드는 컬럼 배경에 잠기고 그림자를 잃어 뒤로 물러난다 —
    글자 취소선만으로는 한눈에 구분되지 않았다 */}
className={cn(
  "flex w-full shrink-0 items-center gap-2 rounded-[8px] px-2.5 py-2",
  task.done ? "bg-muted opacity-60" : "bg-background shadow-xs",
)}
```

완료 시 배경을 컬럼과 같은 `bg-muted`로 바꾸고 **그림자를 제거**한다. 흰 카드가 회색 컬럼 위에 떠 있다가 배경에 잠기면서 확실히 뒤로 물러난다.

**내 작업 백로그** (`my-work-backlog.tsx`) / **프로젝트 백로그** (`project-backlog.tsx`)

두 곳 모두 이미 `bg-muted`가 기본이라 배경을 더 낮출 여지가 없다. 행 전체에 `opacity-60`을 걸어 목록에서 물러나게 했다.

프로젝트 백로그는 요청 범위("내 작업") 밖이었지만, **보드 카드와 같은 화면에 나란히 있어** 한쪽만 바꾸면 같은 패널 안에서 표시가 어긋난다. 검증 스크린샷에서 실제로 그 어긋남이 보여 함께 맞췄다.

## 검증

**이번 사이클부터 실제 브라우저로 화면을 확인했다.** 그동안 "빌드는 통과했지만 눈으로는 못 봤다"가 반복돼, 시각 요구사항("누가 봐도 완료된 것처럼")을 검증할 수 없었다.

**새 의존성은 추가하지 않았다.** 로컬에 설치된 Chrome을 `--headless=new --remote-debugging-port`로 띄우고, Node 내장 `WebSocket`으로 CDP를 직접 호출했다(playwright·puppeteer 불필요). 스크립트는 스크래치패드에만 두고 저장소에 커밋하지 않았다.

`npm run build` 성공, `npm run lint` 경고 0.

### ① 완료 음영 — 실제 화면

내 작업(`/projects/my-tasks`): 백로그의 "백로그 완료" 행만 흐려지고 "백로그 미완료"는 선명하다. DOM 확인에서도 `opacity-60`이 완료 행에만 붙었다.

프로젝트 상세(`/projects/p-vis`): "완료된 할일" 카드가 컬럼 배경에 잠겨 그림자 없이 물러나고, "미완료 할일"은 흰 카드로 떠 있다. 오른쪽 프로젝트 백로그의 "백로그 완료"도 함께 흐려진다.

### ② 캘린더 할일 칩 클릭 → 상세

실제 마우스 press/release(이동 없음)로 검증했다.

```
실제 클릭 결과: {"dialogOpen":true,"title":"미완료 할일"}
```

작업 상세 오버레이가 열리고 제목이 일치한다. **덤으로 직전 사이클(docs/66)에서 눈으로 확인하지 못했던 "완료날짜 — 완료 체크 시 자동 기록" 필드도 이 스크린샷에서 처음으로 확인됐다.**

### ③ 드래그 회귀 — `<div>` → `<button>` 전환이 깨뜨리지 않았는가

실제 포인터 드래그(press → 6회 move → release)를 걸었다.

```
드래그 전 예정일: 2026-07-09
드래그 후 예정일: 2026-07-10
날짜 변경됨: 예 (드래그 정상)
상세가 열렸는가: 아니오 (정상)
```

드래그로 날짜가 바뀌고, **드래그 중에는 상세가 열리지 않는다.** `moved` 플래그가 의도대로 클릭을 억제한다.

### 검증 중 발견 — 캘린더 아래쪽 주의 칩이 다른 섹션에 가린다

첫 시도에서 드래그가 먹지 않아 원인을 파고든 결과, **할일 칩이 "요청 알림" 섹션에 덮여 있었다.**

```
chip rect: { y: 729, h: 20 }  visible: true  inViewport: true
hitIsSelf: false
hitChain: ["DIV.flex.w-full", "ARTICLE.flex.min-w-0", …]   ← 요청 알림 카드
```

단계 칩에는 포인터 이벤트가 도달하는데 할일 칩에는 전혀 도달하지 않아 이상해서 히트 테스트로 확인한 것이다. 칩 날짜를 캘린더 위쪽 주(7/8·7/9)로 옮기니 `hitIsSelf: true`가 되고 드래그·클릭 모두 정상 동작했다.

**이 겹침은 이번 변경과 무관한 기존 레이아웃 문제로 보인다**(칩이 없던 시절에도 같은 좌표에 단계 막대가 있었을 것). 다만 캘린더 아래쪽 주에 놓인 할일은 클릭도 드래그도 안 될 수 있으므로 별도 확인이 필요하다 — 아래 "알려진 이슈" 참고.

## 알려진 이슈 / 주의점

### 캘린더 아래쪽 주의 칩이 다음 섹션에 가릴 수 있다

위 검증 절에서 관측했다. 주 행 높이가 레인 수에 따라 늘어나는데, 늘어난 만큼 컨테이너가 따라 커지지 않으면 마지막 주의 칩이 "요청 알림" 영역과 겹친다. 겹치면 **클릭도 드래그도 먹지 않는다.**

이번 변경으로 생긴 문제가 아니라 기존 레이아웃 문제이며, 이번에는 재현 조건(한 달 내내 걸친 단계 + 여러 레인)이 만들어져 드러났다. 캘린더 높이 계산을 손보는 별도 사이클이 필요하다. 요청 범위를 넘고 다른 세션이 캘린더 레이아웃을 작업 중일 수 있어 이번에는 손대지 않았다.

### 완료 표시가 `opacity`라 대비가 약해질 수 있다

`opacity-60`은 배경까지 함께 흐려 텍스트 대비가 떨어진다. 접근성 기준(WCAG 대비비)을 엄격히 본다면 완료 항목의 본문 대비가 부족할 수 있다. 다만 완료 항목은 "덜 중요해진 정보"라 의도된 약화이고, 취소선·체크 표시가 상태를 중복으로 전달한다.

### 캘린더 칩의 완료 표현은 그대로 뒀다

캘린더 칩은 이미 배경 음영 + ✓ + 취소선으로 충분히 구분돼 있어 건드리지 않았다. 목록형(행·카드)과 칩형은 형태가 달라 같은 규칙을 강제하기보다 각 맥락에 맞는 표현을 유지하는 편이 낫다고 판단했다.

### 터치 환경

칩 클릭은 포인터 이벤트 기반이라 터치에서도 동작하지만, 드래그와의 구분 임계값(`DRAG_THRESHOLD`)이 마우스 기준이라 손가락으로는 의도치 않은 드래그가 잡힐 수 있다. 실기기 확인은 하지 못했다.
