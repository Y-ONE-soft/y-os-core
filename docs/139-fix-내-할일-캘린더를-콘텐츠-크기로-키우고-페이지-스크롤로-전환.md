# fix: 내 할일 캘린더를 콘텐츠 크기로 키우고 페이지 스크롤로 전환

## 배경 — 앞선 수정(#124)이 놓친 것

바로 앞 PR(#124, docs/137)에서 주 행에 최소 높이(`WEEK_MIN_PX = 104`)를 줬는데도, 사용자 화면에서는 캘린더가 여전히 작았다. 스크린샷을 받아 보니 진짜 원인이 드러났다.

**아래 "요청 알림" 섹션이 커지면 캘린더가 위로 눌린다.** `MyWorkRequests`는 `shrink-0`이고 요청 카드가 세로로 쌓이는데 높이 제한이 없어, 요청이 여러 건이면(사용자 화면은 7건) 그 섹션이 큰 공간을 차지한다. 캘린더는 `flex-1`이라 남는 공간만 받으므로, 요청이 많을수록 눌려 2주 정도만 보였다. 앞선 수정은 요청이 0인 상태로 측정해 이 관계를 놓쳤다.

## 원인 구조

```
my-work-page (h-full min-h-0 — 화면 높이에 가둠)
└─ 본문 (flex-1 min-h-0)
   ├─ 왼쪽 열 (flex-col)
   │  ├─ MyWorkCalendar (flex-1 overflow-y-auto ← 남는 공간만, 그 안에서 스크롤)
   │  └─ MyWorkRequests (shrink-0 ← 요청만큼 커짐)
   └─ MyWorkBacklog
```

페이지가 화면 높이에 갇히고(`h-full`), 캘린더가 그 안에서 요청 섹션과 공간을 나눠 갖는 구조였다. 요청이 크면 캘린더가 작아진다.

## 수정 — 캘린더는 콘텐츠 크기로, 넘치면 페이지가 스크롤

`(main)` 레이아웃의 `<main>`은 이미 `overflow-y-auto`다. my-work-page가 스스로를 화면 높이에 가두는 것만 풀면, 캘린더가 온전한 크기로 커지고 넘치는 만큼 바깥(main)이 스크롤한다.

### `my-work-calendar.tsx`

캘린더 루트를 `flex-1 overflow-y-auto` → **`shrink-0`**(콘텐츠 높이)로.

```diff
- className="flex min-h-0 w-full flex-1 flex-col overflow-x-clip overflow-y-auto rounded-[10px] border bg-card"
+ className="flex w-full shrink-0 flex-col overflow-x-clip rounded-[10px] border bg-card"
```

이제 캘린더 높이 = 요일 헤더 + (주 수 × 최소 104px). 화면 높이나 아래 요청 섹션과 **완전히 독립**이다. `shrink-0`이라 요청이 아무리 커도 눌리지 않는다.

### `my-work-page.tsx`

- 최상위 `h-full min-h-0` → **`min-h-full`**: 화면을 최소로 채우되 콘텐츠가 크면 커진다(main이 스크롤).
- 본문 `min-h-0 flex-1 items-stretch` → **`flex-1 items-start`**: 세로 늘림(`items-stretch`)을 풀어 백로그가 캘린더 높이만큼 늘어나지 않게.
- 백로그를 `sticky top-0`로 감싸 스크롤해도 화면에 머무르게.

## 공유 컴포넌트 안전성 — 작업 현황 캘린더 뷰

`MyWorkCalendar`는 **작업 현황 캘린더 뷰**(`task-status-calendar`)도 쓴다. 그쪽 페이지(`task-status-page`)는 이미 `overflow-y-auto`(페이지 스크롤) 구조라, 캘린더가 콘텐츠 높이가 되는 이번 변경과 오히려 패턴이 일치한다. 검증에서 데이터가 많은(59개 할일) 작업 현황 캘린더도 각 주 104px로 정상 렌더됨을 확인했다.

## 검증 (실제 브라우저)

dev(`localhost:3032`)에서 확인. **전부 통과.**

| 항목 | 결과 |
| --- | --- |
| 내 할일 캘린더 각 주 (step01) | 104px, 페이지 스크롤 ✅ |
| 작업 현황 캘린더 각 주 (master, 59개 할일) | 104px, 안 깨짐 ✅ |
| **요청 7건을 쌓은 뒤** 캘린더 각 주 | **104px 유지** ✅ (요청 섹션이 1417px여도 캘린더 안 눌림) |
| 요청 있을 때 페이지 스크롤로 요청 접근 | ✅ |

핵심 검증은 마지막 줄이다. master가 step01에게 ASSIGN 요청 5건을 실제로 만들어 "요청 알림"을 크게 쌓은 상태에서도 캘린더 각 주가 104px로 유지됐다(이전 구조라면 눌렸다). 만든 요청 5건은 검증 후 삭제해 잔여 0.

화면 캡처로 캘린더가 시원하게 커지고 할일 칩·단계 막대가 각 칸에 또렷이 보이는 것을 확인했다.

- `npm run lint` 통과 (경고 0)
- `npx tsc --noEmit` 통과
- `npm run build` 통과

## 알려진 이슈 / 후속

- 요일 헤더의 `sticky top-0`는 이제 캘린더 내부 스크롤이 아니라 페이지(main) 스크롤을 기준으로 붙는다. 캘린더가 온전히 보이면 스티키가 발동할 일이 적지만, 매우 낮은 화면에서 스크롤 시 요일 헤더가 main 상단에 붙는다. 동작상 문제는 없으나 필요하면 별도로 다듬을 수 있다.
- docs 번호는 작성 시점 main 최대(138) + 1로 139를 썼다.
