# 67. 단계·할일 색을 프로젝트 색 파생으로 통일

- **예정 커밋 메시지**: `refactor: 단계·할일 색을 프로젝트 색 파생으로 통일`
- **작업일**: 2026-07-22
- **작업 브랜치**: `프로젝트-색-체계-통일` (요청 사이클 3태스크 중 ②)

---

## 1. 배경 — 한 프로젝트 안에서 색이 제각각이었다

단계 색은 프로젝트와 **무관하게** 공용 팔레트(`PROJECT_COLORS`)에서 순번대로 배정됐다. 그래서 파란 프로젝트인데 단계는 파랑·보라·초록이 섞여 나왔고, 여러 프로젝트를 한 화면에 겹쳐 보는 **작업 현황 로드맵·내 작업 캘린더에서 어느 막대가 어느 프로젝트인지 색으로 구분할 수 없었다.**

## 2. 변경 — 색상은 프로젝트에서, 구분은 명도·채도로

프로젝트 색 하나에서 단계·할일 색을 파생한다. **색상(hue)은 프로젝트 색 그대로 두고 명도·채도만 흔든다.**

| 대상 | 색 |
|---|---|
| 프로젝트 | 사용자가 고른 색 (태스크 ①) |
| 단계 | 프로젝트 색과 **같은 hue**, 순번에 따라 명도·채도만 다른 톤 |
| 할일(캘린더 칩) | 소속 단계 톤을 **한 단계 더 옅게** (같은 계열, 단계 막대와는 구분) |

### 저장하지 않고 화면에서 파생한다

`stage.color` 컬럼에 값을 써 두면 **나중에 프로젝트 색을 바꿨을 때 옛 색이 남아 다시 어긋난다.** 그래서 저장값을 읽지 않고 화면에서 계산한다.

파생 지점을 **보드 스토어 훅 한 곳**(`useProjectBoard` / `useBoardState`)으로 잡은 것이 핵심이다. 프로젝트 상세(로드맵·보드·백로그), 작업 현황 로드맵, 내 작업 캘린더·백로그가 모두 이 훅을 통해 단계를 받으므로 **소비처를 하나도 고치지 않고** 전 화면이 일관돼진다.

DB에는 여전히 `color`가 필요하므로(생성 API 계약) 단계 생성 시 같은 규칙으로 계산한 값을 저장한다 — 화면은 그 값을 쓰지 않지만 API·데이터 정합성은 유지된다.

## 3. 톤 사다리 설계

`project-palette.ts`에 hex ↔ HSL 변환과 톤 계산을 모았다.

- 6단계 사다리를 순번에 따라 순환 적용 (명도 델타 `+8·−12·+14·−4·+11·−8`, 채도도 함께 조정)
- **기준 명도를 46~58로 당긴 뒤** 사다리를 적용한다. 처음엔 프로젝트 색의 명도에 그대로 더했는데, 밝은 색(예: `#8b5cf6`, L 66)에서는 사다리가 상한에 눌려 **서로 다른 순번이 같은 톤(L 66 vs 70)** 이 되는 문제가 실측으로 확인됐다. 기준을 중간대로 당겨 어떤 프로젝트 색이든 간격이 유지된다
- 명도는 32~72%로 가둔다 — 배경 위에서 읽히지 않을 만큼 밝거나 어두워지지 않게

## 6. 알려진 사항 / 후속 과제

- DB의 `stage.color`는 이제 **화면에서 쓰이지 않는 값**이다. 컬럼 제거는 서버 스키마 변경이라 별도 태스크로 남긴다(여러 세션이 `service.ts`를 동시에 수정 중)
- 기존에 저장된 단계 색은 그대로 남아 있으나 화면에는 영향이 없다 — 파생이 항상 덮어쓴다
- 프로젝트가 없는(미배정) 할일 칩은 종전대로 프로젝트 색 자리에 회색을 쓴다
- 태스크 ③(프로젝트 색 변경)이 붙으면, 색만 바꿔도 그 프로젝트의 모든 단계·할일이 즉시 따라 바뀐다 — 이번 파생 방식의 이점
