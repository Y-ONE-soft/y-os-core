# feat: 비밀번호 변경 API 추가

요청 사이클 `비밀번호-변경`의 1번 태스크. 서버·클라이언트 API 계층만 만든다. 화면은 2번 태스크.

## 배경

**비밀번호를 바꿀 수단이 아예 없었다.** 시드 비밀번호 `1111`이 고정이고 변경 경로가 없어, docs/83·84(내 정보 페이지)에서도 "운영 전 필수"로 남겨뒀던 항목이다.

## 결정 사항 (사용자 승인)

| 항목 | 선택 |
|---|---|
| 최소 길이 | **4자** — 시드가 `1111`(4자)이라 우선 맞춘다 |
| 다른 세션 처리 | **무효화** — 지금 쓰는 세션만 남기고 전부 끊는다 |

**최소 4자는 약하다.** 8자 이상을 권했으나 우선 4자로 진행하기로 했다. 아래 "알려진 이슈" 참고.

## 변경 내용

### 1. `src/server/auth/service.ts` — `changePassword`

```ts
export async function changePassword(
  userId: string,
  currentPassword: string,
  nextPassword: string,
  keepToken: string,
): Promise<void> {
  const user = await db.user.findUnique({ where: { id: userId } });
  if (!user) throw new WrongPasswordError();

  const matches = await bcrypt.compare(currentPassword, user.passwordHash);
  if (!matches) throw new WrongPasswordError();

  const passwordHash = await bcrypt.hash(nextPassword, 10);
  await db.$transaction([
    db.user.update({ where: { id: userId }, data: { passwordHash } }),
    db.session.deleteMany({ where: { userId, id: { not: keepToken } } }),
  ]);
}
```

**현재 비밀번호를 반드시 확인한다.** 세션 쿠키만으로 비밀번호를 바꿀 수 있으면, 세션이 탈취된 순간 공격자가 비밀번호를 갈아 계정을 통째로 가져간다. 현재 비밀번호 확인이 그 사이의 최소 방어다.

**성공하면 다른 세션을 전부 끊는다.** 비밀번호를 바꾸는 동기 자체가 "다른 데서 누가 내 계정을 쓰고 있다"인 경우가 있는데, 바꿔도 기존 세션이 살아 있으면 목적을 달성하지 못한다. `id: { not: keepToken }`으로 **지금 요청을 보낸 세션만 남긴다** — 자기 자신까지 끊으면 바꾸자마자 로그아웃돼 불편하다.

**트랜잭션으로 묶었다.** 비밀번호만 바뀌고 세션 정리가 실패하면 "새 비밀번호인데 옛 세션이 살아 있는" 어중간한 상태가 된다.

사용자를 못 찾는 경우도 `WrongPasswordError`로 처리했다. 세션이 가리키는 사용자가 없는 건 정상 흐름에서 생기지 않고, 굳이 다른 응답을 주면 계정 존재 여부를 흘릴 수 있다.

최소 길이는 상수로 뽑아 라우트가 메시지에 쓰도록 했다.

```ts
/**
 * 비밀번호 최소 길이. 시드 계정이 4자(`1111`)라 우선 4로 둔다.
 * 운영 전에는 반드시 올려야 한다 — docs 참고.
 */
export const PASSWORD_MIN_LENGTH = 4;
```

### 2. `src/app/api/auth/password/route.ts` (신규)

**`/api/auth/me`에 합치지 않고 분리했다.** 프로필 수정과 비밀번호 변경은 검증 규칙도, 실패 사유도, 성공 시 부작용(세션 무효화)도 다르다. 한 핸들러에 섞으면 분기가 지저분해지고 "이름만 고치는데 왜 비밀번호를 요구하나" 같은 혼선이 생긴다.

대상은 `/api/auth/me`와 같은 방식으로 **세션에서 정한다** — `userId`를 받지 않으므로 남의 비밀번호를 가리키는 우회가 성립하지 않는다.

거부 사유를 구분해 돌려준다.

| 상황 | 응답 |
|---|---|
| 미인증 | 401 `인증이 필요합니다.` |
| 본문 형식 오류 | 400 `잘못된 요청입니다.` |
| 최소 길이 미달 | 400 `새 비밀번호는 4자 이상이어야 합니다.` |
| 현재와 동일 | 400 `현재 비밀번호와 다른 비밀번호를 입력하세요.` |
| 현재 비밀번호 불일치 | 400 `현재 비밀번호가 올바르지 않습니다.` |

"현재와 동일"을 막는 이유는 바꿨다고 생각했는데 그대로인 상태를 방지하기 위해서다.

### 3. `src/lib/api/auth.ts` — `changePasswordApi`

/** 비밀번호 변경 — 성공하면 다른 기기의 세션은 모두 끊긴다 */

`npm run build` 성공, `npm run lint` 경고 0.

dev 서버(포트 3111)에 실제 HTTP 요청으로 검증했다. **같은 계정으로 세션 2개(A·B)를 만들어** 세션 무효화까지 확인했다.

### 거부 경로

| # | 시나리오 | 결과 |
|---|---|---|
| ① | 현재 비밀번호 틀림 | **400** `현재 비밀번호가 올바르지 않습니다.` |
| ② | 새 비밀번호 3자 | **400** `새 비밀번호는 4자 이상이어야 합니다.` |
| ③ | 현재와 동일 | **400** `현재 비밀번호와 다른 비밀번호를 입력하세요.` |
| ④ | 비로그인 | **401** `인증이 필요합니다.` |

### 정상 변경 + 세션 무효화 (핵심)

세션 A로 `1111 → 2222` 변경 후:

```
⑥ 세션 상태
  A: 200  ← 변경한 기기는 유지
  B: 401  ← 다른 기기는 끊김

⑦ 로그인 검증
  구 비밀번호 1111: 401
  신 비밀번호 2222: 200
```

### 원상복구

검증 후 **비밀번호를 `1111`로 되돌렸다.**

```
1111 로그인: 200   ← 원상복구
2222 로그인: 401
master01 영향 없음: 200
```

되돌리지 않으면 사용자가 로그인하지 못한다. 다른 계정(`master01`)이 영향받지 않은 것도 확인했다.

## 알려진 이슈 / 주의점

### 최소 4자는 운영에 부적합하다

우선 시드에 맞춰 4자로 뒀다. `1111`·`0000` 같은 값이 그대로 통과한다. **운영 전 최소 8자 이상 + 흔한 비밀번호 차단을 붙여야 한다.** `PASSWORD_MIN_LENGTH` 상수 하나만 올리면 되도록 뽑아뒀다.

### 이 커밋만으로는 화면에서 쓸 수 없다

폼이 없어 사용자가 바꿀 방법이 아직 없다. 2번 태스크에서 완결된다.

### 시도 횟수 제한이 없다

현재 비밀번호를 무한히 틀려도 잠기지 않는다. 세션이 이미 있어야 호출할 수 있으므로 외부 무차별 대입보다는 위험이 낮지만, 로그인 라우트를 포함해 **rate limit이 어디에도 없다**는 점은 같다. 별도로 다룰 사안이다.

### 비밀번호 재설정(잊어버림) 경로는 없다

로그인 화면에 "비밀번호를 잊으셨나요?" 링크가 있으나 동작하지 않는다. 이번 변경은 **로그인한 사용자가 스스로 바꾸는 것**만 다룬다. 이메일 발송이 필요한 재설정은 별도 작업이다.

### 다른 기기가 끊긴 것을 사용자가 알 방법

세션이 끊긴 기기는 다음 요청에서 401을 받아 로그인 화면으로 간다. 왜 로그아웃됐는지는 알려주지 않는다. 2번 태스크에서 **변경한 화면에 안내 문구**를 넣는다.
