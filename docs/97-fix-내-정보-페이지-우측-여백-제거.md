# fix: 내 정보 페이지 우측 여백 제거

요청 사이클 `내-정보-여백-개선`의 단일 태스크. "여백이 너무 커, 이것만 개선해줘" 요청. 레이아웃 폭만 손대고 다른 것은 건드리지 않는다.

## 원인

편집 폼 패널에 `max-w-[560px]`가 걸려 있었다. 계정 카드(280px) + 폼(560px) = 약 856px에서 콘텐츠가 끝나는데, 넓은 화면의 콘텐츠 영역은 1500px가 넘는다. **오른쪽 절반 가까이가 죽은 공간**이라 콘텐츠가 페이지에 고립돼 보였다.

**이 상한은 이 화면만의 예외였다.** 같은 성격의 형제 페이지를 확인한 결과 어디에도 `max-w`가 없다.

| 페이지 | 폭 처리 |
|---|---|
| 프리셋 관리 | `w-full` 2곳 + `flex-1` |
| 내 할일 | `flex-1` 2곳 |
| 내 정보 (이전) | **`max-w-[560px]`** ← 예외 |

docs/87에서 2단 레이아웃으로 바꾸면서 폼이 과하게 넓어지는 것을 막으려 넣었던 값인데, 결과적으로 앱 관행에서 벗어나 여백 문제를 만들었다.

## 시안 비교

폭은 눈으로 판단해야 해서 실제로 렌더해 비교했다.

| 시안 | 폼 폭 | 결과 |
|---|---|---|
| A | 상한 없음 (`flex-1`) | 우측 여백 없음. 입력이 약 450px로 넓어짐 |
| B | `max-w-[720px]` | 입력 폭은 알맞으나 **우측에 약 270px 여백이 다시 생김** |

B가 절충처럼 보였지만, 형제 페이지들이 전부 폭을 채우는 것을 확인하고 **A**로 정했다. 임의의 상한을 새로 정하는 것보다 앱 관행을 따르는 편이 낫다. 입력이 넓어지는 것은 설정 화면에서 흔한 형태이기도 하다.

## 변경 내용

`src/components/features/me/my-info-page.tsx` — 두 줄.

```diff
- className={cn("flex min-w-[320px] max-w-[560px] flex-1 flex-col", PANEL)}
+ className={cn("flex min-w-[320px] flex-1 flex-col", PANEL)}
```

로딩 스켈레톤도 실제 레이아웃과 어긋나지 않도록 함께 맞췄다.

```diff
- <Skeleton className="h-[236px] max-w-[560px] flex-1 rounded-[8px]" />
+ <Skeleton className="h-[236px] flex-1 rounded-[8px]" />
```

`min-w-[320px]`와 `flex-wrap`은 그대로 두어 좁은 화면에서 세로로 쌓이는 동작은 유지된다.

## 검증

`npm run build` 성공, `npm run lint` 경고 0.

시안 A·B를 각각 렌더해 비교했고, 최종 A에서 **우측 죽은 공간이 사라져 콘텐츠가 형제 페이지들과 같이 폭을 채우는 것**을 화면으로 확인했다.

저장 흐름 회귀도 확인했다.

```
드롭다운 항목: ["알림2","내 정보","프리셋 관리","설정","로그아웃"]
이동한 경로: /me
저장 결과: 저장했습니다.
헤더 표시 이름: 검검증용 이름
저장 버튼 비활성(변경 없음): true
```

### 검증 중 하마터면 실사용 데이터를 지울 뻔했다

`step01` 계정의 이름이 시드 값 `스탭`이 아니라 **`김주웅`으로 바뀌어 있었다.** 누군가 실제로 앱에서 수정한 값이다.

이전 사이클들에서는 검증 후 "시드 값으로 복구"하는 절차를 따랐는데, 이번에 그대로 했다면 **실사용 데이터를 시드로 덮어썼을 것**이다. 스크린샷에 `김주웅`이 찍혀 있어 알아차렸고, 시드가 아닌 **원래 값(`김주웅 / 사원 / step01@y-os.local / 010-0000-0002`)으로 복구**했다.

앞으로 계정 정보를 건드리는 검증에서는 **"시드로 복구"가 아니라 "검증 직전 값으로 복구"** 해야 한다. 개발 DB가 실사용 데이터를 담기 시작했다는 뜻이다.

## 알려진 이슈 / 주의점

### 아래쪽 여백은 그대로다

편집 항목이 4개뿐이라 콘텐츠 높이가 짧고, 뷰포트가 높으면 아래가 빈다. **이번 변경은 우측 여백만 해결한다.** 아래 여백을 채우려면 내용을 늘려야 하는데(비밀번호 변경 등), 요청이 "여백만"이었으므로 내용은 추가하지 않았다. 프리셋 관리 등 다른 짧은 페이지도 같은 상태라 이 화면만의 문제는 아니다.

### 초광폭 화면에서 입력이 매우 넓어진다

상한이 없으므로 2560px 같은 화면에서는 입력 하나가 900px를 넘을 수 있다. 연락처처럼 짧은 값에는 과하다. 다만 형제 페이지들도 동일한 조건이라 이 화면만 다르게 처리하지 않았다. 앱 전체에 콘텐츠 최대 폭을 도입한다면 그때 함께 다루는 편이 낫다.
