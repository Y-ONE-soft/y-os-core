# refactor: 단계 추가 오버레이를 단계 상세 디자인에 통일

## 1. 작업 요약

`단계 추가` 진입점(보드의 점선 컬럼)으로 열리는 **단계 추가 오버레이**를 **단계 상세 오버레이**와 같은 디자인으로 맞췄다.

두 화면은 원래 같은 설계(docs/45에서 추가 오버레이를 대형 오버레이로 전환)였으나, 이후 단계 상세만 작업 상세와의 통일 작업(docs/62·65·66 계열)으로 계속 갱신되면서 **추가 오버레이만 옛 디자인에 남았다.** 셸 크기·헤더·타이포·세부 사항 사이드바가 전부 어긋난 상태였다.

기능 변경은 없다. 생성 동작·검증 규칙·기간 기본값은 그대로다.

## 2. 변경 파일

| 구분 | 파일 | 내용 |

> 문서 번호는 작성 시점에 68이었으나, 머지 직전 `작업-칩-할일-고정`(PR #62)이 먼저 들어가며 68을 선점했다. "나중에 머지되는 쪽이 번호를 올린다"는 규칙에 따라 **69**로 조정했다.

단계 상세(`stage-detail-overlay.tsx`)는 **수정하지 않았다** — 이번 작업의 기준점이므로 손대면 통일의 근거가 사라진다.

기준은 현재 `stage-detail-overlay.tsx`다.

칩 텍스트는 `단계` 고정이다. 작업 도중 처음에는 상세를 따라 `단계 {stages.length + 1}`로 번호를 미리 보여주게 만들었으나, 머지 직전 리베이스에서 상세 쪽이 **순번을 빼고 종류 라벨로 고정**된 것을 확인하고(docs/68 — 사용자 요청 "작업1로 하지 말고 그냥 할일로 고정") 같이 맞췄다. 순번 계산에만 쓰이던 `useProjectBoard` 구독도 함께 제거했다 — 상세에서 `stageIndex`를 없앤 것과 같은 정리다.

경로는 상세와 같은 `프로젝트 › 단계` 규칙을 쓰되, 이름 입력 전에는 `새 단계`를 자리표시로 넣고 **입력하는 대로 즉시 반영**된다.

## 4. 의도적으로 남긴 차이

| 항목 | 추가 | 상세 | 이유 |
|---|---|---|---|
| 제목 | `Input` (입력) | `DialogTitle` (표시) | 생성 화면은 이름을 받아야 한다. `h2`는 `sr-only`로 접근성 제목만 유지 |
| Checkbox | `disabled` | 활성 | 단계가 아직 없어 완료 처리 대상이 없다 |
| 산출물·연결·댓글 | 안내 문구 | 실제 기능 | 생성 전에는 붙일 대상이 없다. 박스·타이포는 동일하게 맞춤 |
| 하단 | 추가 / 취소 | 단계 삭제 + `변경사항은 즉시 저장됩니다` 메타 | 생성은 명시적 커밋, 상세는 즉시 저장 |

## 5. 작업 중 발견해 고친 것 — 제목 행 12px 밀림

실측 대조에서 셸·헤더·타이포가 다 일치하는데도 **왼쪽 컬럼 전체가 추가 쪽에서 약 12px 아래로** 밀렸다.

원인은 제목 `Input`에 shadcn 기본 세로 패딩(`py-1`)이 남아 있던 것. `h-auto px-0`만 덮어써서 좌우 패딩과 높이는 풀었지만 상하 패딩이 살아 있어 제목 행이 상세의 `h2`보다 높아졌고, 그 차이가 아래 섹션 전부로 전파됐다.

`py-0`을 추가해 해결했다. 실측으로 제목 행 높이 `38px`, 첫 섹션 시작 위치 `159px`가 양쪽 동일해졌다.

이런 종류의 어긋남은 클래스 대조로는 잡히지 않는다(양쪽 다 `text-[30px] font-semibold`가 맞다). 렌더된 박스를 재봐야 나온다.

### 검증 환경 특성 (재현 시 참고)

- **공유 개발 DB에 프로젝트가 없다** (docs/46에서 시드를 그룹 골격만 남기고 축소). 검증 스크립트가 임시 프로젝트·단계를 API로 만들고 **끝나면 삭제**한다. 중간에 죽어 남은 경우를 대비해 시작 시 같은 이름의 잔여 프로젝트를 먼저 정리한다.
- **Radix Dialog는 `zoom-in-95`로 열린다.** 애니메이션 도중 `getBoundingClientRect()`를 재면 transform 배율이 섞여 1280이 1230~1275로 잡힌다(실제로 이 때문에 "불일치 3건" 오탐이 났다). 폭이 연속 프레임 동일해질 때까지 기다린 뒤 측정해야 한다.
- 오버레이 닫기는 헤더의 `[aria-label="닫기"]`를 직접 클릭한다 (Escape는 이 환경에서 불안정했다).

## 7. 알려진 사항 / 후속 과제

- **산출물·연결·댓글은 생성 전 안내 문구 상태**다. 단계 생성 직후 상세 오버레이로 이어가는 `onCreated` 경로가 이미 있어 실사용에는 문제가 없지만, 추가 화면에서 바로 입력받게 하려면 생성-후-패치 흐름(현재 내용·공동작업자가 쓰는 방식)을 확장해야 한다.
- 두 오버레이가 셸·헤더·세부 사항 골격을 거의 그대로 중복하고 있다. `OverlayBreadcrumb`처럼 **셸(`OverlayShell`)과 세부 사항 필드**도 공용 컴포넌트로 뽑으면 이런 드리프트가 구조적으로 재발하지 않는다 — 다만 작업 상세까지 3개를 함께 봐야 해서 별도 사이클이 맞다.
- 이번에 겪은 것처럼 **클래스는 같은데 렌더 박스가 다른** 경우가 있어, 오버레이 계열 정합은 실측 대조를 붙이는 편이 안전하다. 이번 검증 스크립트는 스크래치패드에만 두었고 리포지토리에 넣지 않았다.
