# feat: 사용자별 공통 작업 기본 프로젝트 보장 및 미배정 할일 이관

## 요청 (사이클 B 태스크 2)

계정마다 "{이름}의 공통 작업" 기본 프로젝트가 있어야 하고(작업 현황에도 일반 프로젝트처럼 노출), 기존 "프로젝트 없음(projectId=null)" 할일을 **담당자 기준**으로 그 공통 작업에 이관한다.

## 접근 — "로드 시 보장(ensure)" 자가치유

앱에 **계정 생성 훅이 없다**(사용자는 `prisma/seed.ts`로만 생성). 그래서 온보딩 훅 대신 **워크스페이스 로드 시점**에 보장한다 — 백필(기존 계정)·신규 계정 자동 보장·미배정 이관을 한 함수로 처리하고, 정상 상태에선 대상이 없어 값싸게 끝난다.

## 변경 — `src/server/workspace/service.ts`

`ensureDefaultProjects()` 신설 + `getWorkspace()` 맨 앞에서 호출.

1. **기본 프로젝트 보장** — `isDefault=true` 프로젝트가 없는 사용자(그룹 있는)에게 생성.
   - 이름 `"{name}의 공통 작업"`, 색 회색(`#71717a`, 옛 미배정과 동일 — 개인 공간 표시), owner=본인, group=본인 그룹, `isDefault=true`.
   - **동시성 안전**: id를 결정적(`p-default-<userId>`)으로 두고 `createMany({ skipDuplicates: true })` — 두 세션이 동시에 로드해도 중복 생성 안 됨.
   - `createProject`와 달리 **기본 단계를 만들지 않는다**(공통 작업은 느슨한 할일 담는 곳 = 백로그/단계 없음).
2. **미배정 이관** — 담당자 있는 `projectId=null` 할일이 있을 때만(`count`로 건너뜀), 각 공통 작업으로 `updateMany`(담당자=owner 기준). 담당자 없는 미배정은 그대로 `null` 유지.

## 그룹 없는 사용자

- `Project.groupId`가 필수라, `groupId=null` 사용자는 이번 보장에서 제외한다(시드 계정은 모두 `g-soft` 소속). 그룹이 생기면 다음 로드에 보장된다.

## 정상 상태 비용

- 백필 후: "기본 프로젝트 없는 사용자" 조회 → 0건, "담당자 있는 미배정" count → 0건(B3에서 새 느슨한 할일은 처음부터 공통 작업에 생성). 즉 매 로드에 값싼 조회 2번으로 끝남.

## 데이터 영향 (공유 개발 DB)

- **비파괴·additive·멱등** — 프로젝트 생성(신규 행)·미배정 이관(projectId만 채움). 되돌리기 가능.
- 첫 워크스페이스 로드에 시드 계정(master01·step01)의 공통 작업이 생기고, 그들 담당의 미배정 할일이 이관된다.

## 알려진 이슈 / 후속

- **삭제 가드 미포함**: 공통 작업 프로젝트를 삭제하면 Task가 Cascade로 함께 지워지고, 다음 로드에 빈 공통 작업이 재생성된다. 삭제 방지 가드는 B3(또는 후속)에서 검토.
- 이름은 생성 시점 사용자명 기준 — 개명 시 자동 갱신 안 함(후속).
