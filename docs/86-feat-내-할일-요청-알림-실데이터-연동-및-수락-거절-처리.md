# feat: 내 할일 요청 알림 실데이터 연동 및 수락·거절 처리

- **예정 커밋 메시지**: `feat: 내 할일 요청 알림 실데이터 연동 및 수락·거절 처리`
- **작업일**: 2026-07-22
- **작업 브랜치**: `요청-알림-실데이터`

---

## 1. 작업 내용 요약

내 할일 화면의 **요청 알림 카드에서 가짜 데이터를 걷어내고 실제 요청을 렌더**하도록 바꿨다. 수락·거절·요청 취소 버튼도 실제로 동작한다.

이 사이클의 원래 요구("기존 가짜 내용을 삭제하고 새로 만들자")가 이 태스크에서 완결된다.

### 걷어낸 가짜 데이터

`my-work-data.ts`의 `REQUESTS` 상수 2건이 전부였다.

- `노윤기 → 김주웅 · "회의1" 할일을 할당했어요` (할일 할당 요청 / 받은 요청)
- `노윤기 → 김주웅 · "회의 안건 정리가 막혀요"` (도움 요청 / 보낸 요청)

등장 인물(노윤기·김주웅)도 코드(YOC-909)도 실재하지 않았고, 수락·거절·요청 취소 버튼은 **모두 무동작**이었다. 한 카드에 세 버튼이 동시에 붙어 있어(받은 요청인데 "요청 취소"까지) 방향 개념도 맞지 않았다.

**파일 전체가 이 상수뿐이라 `my-work-data.ts`를 삭제했다.** 참조처가 하나도 남지 않은 것을 확인했다.

## 2. 변경 파일 내역

| 구분 | 파일 | 내용 |
|---|---|---|
| 수정 | `src/components/features/my-work/my-work-requests.tsx` | 실데이터 렌더 + 수락·거절·취소 동작, 카드 구조 재작성 |
| 삭제 | `src/components/features/my-work/my-work-data.ts` | 가짜 `REQUESTS` 상수 (파일 전체가 이것뿐) |
| 추가 | `docs/86-feat-내-할일-요청-알림-실데이터-연동-및-수락-거절-처리.md` | 본 문서 |

`my-work-page.tsx`는 **건드리지 않았다** — `<MyWorkRequests />`가 props 없이 호출돼 있어 호출부 변경이 필요 없었다. (동시에 `내작업-타임라인` 세션이 이 파일을 수정 중이라 충돌도 피했다)

## 3. 카드 구성

| 요소 | 내용 |
|---|---|
| 종류 배지 | `할일 요청`(primary) / `도움 요청`(warning 톤) — 기존 자리표시의 색 규칙 유지 |
| 방향 배지 | `받은 요청` / `보낸 요청` — 로그인 사용자 기준으로 서버가 판정해 내려준다 |
| 상태 배지 | 대기 중이 아닐 때만 `수락됨`·`거절됨`·`취소됨` 표시 |
| 대상 | `단계`/`할일` 배지 + 대상 이름 + 소속 프로젝트명 |
| 상대방 | 아바타(id 파생 색) + `상대 → 나` 또는 `나 → 상대` + 메시지 |
| 동작 | 받은 요청이면 **수락·거절**, 보낸 요청이면 **요청 취소**. 대기 중일 때만 노출 |

**정렬**: 대기 중인 요청을 앞으로 보낸다. 처리가 끝난 건 뒤로 밀린다.
**헤딩 카운트**: 전체가 아니라 **대기 중 건수**다 — "요청 알림 · N"의 N은 조치가 필요한 수여야 의미가 있다.

## 4. 설계 결정과 이유

| 결정 | 선택 | 이유 |
|---|---|---|
| 방향 판정 | 서버가 `direction` 필드로 내려줌 | 클라이언트가 `toUserId === 내 id`를 매번 비교하는 것보다 명확하고, 세션 사용자를 이미 아는 서버가 판정하는 편이 안전 |
| 받은·보낸 요청 한 목록 | 분리하지 않음 | 디자인 자리표시가 한 줄에 섞어 놓았고, 방향 배지로 구분된다. 건수가 늘면 필터가 필요해질 수 있다 |
| 버튼 노출 | 대기 중일 때만 | 이전 자리표시는 받은 요청에도 "요청 취소"가 붙어 있었다 — 방향·상태에 따라 가능한 동작만 보여준다 |
| 완료 요청 유지 | 목록에 남김(뒤로 정렬) | 수락·거절 결과를 확인할 수 있어야 한다. 숨기면 눌렀는데 사라져 결과를 알 수 없다 |
| 가로 스크롤 | `overflow-x-auto` 추가 | 카드가 3장 이상이면 넘친다. 자리표시는 2장 고정이라 문제가 없었다 |

## 6. 알려진 사항 / 후속 과제

- **폴링이 없다.** 상대가 방금 보낸 요청은 화면을 다시 열거나 새로고침해야 보인다. 실시간 알림(SSE·폴링)은 별도 사이클 과제다.
- **읽음 처리 개념이 없다.** 상태만 있고 "확인함"은 두지 않았다 — docs/84의 결정과 같다.
- **완료된 요청이 계속 쌓인다.** 지금은 전부 목록에 남는다. 오래된 처리 완료 건을 접거나 보관하는 정리가 나중에 필요하다.
- 실제 사용자가 2명(`master01`·`step01`)뿐이라 **여러 명에게 동시에 보내는 상황은 검증하지 못했다.** 서버는 사람별 1건씩 만들도록 짜여 있다(docs/84).
