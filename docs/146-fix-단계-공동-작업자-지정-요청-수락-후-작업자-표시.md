# fix: 단계 공동 작업자 지정 요청 수락 후 작업자가 표시되도록 수정

## 증상

단계 상세에서 "공동 작업자 지정 요청"을 보내고 상대가 수락해도, 단계 상세의 "작업자"에 그 사람(공동 작업자)이 나타나지 않았다. 새로고침해야만 보였다.

## 원인 (재현으로 확정)

서버·표시 컴포넌트는 정상이었다. `master01`이 단계 ASSIGN 요청 → `step01` 수락 → `GET /api/admin/workspace` 하면 서버는 그 단계에 `collaborators: [{step01}]`을 정확히 실어 준다(`acceptedCollaborators()` + `toStage()`), `AssigneeList`도 공동 작업자를 렌더한다.

문제는 **클라이언트 스냅샷 드리프트**였다:

- 공동 작업자(`stage.collaborators`)는 **워크스페이스 스냅샷**(`workspace-cache`)에 실려 온다.
- 수락 여부는 **요청 도메인**(`use-requests`의 별도 store)이 관리한다.
- 요청 수락은 받은 사람이 하고(다른 세션), 수락 시 갱신되는 건 요청 store뿐 — **워크스페이스 스냅샷은 재조회되지 않는다.** 그래서 단계 상세를 봐도 리로드 전엔 공동 작업자가 안 나온다.

## 수정

`src/components/features/projects/stage-detail-overlay.tsx`

- 단계 상세가 열릴 때(`stageId`가 non-null이 될 때) `workspace-cache`의 `refresh()`를 호출해 워크스페이스를 다시 받는다. 서버가 수락된 공동 작업자를 계산해 주므로, 재조회 후 `stage.collaborators`가 채워지고 `AssigneeList`에 공동 작업자가 나타난다.

```tsx
useEffect(() => {
  if (stageId !== null) void refreshWorkspace();
}, [stageId]);
```

- 열 때만 재조회하므로(사용자가 확인하려고 여는 시점) 낙관적 편집과 충돌할 여지가 작다. 요청과 워크스페이스 두 스냅샷의 드리프트를 "볼 때 최신화"로 메운다.

## 결정 이유

- **서버는 이미 정답을 준다** — 새 API·계산 없이, 클라이언트가 그 결과를 제때 받아오게만 하면 된다.
- 실시간(웹소켓/폴링)은 이 앱 범위 밖 — 열 때 재조회가 가장 작고 확실한 방법이다. 열려 있는 동안 다른 세션이 수락하면 닫았다 다시 열면 반영된다.
- 할일 상세의 공동 작업자도 같은 스냅샷 구조지만, 이번 사이클에서 할일의 공동 작업자 지정 경로 자체를 제거(docs/145)했으므로 대상에서 뺐다. 보고된 단계 케이스만 고친다.

## 실행/검증

```bash
npm run lint    # 통과
npm run build   # 통과 (타입 포함)
```

**실제 앱 재현→수정 확인 (dev 서버 + 헤드리스 Edge, 전용 테스트 데이터):**

1. `master01`로 프로젝트·단계 생성, 프로젝트 상세 진입(워크스페이스 로드 — 아직 공동 작업자 없음).
2. `master01`→`step01` 단계 ASSIGN 요청 발송, `step01`이 수락(다른 세션에서 수락됨을 흉내).
3. `master01` 화면에서 단계 상세 열기 → 재조회 후 "작업자"에 **김주웅(step01) 공동 표시됨**을 확인.
   - 수정 전: 안 나옴(스냅샷 미갱신). 수정 후: 스토어·렌더 모두 공동 작업자 표시. 닫았다 다시 열어도 유지.
