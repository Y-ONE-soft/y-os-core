- **예정 커밋 메시지**: `feat: 백로그 프로젝트·단계 지정 드롭다운 라벨`
- **작업일**: 2026-07-22
- **작업 브랜치**: `백로그-프로젝트-단계-지정` (워크트리 `.claude/worktrees/backlog-assign-label`, base: main `60b712d`)

## 1. 작업 요약

내 작업 백로그의 각 항목 오른쪽에 **프로젝트 · 단계 드롭다운 라벨**을 붙여, 티켓을 열지 않고도 작업의 소속을 바꿀 수 있게 했다. 함께 겹쳐 있던 스토어 액션 2개를 하나로 합치고, 서버가 프로젝트+단계 동시 지정을 안전하게 처리하도록 보강했다.

## 2. 변경 내용

### 2-1. 백로그 행의 드롭다운 라벨 (`my-work-backlog.tsx`)

- 행 구조: `체크박스 → 작업 이름 → [프로젝트 색점 + "프로젝트 · 단계"] 라벨` (작업 이름 위치는 docs/29 그대로 유지)
- 라벨을 누르면 **프로젝트별 그룹** 드롭다운이 열린다. 각 그룹에 프로젝트명 헤더(색 점)와 `백로그` + 해당 프로젝트의 단계 목록이 나오고, 현재 소속에는 체크 표시
- 항목을 고르면 `assignTask`로 즉시 이동 (낙관적 업데이트 후 저장)

**단계를 지정하면 그 작업은 백로그 목록에서 사라진다** — 백로그의 정의가 "단계 미지정(stageId = null)"이기 때문이다. 디자인 문구("백로그 → 단계로 드래그하면 편입됩니다")와 같은 동작을 드래그 없이 제공하는 셈이다.

### 2-2. 스토어 액션 통합 (`board-store.tsx`)

겹치던 `moveTask`(같은 프로젝트 내 단계 이동)와 `moveTaskToProject`(프로젝트 이동)를 **`assignTask(fromProjectId, taskId, toProjectId, toStageId)`** 하나로 합쳤다. 같은 프로젝트 단계 이동·타 프로젝트 이동·백로그 편입을 모두 한 경로로 처리하고, 저장도 `{ projectId, stageId }` 한 번의 PATCH로 끝난다. 호출부는 티켓의 프로젝트·단계 Select 2곳과 이번 라벨.

### 2-3. 서버: 프로젝트+단계 동시 지정 허용 및 검증 (`service.ts`)

직전 태스크(docs/29)에서는 `projectId`가 오면 `stageId`를 **무조건 null**로 밀어냈다. 이번엔 프로젝트와 단계를 함께 지정할 수 있어야 하므로 규칙을 바꿨다:

- `projectId`가 없으면 기존과 동일 (추가 쿼리 없음)
- `projectId`가 있으면 — 함께 온 `stageId`가 **대상 프로젝트 소속인지 조회로 검증**하고, 아니거나 없으면 대상 프로젝트의 백로그(null)로 안착

교차 프로젝트 오염(프로젝트 B의 작업이 프로젝트 A의 단계를 가리키는 상태)을 서버에서 원천 차단한다. 추가 조회는 이 경로에서만 1회 발생한다.

### 검증 중 겪은 병렬 이슈 2건 (기록)

- **공유 DB 마이그레이션 선반영으로 프로덕션 로그인 중단**: 다른 세션이 `20260722145429_add_user_group_fk_and_project_owner`(User.group → groupId)를 공유 개발 DB에 적용했으나 코드가 미머지 상태여서, main 코드가 없어진 컬럼을 조회해 **프로덕션 로그인이 500**이 됐다(P2022 ColumnNotFound). 사용자 보고 후 해당 세션이 `group` 컬럼을 복구해 해소. CLAUDE.md의 "DB 스키마 변경은 한 브랜치만" 규칙은 *머지 순서*를 다루지만, 이번처럼 **머지 전에 DB만 먼저 바뀌는** 경우도 즉시 전파된다.
- **dev 포트 충돌로 남의 앱을 검증할 뻔함**: 3011을 다른 세션(`staff-project-create`)이 선점해 내 `npm run dev -- -p 3011`이 바인딩에 실패했고, 요청이 그 세션의 앱으로 갔다. "드롭다운이 렌더되지 않음"으로 보였으나 실제로는 다른 워크트리의 앱이었다. **포트 점유 여부를 먼저 확인하고 고유 포트를 쓸 것** (이번엔 3017).

## 6. 알려진 사항 / 후속 과제

- 라벨은 백로그 목록에 있으므로 단계 부분이 항상 "백로그"다. 보드 카드에도 같은 라벨을 붙이면 단계 값이 의미를 갖는다 — 후속 과제
- 백로그 → 캘린더 드래그 일정 지정은 여전히 미구현(문구만)
- 담당자(assignee) 도메인 도입 시 내 작업 필터 재정의 필요 (docs/26·29와 동일)
