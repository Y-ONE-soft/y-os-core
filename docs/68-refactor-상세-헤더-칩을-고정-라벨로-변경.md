## 1. 작업 내용 요약

상세 오버레이 헤더 좌측 칩에서 **순번을 빼고 종류 라벨로 고정**했다 (사용자 요청: "작업1로 하지 말고 그냥 할일로 고정").

| 화면 | 이전 | 이번 |
|---|---|---|
| 할 일 상세 | `작업 1`, `작업 2` … (소속 안 순번) | **`할 일`** 고정 |
| 단계 상세 | `단계 5` … (로드맵 순번) | **`단계`** 고정 |

- 단계 칩은 사용자에게 확인 후 함께 고정 — 두 화면 헤더가 `[종류 칩] + [브레드크럼]`으로 완전히 대칭이 됐다.
- 순번은 브레드크럼(`프로젝트 › 단계 › 할 일`)이 위치를 알려주므로 중복이었고, 목록에서 항목이 추가·삭제될 때마다 바뀌어 식별자로도 불안정했다.

## 2. 함께 정리한 것

칩 표기에만 쓰이던 계산 로직을 제거했다.

- `task-detail-overlay.tsx` — `siblings`(단계/백로그/미배정 목록 분기) + `taskIndex` 계산 삭제
- `stage-detail-overlay.tsx` — `stageIndex` 계산 삭제

렌더마다 목록을 훑어 인덱스를 찾던 작업이 사라져 코드도 단순해졌다. (`unassigned`는 다른 곳에서도 쓰이므로 유지)

## 4. 검증

`npm run build` ✓ · `npm run lint` ✓ · 브라우저 실로그인(master01), 콘솔 에러 0.

검증용 단계 1개 + 작업 3개(단계 소속 2개, 백로그 1개)를 만들어 확인:

| 대상 | 칩 | 브레드크럼 |
|---|---|---|
| 단계 소속 작업 ① | **`할 일`** | `Y.OS core › 칩 검증 단계2 › 칩 작업 하나2` |
| 단계 소속 작업 ② | **`할 일`** (①과 동일 — 순번 사라짐 확인) | `… › 칩 작업 둘2` |
| 백로그 작업 | **`할 일`** | `Y.OS core › 백로그 › 칩 백로그 작업2` |
| 단계 상세 | **`단계`** | `Y.OS core › 칩 검증 단계2` |

- 같은 단계의 서로 다른 작업이 모두 `할 일`로 나오는 것으로 순번 제거를 확인했다.
- 검증용 단계·작업은 종료 후 삭제해 원상 복구했다.

## 5. 알려진 사항

- 로드맵 막대의 단계 순번 배지(①②③)는 그대로다 — 그건 타임라인에서 순서를 읽는 용도라 유지가 맞다.
